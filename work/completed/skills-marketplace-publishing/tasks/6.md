## Description

This task copies the global command at `/root/.claude/commands/generate-product-showcase-gallery.md` into the aisell project scope at `.claude/skills/generate-product-showcase-gallery.md`. Before placing the copy, the file must be audited and sanitized: all internal infrastructure references are replaced with generic placeholders so the file is safe for a public repository.

The original global command must NOT be removed — it is used across multiple products (SimpleSite and SimpleDashboard) and is referenced from the global CLAUDE.md as a `/generate-product-showcase-gallery` slash command.

This task belongs to Wave 3 and depends on Tasks 1–4 completing first. The showcase gallery skill is tightly coupled to the products being worked on in earlier waves.

Security requirement: the destination file will live inside a public GitHub repo. It must contain zero occurrences of:
- `localhost:` URLs (webchat-platform-specific endpoints)
- `/root/` filesystem paths (server-specific absolute paths)
- `HYDRA_API_KEY` variable name (internal API key identifier)

## What to do

1. Read the source file `/root/.claude/commands/generate-product-showcase-gallery.md` in full and identify every occurrence of the three forbidden patterns: `localhost:`, `/root/`, `HYDRA_API_KEY`.

2. Produce a sanitized copy at `/root/aisell/.claude/skills/generate-product-showcase-gallery.md` with all forbidden patterns replaced:
   - Any `localhost:PORT` URL (e.g. `localhost:8091`, `localhost:8092`) → `<WEBCHAT_URL>`
   - Any absolute `/root/` filesystem path → `<project-root>/` for project paths, `<user-workspace>/` for user data paths (e.g. `/root/aisellusers/user_999999999/` → `<user-workspace>/`)
   - `$HYDRA_API_KEY` → `$YOUR_IMAGE_API_KEY`

3. Keep the original at `/root/.claude/commands/generate-product-showcase-gallery.md` intact — do not move or delete it.

4. Run the verification commands from the Verification Steps section to confirm the task is complete.

**Forbidden patterns found in source — known occurrences:**

`localhost:` references (replace all with `<WEBCHAT_URL>`):

- `https://noxonbot.wpmix.net/showcases/$SLUG/screenshot.png` — this is a public URL, NOT a localhost reference; keep as-is

`/root/` references (replace with `<project-root>/` for project files, `<user-workspace>/` for user data):
- `/root/aisell/products/$PRODUCT/showcases/$SLUG` → `<project-root>/products/$PRODUCT/showcases/$SLUG`
- `/root/aisell/products/$PRODUCT/CLAUDE.md` → `<project-root>/products/$PRODUCT/CLAUDE.md`

- `/root/space2/hababru/telegram_file_sender.py` → `<project-root>/hababru/telegram_file_sender.py`
- `/root/aisell/bananzabot/.env` if present → `<project-root>/bananzabot/.env`

`HYDRA_API_KEY` references (replace with `YOUR_IMAGE_API_KEY`):
- `$HYDRA_API_KEY` in the balance-check curl command near the end of the file
- Any other occurrence of the variable name in env sourcing or comments

**Edge cases:**

- Some forbidden patterns appear inside heredoc blocks and bash string literals — grep will still find them, they must all be replaced
- `WEBCHAT_PORT=8091` — contains the port number but NOT `localhost:`, so not caught by the grep pattern; no replacement needed
- Relative paths like `extensions/webchat-sidebar/scripts/render_promo_preview.js` (without `/root/` prefix) — these are fine as-is, do not replace
- The source file references `products/simple_dashboard/CLAUDE.md` and `products/simple_site/CLAUDE.md` (relative paths) in some places — keep as-is, no `/root/` prefix present
- URLs like `https://d999999999.habab.ru/` are public domain URLs, not internal — keep as-is
- JavaScript code in the source that references `localhost:8091` must also be sanitized

**Decision 5 from tech-spec (rationale):** Copy, don't move. The global command `/root/.claude/commands/generate-product-showcase-gallery.md` is registered as a slash command in global CLAUDE.md and is used for both simple_site and simple_dashboard. Deleting it would break that workflow. A symlink was considered and rejected — symlinks are unreliable across git clones. The project-level copy gives aisell-scope Claude sessions access to the skill without breaking global availability.

**Known limitation:** The project-level copy can get out of sync if the global command is updated. This is an accepted trade-off, documented in tech-spec. Future updates to the global command must be manually mirrored to this copy.
