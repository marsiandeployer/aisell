## Description

This task implements `GET /api/auth/invite/status?dashboardId=<id>` — the endpoint that enables returning guests to silently re-authenticate without going through Google OAuth again.

When a guest who has already registered (has an Ethereum keypair in ChatSettings and a record in `dashboard_access`) revisits `d{userId}.wpmix.net`, the auth widget script calls this endpoint via a CORS fetch with `credentials:'include'`. The endpoint reads the `webchat_session` cookie, looks up the session, loads the guest's ChatSettings, calls `signChallenge()` to produce a signed challenge, then calls `POST /api/auth/login` server-side to get a dashboard JWT, wraps it in a magic-link token, and returns `{ mlToken }`. The widget then reloads the page with `?ml=TOKEN` so the existing magic-link mechanism delivers the JWT into `sessionStorage.dashboard_jwt`.

**Why this endpoint must be on `simpledashboard.wpmix.net` (not inside `d*.wpmix.net` middleware):** The `webchat_session` cookie is scoped to `simpledashboard.wpmix.net` (set there by the webchat server). If the endpoint were inside the `d*.wpmix.net` middleware, the cookie would not be sent by the browser — it lives on a different origin. Cross-origin CORS with `credentials:'include'` from `d*.wpmix.net` to `simpledashboard.wpmix.net` is the correct mechanism (Decision 3 in tech-spec).

The endpoint must also handle CORS preflight (`OPTIONS`) requests from `d*.wpmix.net` origins with the correct headers: `Access-Control-Allow-Origin: <exact-origin>` and `Access-Control-Allow-Credentials: true`.

This task depends on Task 7 (JWT enforcement on `/api/data/`) and is in Wave 5 — tasks within Wave 5 that touch `webchat.ts` must be executed sequentially. The foundation from Task 3 (`signChallenge()`, extended `MagicLinkEntry` type, `readSessions`/`cleanupExpired` and `parseCookieHeader` utilities) must already be present.

## What to do

1. Register `GET /api/auth/invite/status` on the main app (`app.get(...)`) **before** the `d*.wpmix.net` middleware block (around line 3558 in webchat.ts). This ensures the endpoint is served from `simpledashboard.wpmix.net` and receives the `webchat_session` cookie.

2. Register a corresponding `OPTIONS /api/auth/invite/status` handler (CORS preflight) at the same location, before the `d*.wpmix.net` middleware.

3. For both handlers, validate that the `Origin` request header matches the pattern `^https?://d\d+\.(wpmix\.net|habab\.ru)$`. If it matches, respond with:
   - `Access-Control-Allow-Origin: <exact-origin-value>`
   - `Access-Control-Allow-Credentials: true`
   - `Access-Control-Allow-Methods: GET, OPTIONS` (for preflight)
   - `Access-Control-Allow-Headers: Content-Type` (for preflight)
   - Preflight returns HTTP 200 with empty body.

4. In the `GET` handler, read the `webchat_session` cookie via `parseCookieHeader(req.headers.cookie)`. If no cookie or the session is not found in `cleanupExpired(readSessions())`, return HTTP 401 `{ error: 'No valid session' }`.

5. Read `dashboardId` from `req.query.dashboardId`. Validate it matches `/^d\d+$/`. Extract the numeric `dashboardUserId` from it (e.g. `dashboardId = 'd123'` → `dashboardUserId = 123`). If invalid, return HTTP 400.

6. Load guest `ChatSettings` using `loadChatSettings(session.userId)`. If `chatSettings.ownerAddress` or `chatSettings.ownerPrivateKey` is absent, return HTTP 401 `{ error: 'No keypair for this guest' }`.

7. Call the Auth API to verify `dashboard_access`: `GET AUTH_API_URL/api/auth/access-list?dashboardId=<dashboardId>` with `Authorization: Bearer INTERNAL_API_KEY`. If the guest's email is not in the returned `emails` array, return HTTP 401 `{ error: 'No dashboard access' }`. If the Auth API call fails, return HTTP 503 `{ error: 'Auth service unavailable' }`.

8. Call `signChallenge(chatSettings.ownerPrivateKey, dashboardId)` (implemented in Task 3) to produce `{ challenge, signature }`.

9. Call `POST AUTH_API_URL/api/auth/login` with `{ challenge, signature, dashboardId }` where `dashboardId` is the value from `req.query.dashboardId` (same body format as in the OAuth callback handler from Task 6; the auth-api.ts login endpoint takes `{ signature, challenge, dashboardId }` — no `address` field). If login fails, return HTTP 503 `{ error: 'Auth service unavailable' }`.

10. Generate a magic-link token: `crypto.randomBytes(32).toString('hex')`. Store in `magicLinkTokens` with `{ userId: String(dashboardUserId), expires: Date.now() + 5 * 60 * 1000, dashboardJwt: <jwt-from-login> }` (using the extended `MagicLinkEntry` type from Task 3).

11. Return HTTP 200 `{ mlToken: <token> }`.

12. Add logging: `console.log('[AUDIT] invite/status: guestUserId=<session.userId> dashboardId=<dashboardId>')` on success.

**Auth API call for dashboard_access check:**
The endpoint `GET /api/auth/access-list?dashboardId=<id>` is added by Task 4 (auth-api.ts). It returns `{ emails: string[] }`. The guest's email is on the `WebSession` or `WebUser` object — load it from `readUsers().find(u => u.userId === session.userId)`.

**CORS origin validation:**
The allowed origin pattern is `/^https?:\/\/d\d+\.(wpmix\.net|habab\.ru)$/`. Use `req.get('Origin')` (Express method) or `req.headers.origin`. For allowed origins, reflect the exact origin value in `Access-Control-Allow-Origin` (do not use `*` — credentials require exact origin). Set `Vary: Origin` header.

**ml-token storage:**
Store with `userId` = the `dashboardUserId` extracted from `dashboardId` (i.e. the owner's userId, not the guest's). This matches how the existing `GET /api/auth/ml` handler resolves the user folder — it checks `entry.userId !== userId` where `userId` is the dashboard owner extracted from the Host header. For returning guest flow, `userId` in the mlToken entry must match the dashboard owner's numeric ID (the `d{N}` in the dashboardId).

**Edge cases:**
- `dashboardId` query param missing or not matching `/^d\d+$/` → return 400 `{ error: 'Invalid dashboardId' }`
- Guest's session is valid but the session's `userId` belongs to the dashboard owner themselves → still serve normally (owner requesting via CORS is unusual but harmless)
- Auth API returns non-200 for `access-list` (e.g. 404 dashboard not found) → treat as "no access" → 401 `{ error: 'No dashboard access' }` (not 503)
- Auth API unreachable (timeout/ECONNREFUSED) → 503
- `signChallenge` throws (invalid private key) → catch, log error, return 503

**Security notes (for security-auditor review):**
- Reflecting `Origin` header verbatim is correct for CORS-with-credentials; the allowlist regex prevents reflecting arbitrary origins
- The endpoint does not require `requireSessionApi` middleware (it reads the cookie directly) because it needs to work from a different origin context; session lookup is done manually
- The ml-token expires in 5 minutes (same as OAuth-generated tokens) to minimize exposure in access logs
- No rate limiter is specified in the tech-spec for this endpoint — add `rlOAuthCallbackIp10m` (20/10min/IP, created in Task 3) as a reasonable limit to prevent brute-force session enumeration
