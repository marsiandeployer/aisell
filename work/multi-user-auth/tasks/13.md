# Task 13: test_server_side_keypair.js

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Create `botplatform/tests/test_server_side_keypair.js` — a new integration test file that exercises the server-side keypair generation and OAuth callback flow implemented in Task 6 (`GET /api/auth/google-dashboard-callback`).

The file uses the `GOOGLE_AUTH_TEST_SECRET` bypass mechanism (same pattern as `test_google_auth.js`) to simulate Google OAuth callbacks without real Google credentials. The server accepts JWT tokens signed with `GOOGLE_AUTH_TEST_SECRET` instead of validating real Google ID tokens.

The test suite covers:
1. **New email first pass** — callback with a fresh email generates a keypair in ChatSettings, stores the ml-token with a pre-signed dashboard JWT, and returns an HTTP 302 redirect to `d{userId}.wpmix.net?ml=TOKEN`. The ml-token is then redeemable at `/api/auth/ml` and returns a valid `dashboardJwt`.
2. **Same email second pass (idempotency)** — the same email run through the callback again does NOT generate a duplicate keypair; the same `ownerAddress` is found in ChatSettings and the login flow proceeds normally, issuing a fresh JWT.
3. **Invalid invite token** — callback where state contains a non-existent/revoked invite token and the guest's email is NOT already in `dashboard_access` → redirect to `?error=no_access`.
4. **Auth API down during register** — a local mock HTTP server on a test port returns 503; `AUTH_API_URL` env overridden for this test case → redirect to `?error=service_unavailable`.
5. **`/api/auth/share` returns error** — mock server returns 500 on the share endpoint while register succeeds → redirect to `?error=service_unavailable` (not `no_access`).
6. **Returning guest via `/api/auth/invite/status`** — with a valid `webchat_session` cookie belonging to a guest who already has `dashboard_access`, the endpoint returns HTTP 200 with `{ mlToken }`.
7. **`/api/auth/invite/status` without session** — no cookie → HTTP 401.
8. **Unit: `signChallenge` ecrecover** — the challenge produced by `signChallenge(privateKey, dashboardId)` can be verified via `ethers.verifyMessage(challenge, signature) === address`.

This test file is the primary verification gate for Task 6 and is run as part of the Wave 7 parallel test execution.

## Details

**Files:**

**Dependencies:**

- Task 3 (`signChallenge` helper, `inviteTokens` Map) must be complete so the unit test and the invite-based scenarios work.
- Task 8 (`GET /api/auth/invite/status`) must be complete for tests 6 and 7.

**Mock Auth API approach:**
- Use Node.js built-in `http.createServer` on port 19095 (or any free port — pick dynamically via `net.createServer({ port: 0 })`).
- The mock server inspects the URL path:
  - For "Auth API down" test: all paths return 503.
  - For "share error" test: `/api/auth/register` returns 201 (success), `/api/auth/share` returns 500.
- Before calling the callback endpoint in these test cases, set `AUTH_API_URL` for the webchat process. However, since we cannot change the running process's env mid-test, the correct approach is: the test sends a special header or query param that the server uses to override `AUTH_API_URL` — BUT this requires server-side support. Check the tech-spec implementation: the callback handler reads `AUTH_API_URL` from `process.env` at call time. Therefore the mock server approach for integration tests requires either: (a) the server supports `X-Test-Auth-Api-Url` header (add this in Task 6 when `GOOGLE_AUTH_TEST_SECRET` is set), or (b) the server is restarted with a different `AUTH_API_URL`. Option (a) is cleaner for tests. Confirm Task 6 implementation supports this pattern and document accordingly.

**State parameter construction:**
- `state` = `Buffer.from(JSON.stringify({ redirect_to: 'dTEST_USER_ID.wpmix.net', invite: TOKEN, nonce: NONCE })).toString('base64')`
- `nonce` must match the nonce stored in the server's session for the test user. Since `GOOGLE_AUTH_TEST_SECRET` bypass is used, the nonce validation may be relaxed — check Task 6 implementation. If the callback stores the nonce in the session before the Google redirect, then the test must first hit the endpoint that initiates the OAuth (which stores the nonce) before calling the callback. Alternatively, if `GOOGLE_AUTH_TEST_SECRET` mode skips nonce validation, document that.

**ChatSettings file location for test 2 (idempotency check):**
- After test 1, read `botplatform/group_data/user_{userId}/settings.json` (this is where ChatSettings is stored — NOT `data/webchat/chats/{userId}.json` which is the chat transcript) and extract `ownerAddress` before and after the second callback call. Assert they are equal.
- Path resolution: `path.join(__dirname, '..', 'group_data', 'user_' + userId, 'settings.json')` from within the `botplatform/tests/` directory.

**Invite token setup:**
- Tests 1, 2, 4, 5 require a valid invite token for the test dashboard. Before each of these tests: POST `/api/auth/invite` with an owner session cookie to obtain a valid token.
- Owner session: POST `/api/auth/google` (using `GOOGLE_AUTH_TEST_SECRET` pattern from `test_google_auth.js`) for the owner email to obtain a session cookie, then POST `/api/auth/invite`. The owner's `ownerAddress` must be set (requires Task 3 keypair flow for the owner — the owner's dashboard must be initialized). If the owner doesn't have a keypair yet, POST `/api/auth/invite` will return 403. Handle this by checking the response and skipping if prerequisites are not met.

**Port selection for mock server:**
- Use port 19095 to avoid collision with real Auth API (8095). Or use `server.listen(0)` and read the assigned port dynamically from `server.address().port`.

**Edge cases:**
- `followRedirect: false` in HTTP helper: Node's `http.request` does not follow redirects by default, so a 302 response will have `Location` in headers without following it. Just do NOT use `fetch()` or any redirect-following client.
- `?error=` in redirect URL: the `Location` header may be URL-encoded; use `decodeURIComponent` when asserting.
- Multiple `set-cookie` headers: use the `extractSessionCookie` helper from `test_google_auth.js` — handles both array and single string forms.
- Race condition in mock server close: call `mockServer.close(callback)` and await in a Promise before proceeding to next test section.

**Implementation hints:**
- Follow `test_google_auth.js` structure exactly: harness at top, helpers section, `runTests(baseUrl, testSecret)` async function, `main()` entry point.
- All test data uses unique `runId = crypto.randomBytes(5).toString('hex')` prefix to avoid cross-run collisions.
- The mock server approach (test 4 and 5) requires cooperation from the implementation (Task 6). If Task 6 does not support `X-Test-Auth-Api-Url`, add a note in the test file explaining what server-side change is needed and create a stub that skips those two tests with `[SKIP]` until the server supports it. The test file must still exit 0 in that case.
