## Description

Write a new integration test file `botplatform/tests/test_guest_auth_widget.js` that verifies the auth widget injection logic, JWT enforcement on `/api/data/`, overlay error states, and CORS headers on the invite/status endpoint.

This task depends on Task 9 (auth widget script injection into d*.wpmix.net HTML). All the server-side behavior being tested ŌĆö HTML injection, JWT enforcement, overlay query-param handling, CORS on `/api/auth/invite/status` ŌĆö is already implemented by the time this task runs (wave 7, depends on 9).

The test file covers six behavioral groups:
1. **Auth widget presence** ŌĆö `<script id="auth-widget-loader">` is present in HTML served for a protected dashboard (one with `ownerAddress` set), and absent when `ownerAddress` is not set.
2. **No webchat navigation links** ŌĆö protected dashboard HTML does not contain "Back to chat", `href="/profile"`, or `href="/logout"` links (guests must not see webchat navigation).
3. **JWT enforcement on `/api/data/`** ŌĆö request without Authorization header ŌåÆ 401; request with a valid dashboard JWT (obtained via the `GOOGLE_AUTH_TEST_SECRET` callback bypass) ŌåÆ 200; request with a JWT that has a wrong `dashboardId` payload ŌåÆ 401.
4. **Error overlay states** ŌĆö GET `d{userId}.wpmix.net/?error=no_access` returns HTML containing a no-access overlay element; GET `?error=service_unavailable` returns HTML containing a service-unavailable overlay element.
5. **CORS on `/api/auth/invite/status`** ŌĆö a request with `Origin: https://d{userId}.wpmix.net` returns `Access-Control-Allow-Origin` header matching that origin (or a wildcard for `d*.wpmix.net`).

The test follows the same harness pattern as `test_google_auth.js`: plain Node.js `http`/`https` modules, `WEBCHAT_TEST_URL` env var for base URL (default `http://127.0.0.1:8094`), `GOOGLE_AUTH_TEST_SECRET` for JWT bypass, colored pass/fail output, exit 0 on all pass / exit 1 on any failure.

## Details

**Files:**

**Dependencies:**
- Task 9 must be complete: auth widget injection into d*.wpmix.net HTML.
- Tasks 7 (JWT enforcement on `/api/data/`) and 8 (`/api/auth/invite/status` + CORS) must also be complete.
- Task 3 (foundation: `inviteTokens`, `signChallenge`, `MagicLinkEntry.dashboardJwt`) must be complete.
- Task 6 (OAuth callback: keypair generation + ml-token with `dashboardJwt`) must be complete for the valid-JWT test in Section 3.

**How to get a valid dashboard JWT for Section 3:**
The `GOOGLE_AUTH_TEST_SECRET` bypass (used in `test_google_auth.js`) authenticates a user via `POST /api/auth/google`. This sets up a webchat session with `ownerAddress` via the keypair flow. For a guest dashboard JWT specifically, the test needs to simulate the OAuth callback path ŌĆö Task 6 adds a test hook to `GET /api/auth/google-dashboard-callback` that accepts `GOOGLE_AUTH_TEST_SECRET`-signed tokens (same pattern). The test can POST to that endpoint with a fake `state` and get an ml-token back, then exchange it via `GET /api/auth/ml?token=...` to receive `dashboardJwt`. Alternatively, if that route is complex, use `jsonwebtoken` directly: `jwt.sign({ dashboardId: 'd' + userId, address: '0x...' }, process.env.JWT_SECRET || 'magic-secret', { expiresIn: '5m' })` ŌĆö this mirrors what the server produces and is sufficient for testing the `/api/data/` JWT verification logic.

**How to get a protected user (with ownerAddress):**
Use a test user created via `POST /api/auth/google` with `GOOGLE_AUTH_TEST_SECRET` (same as `test_google_auth.js`), but the test also needs to trigger keypair registration so that `chatSettings.ownerAddress` is set for that user. Task 3 adds the `signChallenge` helper; Task 6's OAuth callback path triggers `POST /api/auth/register`. In test context, if `AUTH_API_URL` is accessible, the test can call `POST /api/auth/google-dashboard-callback` (with test-secret bypass) which will register the keypair automatically. If Auth API is not running in test environment, the test can directly write `chatSettings.ownerAddress` to the test user's settings file on disk (same approach `cleanupTestUser` uses in `test_google_auth.js`).

**Edge cases to cover:**
- Unprotected dashboard: a user whose `settings.json` does not have `ownerAddress` set ŌĆö widget must be absent.
- JWT with incorrect `dashboardId` (e.g., `d9999` on a request to `d{realUserId}.wpmix.net`) ŌĆö must return 401.
- CORS preflight (OPTIONS) for `/api/auth/invite/status` ŌĆö optionally verify `Access-Control-Allow-Methods` as well.

**Implementation hints:**
- Use `getWithHost(baseUrl, path, hostHeader, cookieHeader)` helper to set the `Host` header on requests (critical ŌĆö the d*.wpmix.net middleware uses `req.hostname` to decide which user folder to serve).
- The `Host` header on a localhost request must be exactly `d{userId}.wpmix.net` or `d{userId}.habab.ru` to trigger the middleware (see webchat.ts line 3563: `host.match(/^d(\d+)\.(wpmix\.net|habab\.ru)$/i)`).
- For overlay assertions, inspect what Task 9 actually injects into the HTML for `?error=no_access` and `?error=service_unavailable`. The widget script checks `new URLSearchParams(location.search).get('error')` client-side, but the test is server-side (checking the raw HTML). The server-side injection by Task 9 should embed the overlay markup statically in the HTML or as a hidden element shown by JS. Check the actual injected HTML for the correct element ID or text to assert against ŌĆö do not guess IDs.
- Follow the cleanup pattern from `test_google_auth.js`: `cleanupTestUser(email)` removes test data from `users.json`, `sessions.json`, chat transcript, and workspace directory.
