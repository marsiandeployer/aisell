## Details

**Test environment notes:**
- `simpledashboard-web` process (port 8094) must be running during tests — restart it if rate limit budget is exhausted from prior test runs.
- `dashboard-auth-api` (port 8095) must also be running — restart it independently if 429 errors appear mid-suite.
- Rate limit budgets are in-memory; restart both processes if tests fail with 429.
- Env var loading: `export $(cat .env.auth | xargs)` sets `GOOGLE_AUTH_TEST_SECRET`, `INTERNAL_API_KEY`, `JWT_SECRET`, `GOOGLE_CLIENT_SECRET`, and `DATABASE_URL` for test runners.
- `test_server_side_keypair.js` uses `GOOGLE_AUTH_TEST_SECRET` bypass (same pattern as `test_google_auth.js`) so it does not need real Google OAuth credentials.

**Not verifiable pre-deploy (defer to post-deploy-qa):**
- Real browser manual smoke test (OAuth with actual Google account + incognito window) — requires live production environment and a real Google OAuth session. Document these in `deferredToPostDeploy` section of qa-report.json.
- CORS preflight behavior in a real browser (can verify header presence via curl, but actual browser credential handling requires browser E2E).

**Edge cases to confirm tested:**
- `inviteTokens` Map keyed by userId (not IP): same userId from two different IPs → single rate-limit budget.
- `redirect_to` CSRF protection: value not matching `/^d\d+\.wpmix\.net$/` → rejected.
- OAuth state nonce mismatch → rejected before code exchange.
- Valid JWT but wrong `dashboardId` in payload → 401 on `/api/data/`.
- Auth API mock returning 503 → `?error=service_unavailable` (not `no_access`).

**Report output:** `logs/working/qa-report.json` per pre-deploy-qa skill format (status, summary counts, testSuite, acceptanceCriteria array, findings array, deferredToPostDeploy array).
