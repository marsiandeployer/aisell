# Task 4: auth-api.ts — GET /api/auth/access-list

## Required Skills

Before starting, load: `/skill:code-writing` — [code-writing/SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Add `GET /api/auth/access-list?dashboardId=<id>` to `botplatform/src/auth-api.ts`. This endpoint is protected by `INTERNAL_API_KEY` bearer token (same middleware already used by `POST /api/auth/register` and `POST /api/auth/share`).

The endpoint executes a JOIN query: `SELECT u.email FROM dashboard_access da JOIN users u ON da.address = u.address WHERE da.dashboard_id = $1` and returns `{ emails: string[] }`.

This endpoint serves the profile page's "Share Dashboard" section: when the owner opens `/profile`, the webchat server calls this endpoint server-side to build the list of guests who currently have access to the dashboard. Without this endpoint, the webchat server has no way to enumerate guest emails because it has no direct PostgreSQL connection — all DB operations go through auth-api.

This is a Wave 2 task (same wave as Task 3 webchat.ts foundation). It modifies a different file (`auth-api.ts`) so it can be executed in parallel with Task 3.

## What to do

1. Add `GET /api/auth/access-list` handler to `botplatform/src/auth-api.ts`, placed after the existing `POST /api/auth/share` endpoint (around line 487) and before the server startup block.

2. Protect the handler with `requireInternalApiKey` middleware (same as register and share endpoints).

3. Read `dashboardId` from query parameter `req.query.dashboardId`. Validate it is a non-empty string; return 400 if missing or invalid.

4. Execute SQL: `SELECT u.email FROM dashboard_access da JOIN users u ON da.address = u.address WHERE da.dashboard_id = $1` with `[dashboardId]` as parameter.

5. Return `{ emails: string[] }` — collect `rows.map(r => r.email as string)`. Return an empty array `{ emails: [] }` if no rows found (not 404).

6. Wrap the query in try/catch; on PG error log `[auth-api] Access-list PG error:` and return 503.

7. Add a JSDoc comment block above the handler following the same pattern as existing endpoints in the file.

## Details

**File: `botplatform/src/auth-api.ts`**

Current state: 496 lines. Endpoints present:

**Pattern to follow — POST /api/auth/share (lines 421-487):**
- JSDoc block with route, body, requires, returns
- `app.METHOD(path, requireInternalApiKey, async (req, res) => { ... })`
- Input validation with early returns and specific error messages
- `pool.query(...)` in try/catch
- `console.log(...)` on success, `console.error('[auth-api] ... PG error:', ...)` on failure
- 503 on PG error

**New handler specifics:**
- Method: `GET` (not POST — reads data)
- Input from `req.query.dashboardId` (not `req.body`) — `typeof req.query.dashboardId !== 'string'` check
- SQL is a two-table JOIN, not a single INSERT. No rate limiter needed (internal server-to-server call, protected by INTERNAL_API_KEY)
- Return shape: `res.json({ emails })` where `emails = result.rows.map(r => r.email as string)`
- No 404 for unknown dashboard — return `{ emails: [] }`. Rationale from tech-spec Decision 7: webchat server needs a list, not an existence check

**Existing infrastructure to reuse:**
- `requireInternalApiKey` middleware (line 208) — pass as second argument to `app.get()`
- `pool` (line 124) — use `pool.query()` directly
- Error pattern from share endpoint: `console.error('[auth-api] Share PG error:', (err as Error).message)` → copy pattern with "Access-list PG error"

**SQL query:**
```sql
SELECT u.email
FROM dashboard_access da
JOIN users u ON da.address = u.address
WHERE da.dashboard_id = $1
```
Tables: `dashboard_access` (columns: `dashboard_id`, `address`, `granted_by`) and `users` (columns: `address`, `email`, `private_key`). Both tables verified from existing queries in the file.

**Test file location:** `botplatform/tests/test_auth_api.js` — already exists with tests for register, login, share. Add new test functions at the end of the file following the existing `runTests()` pattern.

**Edge cases:**
- `dashboardId` provided as array in query string (e.g., `?dashboardId=a&dashboardId=b`) — `typeof req.query.dashboardId` will be `'object'`, not `'string'` → caught by the string type check → 400
- Empty string dashboardId `?dashboardId=` — `''` is falsy → caught by `!dashboardId` check → 400
- Dashboard exists but all guests were removed from `dashboard_access` — query returns 0 rows → `{ emails: [] }` is correct
