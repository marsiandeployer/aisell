# Task 11: test_invite_flow.js

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Create `botplatform/tests/test_invite_flow.js` — integration test suite for the invite generation and revocation system implemented in Task 5.

The tests cover the full invite lifecycle: generate → persist → overwrite → revoke → rate limit. They also verify the correctness of the `rlInviteUser1h` rate limiter key strategy (keyed by userId, not by IP), which is a deliberate architectural decision: multiple requests from the same user across different IPs share one budget, while two different users have independent budgets.

This test file does not use `GOOGLE_AUTH_TEST_SECRET` (that is used in Task 13). It works at the HTTP level: POST invite endpoints, read `invites.json` directly from disk to assert persistence, and use GET `/api/auth/invite/status` with an old token to verify 401. No PM2 restart is performed — the test reads the JSON file from disk directly.

Depends on Task 5 (`POST /api/auth/invite`, `POST /api/auth/invite/revoke` endpoints) being implemented.

## What to do

1. Create `botplatform/tests/test_invite_flow.js` as a new file.

2. Use the same test harness pattern as `test_google_auth.js`: colors, `assert(condition, description)`, `section(name)`, `passed`/`failed` counters, summary block, `process.exit(failed === 0 ? 0 : 1)`.

3. Use the same `httpRequest`/`post`/`get` HTTP helpers from `test_google_auth.js` (copy or reimplement inline — no external test framework).

4. Implement a helper that creates a session via `POST /api/auth/google` using `GOOGLE_AUTH_TEST_SECRET`, extracts the `webchat_session` cookie, and returns it. This provides an authenticated owner session for all invite endpoint calls. Use a unique runId email per test run.

5. Implement a second session helper that creates a second owner user (different email), so tests can verify userId-keyed rate limiting between two independent users.

6. Write the test sections described in the TDD Anchor section below.

7. Implement cleanup: after all tests, remove created users via `cleanupTestUser(email)` (same filesystem-based cleanup from `test_google_auth.js`).

## Details

**Files:**
- `botplatform/tests/test_invite_flow.js` — new file, create from scratch.

**HTTP helpers pattern (from test_google_auth.js):**
- `httpRequest(url, options)` — returns `{ status, headers, data }`. Handles both http and https. Parses JSON body. `rejectUnauthorized: false` for HTTPS.
- `post(baseUrl, path, body, extraHeaders)` — wraps httpRequest for POST with JSON body.
- `get(baseUrl, path, cookieHeader)` — wraps httpRequest for GET with optional Cookie header.
- Copy these helpers verbatim or adapt — no external npm test framework is used.

**Session creation pattern:**
- Call `POST /api/auth/google` with a `makeGoogleToken(testSecret, { email, name, email_verified: true })` payload. The server accepts this when `GOOGLE_AUTH_TEST_SECRET` matches.
- Extract `webchat_session` cookie from response `set-cookie` header using `extractSessionCookie(headers)`.
- The owner must have `chatSettings.ownerAddress` set (Task 5 returns 403 if not set). For the rate limit tests: either use a pre-seeded owner with ownerAddress, or adjust the test to account for 403 vs 429. Check what the server returns when ownerAddress is not set — the test should be structured to use an owner that has completed the keypair flow. If needed, note this as a prerequisite for the test environment.

**Invite token format:**
- 64 hex chars (`crypto.randomBytes(32).toString('hex')`). Validate with `/^[0-9a-f]{64}$/.test(token)`.

**Extracting token from URL:**
- Parse `url` with `new URL(url)`, then `searchParams.get('invite')`.

**invites.json path:**
- `botplatform/data/webchat/invites.json` relative to the repo root. Use `path.join(__dirname, '..', 'data', 'webchat', 'invites.json')` inside the test (test is in `botplatform/tests/`, so `..` is `botplatform/`). Read with `JSON.parse(fs.readFileSync(path, 'utf8'))`.

**Rate limit test — userId-keyed verification:**
- To test same-userId-different-IP: add `'X-Forwarded-For': '10.0.0.1'` header to requests for user A in the first half of the budget, then switch to `'X-Forwarded-For': '10.0.0.2'` for the remainder. The 21st request should still be 429.
- The server may not forward `X-Forwarded-For` to the rate limiter key if keyed strictly by userId. The test verifies the behavioral outcome: same userId always shares budget regardless of IP.
- For the second-user independence test: create a second Google OAuth session with a different email, then immediately call POST /api/auth/invite with that session after user 1 is rate-limited. Expect 200 (or 403 if ownerAddress missing — handle accordingly).

**Behavioral check for "old token → no_access":**
- GET `/api/auth/invite/status?dashboardId=d{userId}` with the old token value passed as query param or header — check the actual API signature from Task 8 implementation. If the endpoint checks the token from the invite, the old token should return 401.
- Alternatively, the test can assert that the token in `invites.json` after overwrite/revoke no longer matches the old token (filesystem assertion), and additionally call the status endpoint to confirm behavioral 401.

**Cleanup:**
- Use the same `cleanupTestUser(email)` pattern as `test_google_auth.js` — removes user from `users.json`, sessions from `sessions.json`, chat transcript, workspace directory.
- Also clean up entries from `invites.json` for the test userId: read the file, filter out test entries, write back atomically.
- Run cleanup in a `finally`-style block so it runs even if assertions fail.

**Dependencies:**
- Task 5 must be complete: `POST /api/auth/invite`, `POST /api/auth/invite/revoke` endpoints live.
- Task 3 must be complete: `inviteTokens` Map, `invites.json` persistence, `rlInviteUser1h` rate limiter.
- Task 8 must be complete (or partially complete): `GET /api/auth/invite/status` endpoint for the old-token-invalid assertions.
- `GOOGLE_AUTH_TEST_SECRET` must be set in the server process for session creation to work.

**Edge cases:**
- If `data/webchat/invites.json` does not exist before the first invite (first ever call): the test should handle `ENOENT` gracefully when reading the file before the first POST.
- Rate limit state is in-memory and resets on server restart. If the test is run repeatedly without restarting the server, the rate limit bucket from a previous run may carry over. Use a fresh owner email per run (using `runId = crypto.randomBytes(5).toString('hex')`) so each run uses a fresh userId.
