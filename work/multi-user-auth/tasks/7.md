# Task 7: JWT enforcement on /api/data/ in d*.wpmix.net

## Description

The `/api/data/` handler inside the `d*.wpmix.net` middleware currently allows unrestricted CRUD access to dashboard data collections. This task gates that access behind a `dashboard_jwt` JWT token on protected dashboards (those that have `chatSettings.ownerAddress` set).

When a dashboard has `ownerAddress` configured, every request to `/api/data/` must carry `Authorization: Bearer <dashboard_jwt>`. The server verifies the token using `jwt.verify(token, JWT_SECRET)` and additionally checks that `payload.dashboardId === "d" + userId` â€” so a guest JWT issued for one dashboard cannot be used to access another.

Dashboards without `ownerAddress` remain publicly accessible â€” enforcement is skipped entirely. This preserves backward compatibility for all existing unprotected dashboards.

This task is part of Wave 5 (sequential inside the wave). It depends on Task 6 (OAuth callback handler), which introduces the JWT delivery mechanism and the `chatSettings.ownerAddress` usage pattern for guests.


**Edge cases:**
- `Authorization` header present but malformed (e.g., no `Bearer` prefix, or empty token after split) â€” `jwt.verify()` will throw, which is caught and returns 401
- JWT signed with wrong secret â€” `jwt.verify()` throws â†’ 401
- JWT expired â€” `jwt.verify()` throws â†’ 401
- `chatSettings.ownerAddress` is an empty string `""` â€” treated as falsy, enforcement skipped (same as no `ownerAddress`)
- `userId` extracted from host regex is a string; `"d" + userId` string concatenation is straightforward

**Implementation hints:**
- Use `try { const payload = jwt.verify(...) as { dashboardId?: string }; } catch { return 401; }` â€” jwt.verify throws on any verification failure
- `req.headers['authorization']` is lowercase in Express regardless of what the client sends â€” safe to access directly
- Do not log the JWT token value in error messages (security)
- The existing `jwt` import and `JWT_SECRET` usage in `webchat.ts` is at line ~3805 (`GET /api/auth/ml` handler) â€” use identical pattern
