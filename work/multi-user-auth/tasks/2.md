# Task 2: CLAUDE.md.template — guest auth documentation

## Required Skills

Before starting, load: `/skill:documentation-writing` — [SKILL.md](/root/.claude/skills/documentation-writing/SKILL.md)

## Description

`products/simple_dashboard/CLAUDE.md.template` is the AI context file that gets copied into every user's workspace. It tells the AI assistant how to generate `index.html` dashboard code — including the auth section.

Currently the Auth section documents two access paths: Desktop (Chrome Extension) and Mobile (Magic Link). With the multi-user guest access feature, a third path is added: Guest Google OAuth. The AI-generated dashboard HTML must work correctly for all three paths, which means the AI needs to understand:

1. That guests authenticate via Google OAuth, not the Extension — so the `install-extension` overlay should not be shown to guests who do not have `window.ethereum`
2. That `sessionStorage.dashboard_jwt` can arrive via the existing magic-link mechanism after a Google OAuth callback, not only via `?ml=TOKEN` from the owner's phone
3. That the auth widget itself (`#auth-widget-loader` script and the Google login button) is injected by the server — the AI does not write it into `index.html`
4. What `authOverlay` and `authDataContainer` element IDs are used for (unchanged, already in template) and that they must remain stable because the server-injected widget references them
5. That `OWNER_ADDRESS` in the Claude prompt context is the **guest's own registered Ethereum address** (server-generated keypair), not an indicator that the guest is the owner — AI code must not confuse the two concepts
6. That `/api/data/` on a protected dashboard now requires `Authorization: Bearer <dashboard_jwt>` — the `fetchWithAuth()` helper handles this automatically

This documentation ensures that any AI-generated dashboard code (now and future) handles auth state correctly across all three paths without breaking existing owner flows.

**What the guest flow adds (key facts for documentation):**

- Existing magic-link script (server-injected, already documented) handles token exchange → stores `sessionStorage.dashboard_jwt`

- `/api/data/` now enforces `Authorization: Bearer <dashboard_jwt>` on protected dashboards. The existing `fetchWithAuth()` helper sends this header automatically. AI-generated code that already uses `fetchWithAuth()` needs no changes.

**Edge cases to note in documentation:**
- A guest who opens the dashboard without `?invite=` and without a valid session will see the auth widget with a "Sign in with Google" button — this is handled entirely by the server-injected widget, not by `initDashboardAuth()` or any code in `index.html`
- The `install-extension` overlay should NOT be shown to users who simply lack the Extension — in a guest context, absence of `window.ethereum` is expected. The server-injected auth widget handles this case with the Google button instead.
- Note: the `install-extension` overlay in the current code block IS still correct for the owner path (where the Extension IS expected). The documentation should clarify the flow difference.
