## Description

Add a "–ü–æ–¥–µ–ª–∏—Ç—å—Å—è –¥–∞—à–±–æ—Ä–¥–æ–º" section to the `/profile` page. This section lets the dashboard owner generate and share an invite link with guests, revoke it, and see the list of guests who already have access.

The section must appear between the existing `#mobile-section` block and the "Back to chat ¬∑ Logout" row, and must only render when `chatSettings.ownerAddress` is set (i.e., the owner has completed the keypair flow). For unprotected dashboards ‚Äî where `ownerAddress` is absent ‚Äî the section must be entirely absent from the rendered HTML.

Additionally, this task removes the current `ownerPrivateKey` display line from the profile HTML. The private key must never be sent to the browser; this is an explicit security requirement from the tech-spec acceptance criteria.

The guest email list is fetched server-side at render time via `GET /api/auth/access-list?dashboardId=d{userId}` (the endpoint added to `auth-api.ts` in Task 4), authenticated with `INTERNAL_API_KEY`. On any error the fallback is an empty list ‚Äî the page still renders without crashing.

## What to do

1. Make the `/profile` route handler `async` so it can await the server-side guest list fetch.

2. At the top of the handler, after loading `chatSettings`, fetch the guest email list from `GET {AUTH_API_URL}/api/auth/access-list?dashboardId=d{userId}` with `Authorization: Bearer {INTERNAL_API_KEY}`. Parse the response as `{ emails: string[] }`. On any network error, non-200 status, or parse failure, fall back to `[]` and log a warning. Skip the fetch entirely when `ownerAddress` is falsy.

3. Remove the `ownerPrivateKey` variable declaration and the HTML line that renders it:
   ```
   ${ownerPrivateKey ? `<div class="row">Private Key: <code>${escapeHtml(ownerPrivateKey)}</code></div>` : ''}
   ```

4. Add CSS rules for the new section inside the existing `<style>` block ‚Äî `.share-section` with the same border-top separator style as `.mobile-section`; `.btn-invite` with the same blue style as `.btn-qr`; `.btn-revoke` as a secondary outline button; `#inviteResult` hidden by default; `.guest-list` as an unstyled list with 13px items.

5. Insert the "–ü–æ–¥–µ–ª–∏—Ç—å—Å—è –¥–∞—à–±–æ—Ä–¥–æ–º" HTML block (wrapped in `ownerAddress ? ... : ''`) after the closing tag of `#mobile-section` and before the "Back to chat" row. The block contains: section heading, short description, "–°–æ–∑–¥–∞—Ç—å invite-—Å—Å—ã–ª–∫—É" button (`id="btnInvite"`), "–û—Ç–æ–∑–≤–∞—Ç—å —Å—Å—ã–ª–∫—É" button (`id="btnRevoke"`), a hidden `#inviteResult` div for the copyable URL, and a guest list subsection that renders the server-fetched emails (or "–ù–µ—Ç –≥–æ—Å—Ç–µ–π —Å –¥–æ—Å—Ç—É–ø–æ–º." when empty). 6. Add `createInvite()` and `revokeInvite()` JavaScript functions to the existing `<script>` block. Each function disables its button during the request, calls the respective endpoint, and on success populates `#inviteResult` with a copyable `<code>` element (same click-to-copy pattern as `#qrUrl`).
