## Description

This task adds two invite management endpoints to the `simpledashboard.wpmix.net` server in `webchat.ts`. These endpoints allow the dashboard owner to generate and revoke invite links that guests can use to register and gain access to the owner's dashboard.

Both endpoints build on the infrastructure added in Task 3: the `inviteTokens` Map, `writeInvites`/`readInvites` helpers, `rlInviteUser1h` rate limiter (20/hour/userId), and the `INVITES_PATH` constant pointing to `data/webchat/invites.json`.

**POST /api/auth/invite** — generates a new invite token for the authenticated owner. Returns `{ url: "https://d{userId}.wpmix.net?invite=TOKEN" }`. Overwrites any existing token in the Map and persists to disk. Rate-limited to 20 requests/hour per userId.

**POST /api/auth/invite/revoke** — regenerates the invite token for the authenticated owner. The old token immediately becomes invalid. Returns `{ url: "https://d{userId}.wpmix.net?invite=NEW_TOKEN" }`. Does not consume the rate limit budget (revoke is a separate action).

Both endpoints must return HTTP 403 if `chatSettings.ownerAddress` is not set — only owners who have completed the keypair flow can share their dashboard. Both require an active session via `requireSessionApi`.

These endpoints are registered on the main `simpledashboard.wpmix.net` app, before the `d*.wpmix.net` subdomain middleware, following the same pattern as `POST /api/auth/magic-link` at line 5210.

## What to do

1. In `botplatform/src/webchat.ts`, locate the block around line 5210 where `POST /api/auth/magic-link` is registered. Add the two new endpoints in the same area, after the magic-link handler and before `/profile`.

2. Implement `POST /api/auth/invite`:
   - Apply `requireSessionApi` middleware
   - Check `rlInviteUser1h` for the current userId; if over limit, return HTTP 429 with `{ error: 'Too many invites generated. Try again later.' }`
   - Consume the rate limit slot on success
   - If `chatSettings.ownerAddress` is falsy, return HTTP 403 with `{ error: 'Dashboard not protected. Complete keypair setup first.' }`
   - Generate token via `crypto.randomBytes(32).toString('hex')`
   - Call `inviteTokens.set(String(user.userId), token)` — overwrites any existing token
   - Persist by calling `writeInvites()` (writes the full Map to `invites.json`)
   - Return HTTP 200 with `{ url: \`https://d${user.userId}.wpmix.net?invite=${token}\` }`

3. Implement `POST /api/auth/invite/revoke`:
   - Apply `requireSessionApi` middleware
   - If `chatSettings.ownerAddress` is falsy, return HTTP 403 with the same error message as above
   - Generate a new token via `crypto.randomBytes(32).toString('hex')`
   - Call `inviteTokens.set(String(user.userId), token)` — old token is no longer in the Map
   - Call `writeInvites()` to persist the replacement
   - Return HTTP 200 with `{ url: \`https://d${user.userId}.wpmix.net?invite=${token}\` }`
   - Note: revoke does NOT consume the rate limit; it is a separate safety operation

4. Verify TypeScript compiles: run `npm run build` in `botplatform/` and confirm exit code 0.


**Dependencies:**
- Task 3 must be complete before this task (wave 3 depends on wave 2). Task 3 provides: `inviteTokens` Map, `writeInvites()`, `rlInviteUser1h`, `INVITES_PATH`.

**Edge cases:**
- Owner calls invite with no `ownerAddress` in chatSettings (keypair flow not completed) → 403
- Owner calls invite/revoke without session cookie → `requireSessionApi` returns 401 automatically
- Owner calls invite 20 times successfully, then calls revoke — revoke must still work (not rate-limited)
- Owner calls invite 20 times, then calls invite again → 429; then calls revoke → 200 (revoke bypasses rl)
- Two different users call invite simultaneously — each has their own key in `inviteTokens` Map, no conflict
- Calling invite twice in quick succession — second token overwrites first in Map and in `invites.json`

**Implementation hints:**
- Rate limiter pattern: `rlInviteUser1h.check(String(user.userId), Date.now())` → if `!rlCheck.ok` return 429 → then `rlInviteUser1h.consume(String(user.userId), Date.now())`
- `loadChatSettings(user.userId)` returns the ChatSettings object; check `.ownerAddress`
- `writeInvites()` is a helper added in Task 3 that serializes `inviteTokens` to `INVITES_PATH` via `writeJsonAtomic`
- Token URL format must exactly match: `https://d${user.userId}.wpmix.net?invite=${token}` (no trailing slash, no extra params)
- Keep the 403 guard AFTER the rate limit check for `/invite` — this way rate limit is consumed even if the user hasn't set up a keypair yet (prevents enumeration of ownerAddress state via rate limit side-channel). For `/revoke`, no rate limit, so order doesn't matter.
