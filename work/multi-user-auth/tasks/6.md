## Description

This task implements the central OAuth authorization-code callback handler for the guest auth flow. It is registered as `GET /api/auth/google-dashboard-callback` on `simpledashboard.wpmix.net` and is the single registered Google OAuth redirect URI for the feature.

When a guest clicks "Đ’ĐľĐąŃ‚Đ¸ Ń‡ĐµŃ€ĐµĐ· Google" on a dashboard page, the browser is redirected to Google with a `state` param containing `{ redirect_to, invite, nonce }`. After Google authentication, Google redirects back to this handler. The handler must:

1. Validate CSRF nonce against the webchat session and validate `redirect_to` against the allowlist `/^d\d+\.wpmix\.net$/`.
2. Exchange the authorization code for a Google ID token via `OAuth2Client.getToken(code)`, extract `email` and `name`.
3. Find or create a WebUser in users.json.
4. If the guest's email is not yet registered in PG (`users` table), generate an Ethereum keypair with `ethers.Wallet.createRandom()`, call `POST /api/auth/register`, and store `ownerAddress` + `ownerPrivateKey` in `ChatSettings`.
5. Validate the invite token from `state.invite`. If valid, call `POST /api/auth/share`. If already in `dashboard_access` (share is already granted), skip share. If neither condition holds, redirect with `?error=no_access`.
6. Call `signChallenge()` (from Task 3 foundation) with the guest's private key to produce `{ challenge, signature }`, then call `POST /api/auth/login` to get a `dashboardJwt`.
7. Store the dashboard JWT in a new `magicLinkTokens` entry with `expires: Date.now() + 5*60*1000` (5 minutes, shorter than the standard 24h). Redirect HTTP 302 to `https://{redirect_to}?ml=ML_TOKEN`.

This task also modifies `GET /api/auth/ml` (inside the `d*.wpmix.net` middleware) to detect `entry.dashboardJwt` and return it directly instead of re-signing a new JWT. This is the mechanism that delivers the pre-signed dashboard JWT to the guest browser.

Additionally, this task ensures the server embeds `window.__OAUTH_NONCE__` in injected HTML and stores the nonce in the webchat session entry before redirecting the user to Google â€” but the full widget injection HTML is handled in Task 9. Here, the nonce generation and session storage are needed so the callback can validate CSRF.

**Error paths (all result in HTTP 302 to `https://{redirect_to}?error=...`):**
- Invalid nonce or bad state JSON â†’ `?error=auth_failed`
- Invalid `redirect_to` domain â†’ log and return 400 (no safe redirect target)
- `OAuth2Client.getToken()` failure (bad/expired code) â†’ `?error=auth_failed`
- Auth API unavailable (network error, 5xx) â†’ `?error=service_unavailable`
- `POST /api/auth/share` returns non-200 â†’ `?error=service_unavailable` (NOT `no_access`)
- Invalid invite AND not in `dashboard_access` â†’ `?error=no_access`

**Logging requirements:**
- On keypair generation failure: `[ERROR] guest keypair generation failed for email=<email>`
- On successful guest registration (new keypair path): `[AUDIT] guest registered: email=<email> dashboardId=<id>`
- On successful guest login (both new and returning): `[AUDIT] guest login: email=<email> dashboardId=<id>`

**Rate limiting:** use the `rlOAuthCallbackIp10m` limiter (from Task 3, 20 requests per 10 minutes per IP) at the top of this handler.

## What to do

1. Add `GET /api/auth/google-dashboard-callback` route to `webchat.ts` (before the `d*.wpmix.net` middleware). Apply `rlOAuthCallbackIp10m` rate limiter keyed by client IP.

2. Parse and validate the `state` query parameter: JSON-decode it, extract `nonce`, `redirect_to`, `invite`. If JSON parse fails or `redirect_to` is missing, return HTTP 400 (no redirect target to use safely).

3. Validate `redirect_to` against `/^d\d+\.wpmix\.net$/`. If invalid, return HTTP 400 with a plain error response â€” do not redirect to an untrusted domain.

4. Load the webchat session from the `webchat_session` cookie. Validate that `state.nonce` matches the nonce stored in the session entry. If missing or mismatched, redirect to `https://{redirect_to}?error=auth_failed`.

5. Exchange the authorization code: use `new OAuth2Client(GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, REDIRECT_URI).getToken(code)`. Then call `client.verifyIdToken({ idToken: tokens.id_token, audience: GOOGLE_CLIENT_ID })` to extract `email` and `name`. On any failure, redirect to `?error=auth_failed`. Include `GOOGLE_AUTH_TEST_SECRET` bypass: if env var is set, accept an HMAC-signed JWT as `code` (same pattern as existing `POST /api/auth/google` handler at lines 4956-4992 of webchat.ts).

6. Find or create WebUser in users.json using the existing `readUsers`/`writeUsers` pattern (same as `POST /api/auth/google`).

7. Load ChatSettings for the WebUser. If `chatSettings.ownerAddress` is falsy (guest not yet registered in PG), generate keypair with `ethers.Wallet.createRandom()`, call `POST AUTH_API_URL/api/auth/register` with `{ address, privateKey, email, dashboardId }` and `Authorization: Bearer INTERNAL_API_KEY`. On failure, log `[ERROR] guest keypair generation failed for email=<email>` and redirect to `?error=service_unavailable`. On success, call `saveChatSettings()` with `ownerAddress` and `ownerPrivateKey`, then log `[AUDIT] guest registered: email=<email> dashboardId=<id>`.

8. Validate the invite token: look up `state.invite` in `inviteTokens` Map (keyed by `dashboardUserId` extracted from `redirect_to`). If the invite token matches, proceed to share step. If invite is absent or invalid, check whether the guest already has access by calling `GET AUTH_API_URL/api/auth/access-list?dashboardId=<id>` and checking if guest email is in the returned list. If neither condition holds, redirect to `?error=no_access`.

9. If invite is valid, call `POST AUTH_API_URL/api/auth/share` with `{ dashboardId, email, ownerAddress: chatSettings.ownerAddress }` and `Authorization: Bearer INTERNAL_API_KEY`. On any non-200 response or network error, redirect to `?error=service_unavailable`. A 404 from share (user not found) is also `service_unavailable` â€” the register step should have created the user; if it did not, something is wrong with Auth API.

10. Call `signChallenge(chatSettings.ownerPrivateKey, dashboardId)` (the helper added in Task 3). This returns `{ challenge, signature }`.

11. Call `POST AUTH_API_URL/api/auth/login` with `{ dashboardId, challenge, signature }` and `Authorization: Bearer INTERNAL_API_KEY`. On failure, redirect to `?error=service_unavailable`. Extract `dashboardJwt = response.token`.

12. Log `[AUDIT] guest login: email=<email> dashboardId=<id>`.

13. Generate `mlToken = crypto.randomBytes(32).toString('hex')`. Store `magicLinkTokens.set(mlToken, { userId: String(user.userId), expires: Date.now() + 5*60*1000, dashboardJwt })`. Note: the type of `magicLinkTokens` entries was extended in Task 3 to include optional `dashboardJwt?: string`.

14. Respond with `HTTP 302` â†’ `https://{redirect_to}?ml=${mlToken}`.

15. Modify the existing `GET /api/auth/ml` handler (lines 3792-3812 of webchat.ts, inside `d*.wpmix.net` middleware): after consuming the token, check if `entry.dashboardJwt` is present. If yes, return `{ jwt: entry.dashboardJwt }` directly â€” do not call `jwt.sign()`. If `entry.dashboardJwt` is absent, keep existing behavior (sign a new magic-type JWT).

16. Run `npm run build` in `botplatform/` to confirm TypeScript compiles with zero errors.

**State parameter encoding:**
The state param is produced by the client widget as `btoa(JSON.stringify({ redirect_to, invite, nonce: window.__OAUTH_NONCE__ }))`. The callback handler must decode with `JSON.parse(Buffer.from(state, 'base64').toString('utf8'))` and catch JSON parse errors.

**Nonce storage in session:**
Before the user is redirected to Google (in the widget construction step â€” handled in Task 9), the server must store the nonce in the session entry so the callback can retrieve it. Because session writes happen in Task 9, this task must read the nonce from whichever field the session entry uses. Coordinate the field name with Task 9 author; suggested field: `oauthNonce: string` on the session entry object. If Task 9 is not yet merged, add a stub: allow the callback to also pass if no session nonce is stored (for test-bypass mode via `GOOGLE_AUTH_TEST_SECRET`).

**`POST /api/auth/register` call pattern:**
Look at the existing `POST /api/auth/register-owner` handler (lines ~5097-5180 of webchat.ts) for how the server calls `AUTH_API_URL/api/auth/register` with `INTERNAL_API_KEY`. Replicate the same fetch pattern: `fetch(AUTH_API_URL + '/api/auth/register', { method: 'POST', headers: { 'Content-Type': 'application/json', Authorization: 'Bearer ' + INTERNAL_API_KEY }, body: JSON.stringify({ address, privateKey, email, dashboardId }) })`.

**`POST /api/auth/share` inputs:**
From auth-api.ts lines 429-487: requires `{ dashboardId, email, ownerAddress }` where `ownerAddress` is the OWNER's address (not the guest's). The owner's address is found in ChatSettings of the owner user. To find the owner: `dashboardId = "d" + dashboardUserId`, so `dashboardUserId` can be extracted from `redirect_to` (`redirect_to.match(/^d(\d+)\.wpmix\.net$/)![1]`). Load the owner's ChatSettings via `loadChatSettings(Number(dashboardUserId))`.

**Checking if guest already has access:**
Call `GET AUTH_API_URL/api/auth/access-list?dashboardId=<id>` (added in Task 4) with `Authorization: Bearer INTERNAL_API_KEY`. Parse the response `{ emails: string[] }` and check if `normalizedEmail` is in the list. If the Auth API call fails, treat as "no access" to be safe (fall through to `?error=no_access` path).

**Rate limiter â€” token test-bypass:**
The `GOOGLE_AUTH_TEST_SECRET` bypass skips Google token verification but still goes through all other validations (nonce, invite, Auth API calls). In tests, the nonce validation can be bypassed by checking `if (testSecret && state.nonce === 'test-nonce') { /* skip session nonce check */ }` â€” document this bypass clearly.

**Error handling pattern:**
Wrap the entire handler body in a single `try/catch`. On caught exception, check whether `safeRedirectTo` was already validated; if yes, redirect to `?error=service_unavailable`; if not, return HTTP 500. This ensures no unhandled rejection even if `redirect_to` validation itself throws.

**Edge cases:**
- Guest tries to authenticate to a dashboard that has no `ownerAddress` set: the owner hasn't completed keypair setup, so `POST /api/auth/share` will fail with 404 (dashboard not found in PG `dashboards` table) â†’ redirect to `?error=service_unavailable`.
- Race condition: two guests register simultaneously. Each independently generates a keypair and calls `POST /api/auth/register`. The second call may succeed or fail depending on whether the email already exists in PG `users`. If the second call fails (email already has a different address), log the error and use the ChatSettings address (which was written by the first concurrent request). In practice, `POST /api/auth/google-dashboard-callback` is per-user (unique Google login) so true concurrent same-email registration is extremely rare; handle gracefully, do not panic.
- `ethers.Wallet.createRandom()` is synchronous and cannot throw in normal operation. Still wrap in try/catch because the subsequent Auth API call can fail.
- The 5-minute ml-token TTL means the guest must complete the redirect quickly. The browser redirect chain (callback â†’ dashboard URL â†’ magic-link exchange) happens in under a second, so 5 minutes is generous. Do not extend TTL.

**Test mock server pattern:**

Shut the server down after the test. Do not use `nock` (not installed). Look at how `test_auth_api.js` starts the server for its own tests.
