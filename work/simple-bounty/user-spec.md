# User Spec: simple-bounty

## Что делаем

Новый продукт в экосистеме Simple* — фабрика bounty-платформ. Любой пользователь через webchat (AI-диалог) создаёт свою bounty-кампанию с произвольными заданиями, пополняет эскроу поинтами и публикует её на поддомене `d{userid}.wpmix.net`. Участники авторизуются, загружают доказательства выполнения (текст или URL) и получают поинты. В будущем поинты можно привязать к любой внешней сети — Telegram Stars, токены, фиат — как отдельный модуль.

## Зачем

Проекты (крипто, стартапы, инфлюенсеры, обычные люди) хотят привлечь аудиторию и вознаградить участников без разработки. Bounty — это community building + геймификация + ожидание роста ценности награды. Эскроу внутри платформы создаёт доверие: участник знает, что деньги зарезервированы до выплаты.

## Как должно работать

### Создатель кампании (через webchat)

1. Открывает `simplebounty.wpmix.net`, авторизуется через Google или Telegram
2. Ведёт диалог с AI: задаёт название кампании, описание, добавляет задания с наградами в поинтах
3. Пополняет эскроу — вводит количество поинтов
4. Публикует кампанию → получает ссылку `d{userid}.wpmix.net`
5. В панели управления видит список заявок: статус, доказательство (текст/URL), кнопки «Одобрить» / «Отклонить»
6. Может запустить несколько кампаний — у каждой свой отдельный эскроу

### Участник (на странице кампании)

1. Открывает `d{userid}.wpmix.net`
2. Авторизуется через Google или Telegram
3. Видит список заданий с описанием и наградой в поинтах
4. Загружает доказательство: текст или URL (ссылка на пост, твит, фото)
5. Один участник — одна заявка на задание. Повтор заблокирован
6. Заявка получает статус **pending**
7. При следующем открытии статуса участником система проверяет: прошло ли 48 часов?
   - Прошло + эскроу > 0 → **auto-approve**: поинты начисляются автоматически
   - Прошло + эскроу = 0 → статус остаётся pending, создатель получает уведомление «Пополните эскроу»
   - Не прошло или создатель уже ответил → текущий статус
8. Лидерборд кампании: топ-10 участников по сумме начисленных поинтов

### Уведомления создателя

- Новая заявка → значок в веб-панели + сообщение в Telegram боте (если подключён)
- Auto-approve сработал → сообщение в Telegram боте
- Эскроу пуст → предупреждение в панели

## Критерии приёмки

- [ ] AC-1: Создатель создаёт кампанию через webchat AI-диалог → кампания доступна на `d{userid}.wpmix.net`
- [ ] AC-2: Участник авторизуется (Google или Telegram) и видит список заданий кампании
- [ ] AC-3: Участник загружает доказательство (текст или URL) → статус заявки "pending"
- [ ] AC-4: Создатель нажимает «Одобрить» → поинты списываются из эскроу, начисляются участнику
- [ ] AC-5: Участник проверяет статус спустя 48ч, эскроу > 0, создатель не ответил → статус "approved", поинты начислены
- [ ] AC-6: Участник проверяет статус спустя 48ч, эскроу = 0 → статус "pending", создатель получает уведомление "Пополните эскроу"
- [ ] AC-9: Повторная заявка по тому же заданию возвращает ошибку 409 с сообщением «Вы уже отправили заявку на это задание»
- [ ] AC-10: При пустом эскроу кампании создатель видит предупреждение в панели управления
- [ ] AC-11: Лидерборд показывает топ-10 участников кампании по сумме начисленных поинтов, обновляется при каждом одобрении
- [ ] AC-12: Участник нажимает «Войти через Telegram», авторизуется через Telegram Login Widget → получает JWT и видит список заданий кампании
- [ ] AC-13: Один создатель может иметь несколько кампаний с независимыми эскроу
- [ ] AC-14: Если создатель удаляет задание с pending заявками — все pending заявки автоматически отклоняются, участники видят статус "rejected"

## Ограничения

- **Валюта MVP:** только поинты (числа в JSON). Telegram Stars, токены, фиат — вне скоупа MVP, добавляются как отдельные модули в будущем.
- **Доказательства:** только текст или URL. Загрузка файлов вне скоупа.
- **Auto-approve:** lazy-check при запросе участника, не cron. Срабатывает когда участник открывает статус заявки.
- **Telegram Mini App:** вне скоупа MVP.
- **1 submission:** один участник — одна заявка на одно задание.
- **Хранилище:** JSON per-user через существующий `/api/data/` CRUD. Отдельный `escrow.json` на каждую кампанию.
- **Домен:** `d{userid}.wpmix.net` — wildcard уже настроен, дополнительный nginx не нужен.

## Риски

- **Race condition эскроу.** Одновременное одобрение двух заявок при остатке на одну. **Митигация:** read-after-write проверка через `writeJsonAtomic` — если баланс ушёл в минус, последнее списание откатывается. Accepted limitation для MVP при низком трафике.
- **Lazy auto-approve.** Если участник не открывает страницу — auto-approve не срабатывает. **Митигация:** для MVP приемлемо; участник мотивирован проверять статус, чтобы получить поинты.
- **Telegram Login Widget на поддоменах.** Widget может требовать явного домена в настройках бота. **Митигация:** при создании бота указать `*.wpmix.net` или использовать один фиксированный домен для auth-колбэка.

## Технические решения

- Новый продукт `products/simple_bounty/` с `product.yaml` и `start-webchat-simplebounty.sh`. PM2 процесс `simplebounty-web` на порту **8097**.
- Хранилище — JSON per-user через существующий `/api/data/` CRUD: `campaigns.json`, `tasks.json`, `submissions.json`, `escrow/{campaignId}.json`.
- Новый endpoint `POST /api/auth/telegram` в `dashboard-auth-api` с HMAC-SHA256 верификацией через встроенный `crypto` (новые npm-пакеты не нужны).
- Auto-approve — lazy-check при запросе статуса участником: если `submittedAt + 48h <= now` и `status === 'pending'` и `escrow.balance >= task.reward` → approve.
- Лидерборд — агрегация из `submissions.json`: sum approved rewards per userId, топ-10.
- Нет claim-шага: участник сразу сабмитит доказательство.
- Дизайн каждой кампании — уникальный AI-фон и визуальный стиль, генерируется через webchat как у других Simple* продуктов.
- **Не делаем в MVP:** Telegram Stars, Telegram Mini App, загрузка файлов, фиат, даты окончания кампании.

## Тестирование

**Unit-тесты:** делаются всегда.
- Эскроу: debit/credit, проверка баланса, атомарность записи
- Auto-approve: `submittedAt + 48h <= now` при разных состояниях эскроу
- Duplicate submission: блокировка повторной заявки

**Интеграционные тесты:** делаем — новый auth flow критичен.
- Telegram Login: HMAC верификация (валидный / невалидный / просроченный)
- Создание кампании через webchat API
- Полный цикл эскроу: пополнение → approve → баланс уменьшился

**E2E тесты:** делаем — L-размер, новый продукт.
- Создатель создаёт кампанию → участник сабмитит → создатель апрувит → поинты начислены
- Auto-approve: установить `submittedAt = now - 49h` → участник открывает статус → approved

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Создать кампанию через webchat API | curl POST /api/bounty/campaigns | 200 OK, campaign_id |
| 2. Страница кампании доступна | curl GET d{userid}.wpmix.net | 200 OK, задания в HTML |
| 3. Telegram Login | curl POST /api/auth/telegram с тестовыми данными | JWT токен |
| 4. Подать заявку | curl POST /api/bounty/submissions | статус "pending" |
| 5. Повторная заявка | curl POST /api/bounty/submissions (тот же task+user) | 409 Conflict |
| 6. Одобрить заявку | curl POST /api/bounty/submissions/:id/approve | эскроу уменьшился, поинты участника увеличились |
| 7. Auto-approve (мок времени) | установить submittedAt = now−49h, GET /api/bounty/submissions/:id | статус "approved" |
| 8. Approve при пустом эскроу | approve при balance=0 | 402 с текстом ошибки |
| 9. Лидерборд | GET /api/bounty/campaigns/:id/leaderboard | список участников с поинтами |

### Пользователь проверяет

- Открыть `d{userid}.wpmix.net` → войти через Telegram → убедиться что задания и лидерборд видны
- В панели создателя: одобрить заявку → убедиться что баланс эскроу уменьшился
- Создать кампанию через webchat → проверить что страница имеет уникальный AI-фон и визуальный стиль
