# Task 10: Unit tests

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Write a standalone unit test suite for `bounty-api.ts` business logic. The tests run without a live HTTP server and without touching the real filesystem — all file I/O from `bounty-api.ts` is replaced with in-memory mocks.

The suite covers the critical paths of the bounty system: escrow debit/credit accounting (including boundary conditions and rollback), auto-approve timer logic (boundary conditions at exactly 48h, just under, just over, and zero-escrow guard), cold-start behavior when escrow file is absent, duplicate submission prevention, cascade-reject on task delete, leaderboard aggregation correctness, and the reject flow (no side-effects on escrow or balances).

This is the last coding task before integration tests (Task 11). Its purpose is to lock down the core logic contracts so that integration and E2E tests can focus on HTTP wiring and end-to-end flows rather than re-proving business rules.

## What to do

1. Create `botplatform/tests/test_bounty_unit.js` as a new file.

2. Implement a test harness that matches the existing project style from `test_sdk_methods.js`: color-coded `[PASS]`/`[FAIL]` output, `assert(condition, description)`, named section banners, a failures array, and a final summary that calls `process.exit(1)` if any assertion failed.

3. Extract (or import) the pure business-logic functions from `botplatform/src/bounty-api.ts` into the test scope. If those functions are not exported, replicate the logic inline as testable pure functions — the goal is to test logic, not HTTP routes. Mock `readJsonFile` and `writeJsonAtomic` with in-memory stores.

4. Implement the following test groups (each as a named `async function`):

   - **Group 1: Escrow debit — balance conditions**
     - balance > reward → success, balance decremented, transaction appended with correct `initiatedBy`
     - balance === reward → success (boundary: debit clears escrow to exactly 0)
     - balance < reward → error returned, balance unchanged, no transaction written

   - **Group 2: Escrow debit rollback**
     - Simulate post-write balance < 0 (concurrent write race): rollback restores previous balance, function returns error

   - **Group 3: Balances credit accumulation**
     - First approve: balance goes from 0 (cold start) to reward amount
     - Second approve on same participantId: balance accumulates (prevBalance + reward = newBalance)

   - **Group 4: Auto-approve timer**
     - `submittedAt + 48h === now` (exactly 48h) → approved, debit+credit executed
     - `submittedAt + 49h` + `escrow.balance >= reward` → approved
     - `submittedAt + 47h 59m` → pending, no state change
     - `submittedAt + 49h` + `escrow.balance = 0` → pending, no debit attempted

   - **Group 5: Escrow cold-start**
     - `readJsonFile` returns ENOENT error → escrow initializes with `balance: 0` and empty `transactions: []`

   - **Group 6: Duplicate submission**
     - First submission (campaignId + taskId + participantId) → created
     - Second submission same key → returns 409 / duplicate error, submissions array length unchanged

   - **Group 7: Task delete cascade**
     - Task has 2 pending submissions → delete task → both submissions status changed to `rejected`
     - Already-approved submissions are not changed by cascade

   - **Group 8: Leaderboard aggregation**
     - Build a submissions array with mixed statuses (approved, pending, rejected), multiple participants, multiple amounts per participant
     - Assert: only approved counted; sums are correct per participantId; sorted descending by totalPoints; result sliced to top-10; no email field in output

   - **Group 9: Reject flow**
     - Submit → reject → assert status = `rejected`; escrow unchanged; balances unchanged

5. Wire all groups into `main()`, call sequentially, print summary.

## Details

**Files:**
- `botplatform/tests/test_bounty_unit.js` (new) — the complete standalone unit test suite

**Dependencies:**
- Task 9 (Creator management panel) must be done; `bounty-api.ts` must be fully implemented before writing tests against its exported functions
- No new npm packages. Use only Node.js builtins: `assert` (optional, but prefer custom `assert()` matching project style), no `jest`/`mocha`/`tap`

**How to mock the filesystem:**

`bounty-api.ts` uses `readJsonFile` and `writeJsonAtomic` (imported from or defined in `webchat.ts`). In the unit test, either:

Option A — if functions are exported: import them and monkey-patch `readJsonFile`/`writeJsonAtomic` before each test group, restore after.

Option B — if functions are not exported: replicate the pure logic (debit, credit, autoApproveCheck, leaderboard aggregate) as local pure functions in the test file, passing the in-memory store as arguments. This is the cleaner approach and avoids fragile patching.

Regardless of approach, each test group must start with a fresh in-memory state (a plain JS object reset at the top of each group function).

**Edge cases to get right:**

- Exactly 48h boundary for auto-approve: `submittedAt + 48 * 60 * 60 * 1000 <= Date.now()`. Use a fixed `now` constant in tests to avoid flakiness.
- Rollback test: simulate by having the mocked `writeJsonAtomic` succeed but the subsequent `readJsonFile` (post-write read) return a value that shows balance < 0. The rollback must write the old value back.
- Leaderboard top-10 slice: create exactly 12 participants (each with distinct totalPoints) and assert `result.length === 10` and `result[0].totalPoints > result[9].totalPoints`.
- Task delete cascade: `rejected` should only apply to `pending` submissions; already-`approved` or already-`rejected` submissions keep their current status.
- Cold-start ENOENT: the mock `readJsonFile` throws an error with `code: 'ENOENT'`. The bounty logic must catch this and treat it as empty escrow.

**Test file structure** (follow `test_sdk_methods.js` style):

```
'use strict';

// COLORS, assert(), section(), passed/failed/failures counters
// in-memory mock helpers
// Group functions: testEscrowDebit(), testBalancesCredit(), testAutoApprove(), ...
// main(): run all groups, print summary, process.exit(failed > 0 ? 1 : 0)
main().catch(...)
```
