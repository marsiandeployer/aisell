# Task 4: Submissions API with auto-approve

## Required Skills

Перед выполнением задачи загрузи:

## Description

Добавить в `botplatform/src/bounty-api.ts` три группы эндпоинтов для работы с заявками участников (submissions): подача, проверка статуса с lazy auto-approve, и ручное одобрение/отклонение создателем.

Задача закрывает главную бизнес-логику bounty-платформы — участник подаёт доказательство выполнения задания, и оно либо одобряется вручную создателем, либо автоматически по истечении 48 часов если создатель не ответил (при наличии средств в эскроу).

**Ключевые security-требования, нарушение которых недопустимо:**
- `participantId` ВСЕГДА берётся из Google `sub` поля, хранящегося в server-side сессии после OAuth (webchat.ts google-dashboard-callback, ~line 6039-6041). Принимать participantId из тела запроса (body) запрещено — это IDOR-уязвимость.
- Поле `proof` HTML-entity кодируется перед сохранением в JSON: `<` → `&lt;`, `>` → `&gt;`, `&` → `&amp;`, `"` → `&quot;`. XSS-prevention.
- URL в proof принимается только с схемой `http://` или `https://`. Иные схемы (`javascript:`, `data:`) отклоняются с 400.
- Ownership check при approve/reject: `submission.campaignId` → читаем кампанию → `campaign.creatorId === req.webUser.userId`, иначе 403. Проверка обязательна.
- Duplicate block: комбинация `(campaignId, taskId, participantId)` уникальна — повторный сабмит → 409.

**Auto-approve (lazy):** вызывается в GET `.../submissions/my` как side-effect. Условие: `status === 'pending'` AND `submittedAt + 48h <= now` AND `escrow.balance >= task.reward`. При срабатывании: вызвать уже реализованную в Task 3 функцию `debitEscrow` с `initiatedBy: 'auto-approve'`, записать поинты в `balances.json`, поменять `status → 'approved'`, сохранить `reviewedAt`, `pointsAwarded`.

**Балансы** (`balances.json`) обновляются атомарно через `writeJsonAtomic` при каждом approve (ручном и auto): `balances[participantId] = (balances[participantId] || 0) + task.reward`.

## What to do

1. **POST `/api/bounty/campaigns/:id/tasks/:taskId/submissions`** — подача заявки участником.
   - Требует аутентификацию участника (Google OAuth session; проверить наличие `participantId` в сессии).
   - `participantId` извлекается из server-side сессии (`req.session.googleSub` или аналог), НЕ из req.body.
   - Проверить что кампания существует и имеет `status === 'published'`.
   - Проверить что задание `taskId` принадлежит этой кампании.
   - Проверить уникальность: если в submissions.json уже есть запись с `campaignId + taskId + participantId` → 409 `{ error: 'Вы уже отправили заявку на это задание' }`.
   - Если `proof` является URL (начинается с `http://` или `https://`) — разрешить. Если начинается с иного (`javascript:`, `data:`, пустая строка) — 400.
   - HTML-entity encode поле `proof` перед записью.
   - Создать Submission объект со `status: 'pending'`, `submittedAt: nowIso()`, сохранить в submissions.json через `writeJsonAtomic`. Вернуть 201 с объектом.

2. **GET `/api/bounty/campaigns/:id/tasks/:taskId/submissions/my`** — статус заявки + lazy auto-approve.
   - Требует аутентификацию участника.
   - Найти submission по `campaignId + taskId + participantId` (из сессии). Если не найден → 404.
   - Если `status === 'pending'` AND `Date.now() - Date.parse(submittedAt) >= 48 * 60 * 60 * 1000` → проверить auto-approve:
     - Загрузить задание, получить `task.reward`.
     - Вызвать `debitEscrow(campaignId, task.reward, submissionId, 'auto-approve')` из Task 3.
     - Если debit успешен: обновить `balances.json` атомарно, обновить submission: `status = 'approved'`, `reviewedAt = nowIso()`, `pointsAwarded = task.reward`. Сохранить submissions.json.
     - Если `debitEscrow` вернул ошибку (balance = 0 или < reward) → статус остаётся `pending`, логируем предупреждение. Уведомить создателя (через bounty-notifications.ts) о пустом эскроу.
   - Вернуть 200 с актуальным объектом submission.

3. **POST `/api/bounty/submissions/:submissionId/approve`** — ручное одобрение создателем.
   - Требует webchat session создателя (`requireSessionApi`).
   - Загрузить submission по `submissionId`. Если не найден → 404.
   - Ownership check: загрузить кампанию по `submission.campaignId`, проверить `campaign.creatorId === req.webUser.userId`. Иначе → 403.
   - Если `submission.status !== 'pending'` → 409 `{ error: 'Заявка уже обработана' }`.
   - Загрузить задание, получить `task.reward`. Вызвать `debitEscrow(campaignId, task.reward, submissionId, 'creator')`.
   - Если debit неуспешен (недостаточно средств) → 402 `{ error: 'Недостаточно средств в эскроу' }`.
   - Обновить `balances.json` атомарно. Обновить submission: `status = 'approved'`, `reviewedAt`, `pointsAwarded`. Сохранить submissions.json. Вернуть 200.

4. **POST `/api/bounty/submissions/:submissionId/reject`** — ручное отклонение создателем.
   - Требует webchat session создателя (`requireSessionApi`).
   - Ownership check аналогично approve.
   - Если `submission.status !== 'pending'` → 409.
   - Обновить submission: `status = 'rejected'`, `reviewedAt = nowIso()`. Эскроу и balances.json НЕ изменяются. Вернуть 200.

5. **Вспомогательная функция `creditBalance(creatorId, participantId, points)`** — читает `balances.json` для данного `creatorId`, прибавляет `points` к `balances[participantId]`, записывает через `writeJsonAtomic`.

6. Убедиться, что все новые эндпоинты смонтированы в роутере, экспортированном из `bounty-api.ts` и уже подключённом в `webchat.ts` (Tasks 2-3 это сделали).

## Details

**Files:**

- `botplatform/src/bounty-api.ts` — единственный файл для изменений. Добавить три группы эндпоинтов (POST submit, GET my-status, POST approve/reject) и вспомогательную функцию `creditBalance`. Задача 2 создала роутер и CRUD кампаний/заданий; задача 3 добавила `debitEscrow` и escrow endpoints. Всё добавляется в тот же файл и роутер.

**Как хранится participantId (критично для безопасности):**

В `webchat.ts` Google OAuth callback (line 6039-6041) извлекает `dashboardUserId` из redirect_to домена. Для SimpleBounty участника `participantId` — это поле `sub` из Google ID token payload. Это значение сохраняется в server-side сессии при OAuth.

В bounty-api.ts при обработке запросов участника: `const participantId = (req as any).bountyParticipantSub` или аналогичное поле, установленное middleware после верификации Google session cookie. Этот идентификатор НИКОГДА не читается из `req.body`. Если `participantId` отсутствует в сессии → 401.

Порядок чтения сессии участника: middleware декодирует webchat_session cookie → достаёт Google sub → записывает в `req.bountyParticipantSub` и `req.bountyParticipantName`. Этот sub и есть participantId.

**Хранилище данных:**

Все JSON-файлы хранятся в `group_data/user_{creatorId}/data/`:
- `submissions.json` — массив `Submission[]`
- `balances.json` — `Record<participantId, number>` (суммарные поинты)
- `escrow_{campaignId}.json` — уже управляется функцией `debitEscrow` из Task 3

Читать через `readJsonFile(path, [])` или `readJsonFile(path, {})`. Писать через `writeJsonAtomic(path, data)`.

**HTML-entity encoding:**

Реализовать функцию `htmlEncode(str: string): string` — замена `&`, `<`, `>`, `"`, `'` на соответствующие HTML-entities. Применять к полю `proof` перед записью. Это единственная защита от XSS, никакого sanitize-html пакета не нужно.

**URL validation:**

Если `proof` начинается с `http://` или `https://` — URL. Проверить схему через `new URL(proof).protocol` или простой regex `^https?://`. Если схема не `http:` или `https:` → 400. Пустая строка — текстовое доказательство, HTML-encode и сохранить. Нет минимальной длины.

**Auto-approve — граничные условия:**

- `submittedAt + 48 * 3600 * 1000 <= Date.now()` — условие (строгое <=, не <). 48h ровно → approve.
- Если `escrow.balance < task.reward` → не approve, не debit. Статус остаётся pending.
- Если `escrow.balance >= task.reward` → вызвать `debitEscrow`, затем `creditBalance`. Если debit вернул ошибку (race condition rollback) → статус pending, ошибку логировать.
- Auto-approve выполняется максимум один раз (повторный GET уже видит `status !== 'pending'`).

**Dependencies:**

- Task 2: campaigns и tasks CRUD (bounty-api.ts существует, роутер экспортирован, смонтирован в webchat.ts)
- Task 3: `debitEscrow(campaignId, amount, ref, initiatedBy)` функция существует в bounty-api.ts; escrow deposit и GET balance endpoints работают
- Нет новых npm-пакетов — только встроенные Node.js

**Edge cases:**

- Submission для несуществующей кампании → 404
- Submission для задания, не принадлежащего данной кампании → 404
- Submission для unpublished кампании (`status !== 'published'`) → 400
- Approve/reject уже обработанной заявки (`status !== 'pending'`) → 409
- balances.json отсутствует на диске (cold start) → `readJsonFile(path, {})` вернёт `{}`, creditBalance создаст файл

**Implementation hints:**

- `writeJsonAtomic` делает `fs.renameSync` (атомарный на Linux) — использовать для всех записей submissions.json и balances.json
- Для деления `creatorId` и `participantId` пространства: balances.json лежит в папке `group_data/user_{creatorId}/data/balances.json`, где `creatorId` — владелец кампании
- Не мутировать исходный массив submissions.json в памяти без последующей записи — читать свежую копию перед каждой операцией
