## Description

Создаём опциональный модуль Telegram-уведомлений для создателей bounty-кампаний. Модуль полностью отдельный от основной бизнес-логики: если `SIMPLEBOUNTY_BOT_TOKEN` не задан — весь код уведомлений просто не активируется. Ошибки внутри уведомлений никогда не прерывают основной HTTP-ответ (best-effort).

Три события требуют уведомлений:
1. Участник подал новую заявку → создатель получает сообщение с кратким описанием задания и сабмита.
2. Auto-approve сработал → создатель получает сообщение с именем участника, названием задания и начисленными поинтами.
3. Эскроу оказался пуст при попытке auto-approve → создатель получает предупреждение с названием кампании.

Creator's Telegram chatId берётся из их webchat session `userId` (числовой Telegram chat id, если создатель авторизовался через Telegram-бота). Если создатель зарегистрировался только через webchat без Telegram-бота — `chatId` отсутствует и уведомление просто пропускается (без ошибки).

Функции `notifyNewSubmission`, `notifyAutoApprove`, `notifyEmptyEscrow` экспортируются из нового файла `botplatform/src/bounty-notifications.ts`. Они вызываются из `bounty-api.ts` после ключевых операций.

## What to do

1. Создать файл `botplatform/src/bounty-notifications.ts` с тремя экспортируемыми функциями:
   - `notifyNewSubmission(creatorChatId: number, taskTitle: string, proofSummary: string): Promise<void>`
   - `notifyAutoApprove(creatorChatId: number, taskTitle: string, participantName: string, points: number): Promise<void>`
   - `notifyEmptyEscrow(creatorChatId: number, campaignTitle: string): Promise<void>`

2. В файле инициализировать Telegraf bot instance через `SIMPLEBOUNTY_BOT_TOKEN`. Если токен не задан — функции возвращают без действий (ранний return). Инстанс создаётся один раз при загрузке модуля (module-level singleton).

3. Каждая функция оборачивается в `try/catch`: при ошибке — `console.warn` с деталями, throw не прокидывается. Это гарантирует best-effort поведение.

4. Добавить вызовы уведомлений в `botplatform/src/bounty-api.ts`:
   - После успешного POST submission → `notifyNewSubmission(creatorChatId, task.title, proofSummary)`
   - После auto-approve в GET my-status → `notifyAutoApprove(creatorChatId, task.title, submission.participantName, task.reward)`
   - После auto-approve при balance=0 → `notifyEmptyEscrow(creatorChatId, campaign.title)`
   - Каждый вызов — `void` (не ждём, не пробрасываем ошибки)

5. Написать тест `botplatform/tests/test_bounty_notifications.js`, который запускает функции с `SIMPLEBOUNTY_BOT_TOKEN=test` через мок Telegraf и проверяет, что попытка отправки залогирована без краша.

## Details

**Files:**

- `botplatform/src/bounty-notifications.ts` — новый файл. Текущее состояние: не существует. Нужно создать с нуля.
  - Импортировать `Telegraf` из `'telegraf'` (пакет уже установлен в botplatform).
  - Module-level: читаем `process.env.SIMPLEBOUNTY_BOT_TOKEN`. Если задан — создаём `new Telegraf(token)` и сохраняем в константу `notifyBot`. Если не задан — `notifyBot = null`.
  - Три экспортируемые async функции. Каждая: если `notifyBot === null` → `return`. Иначе: `await notifyBot.telegram.sendMessage(creatorChatId, message)` в блоке `try/catch`. В `catch` — `console.warn('[bounty-notifications]', err)`.
  - Текст сообщений на русском языке, понятный для создателя кампании.

- `botplatform/src/bounty-api.ts` — существующий файл (создан в Task 5, содержит leaderboard endpoint + все предыдущие). Нужно добавить:
  - `import { notifyNewSubmission, notifyAutoApprove, notifyEmptyEscrow } from './bounty-notifications';`
  - В POST submission handler (после успешного сохранения заявки): `void notifyNewSubmission(creatorChatId, task.title, proofSummary)` — не await, не try/catch вокруг (функция сама обрабатывает ошибки).
  - В GET my-status handler (после auto-approve): `void notifyAutoApprove(creatorChatId, task.title, submission.participantName, task.reward)`.
  - В GET my-status handler (после auto-approve при escrow.balance === 0): `void notifyEmptyEscrow(creatorChatId, campaign.title)`.
  - `creatorChatId` берётся из campaign.creatorId (это уже числовой webchat userId = Telegram chatId для пользователей, авторизованных через бота).

**Dependencies:**
- Зависит от Task 5 (bounty-api.ts с leaderboard и submissions) — файл должен существовать.
- `telegraf` — уже в `package.json` botplatform (используется в `bot.ts`).

**Edge cases:**
- `SIMPLEBOUNTY_BOT_TOKEN` задан, но токен невалидный → Telegram вернёт 401. Ловим в catch, логируем, не бросаем.
- `creatorChatId` — участник создал кампанию через webchat без Telegram-бота. `creatorId` числовой, но Telegram-бот с ним не общался → `sendMessage` вернёт ошибку «chat not found». Ловим в catch.
- Параллельные auto-approve нескольких сабмитов → несколько `void notify...` запускаются одновременно. Это нормально: каждый независим.
- Модуль загружается даже если `SIMPLEBOUNTY_BOT_TOKEN` не задан (никаких throw при инициализации).

**Implementation hints:**
- `notifyBot.telegram.sendMessage` — это метод Telegraf Telegram instance, аналогично паттерну в `bot.ts` (там `this.bot` типа `Telegraf`).
- `proofSummary` для `notifyNewSubmission` — первые 100 символов `submission.proof` (уже HTML-entity encoded на этом этапе, можно обрезать).
- Для тестов: мокать Telegraf не обязательно — достаточно проверить что функция не бросает при невалидном токене (network error поймается в catch).
