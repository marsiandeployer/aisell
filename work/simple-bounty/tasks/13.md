## Description

Финальная приёмочная проверка фичи simple-bounty перед деплоем на прод. Задача охватывает три направления: полный тест-сьют (unit + integration + E2E), проверку всех acceptance criteria из user-spec (AC-1 — AC-14) и tech-spec (TC-1 — TC-15), а также coverage-верификацию — убедиться, что тесты действительно покрывают написанный код.

Задача не пишет новый код. Она верифицирует результаты задач 1–12 в совокупности и фиксирует итоговый статус в `logs/working/qa-report.json` и `decisions.md`. Если найдены criticals — деплой не должен происходить до исправлений.

## What to do

1. Запустить unit-тесты: `node tests/test_bounty_unit.js` из `/root/aisell/botplatform/` — ожидаемо 0 failures.
2. Запустить integration-тесты: `node tests/test_bounty_integration.js` — ожидаемо 0 failures.
3. Запустить E2E-тесты: `node tests/test_bounty_e2e.js` — ожидаемо 0 failures.
4. Запустить regression-тест SimpleDashboard: `node tests/test_sdk_methods.js` — убедиться, что нет регрессий.
5. Проверить каждый AC из user-spec (AC-1 — AC-14): passed / failed / not_verifiable с evidence.
6. Проверить каждый TC из tech-spec (TC-1 — TC-15): passed / failed / not_verifiable с evidence.
7. Проверить coverage: каждый файл из scope фичи (`bounty-api.ts`, `bounty-notifications.ts`, `index.html.template`, `start-webchat-simplebounty.sh`, `ecosystem.config.js`) имеет соответствующие тесты.
8. Сохранить полный JSON-отчёт в `logs/working/qa-report.json`.
9. Записать краткий итог в `decisions.md`.

## Details

**Где запускать тесты:** `/root/aisell/botplatform/` — все `node tests/...` команды отсюда.

**Перед запуском integration и E2E тестов:** убедиться, что `simplebounty-web` запущен (`pm2 status simplebounty-web` → online, порт 8097). E2E тесты используют Chrome CDP на `localhost:9222` — демон `chrome-cdp` должен быть запущен.

**Acceptance Criteria для проверки:**

Из user-spec:
- AC-1: Создатель создаёт кампанию через webchat AI-диалог → доступна на `d{userid}.wpmix.net`
- AC-2: Участник авторизуется через Google, видит список заданий
- AC-3: Участник загружает доказательство (текст или URL) → статус "pending"
- AC-4: Создатель нажимает «Одобрить» → поинты из эскроу начислены участнику
- AC-5: После 48ч при эскроу > 0 и без ответа создателя → auto-approve
- AC-6: После 48ч при эскроу = 0 → статус остаётся pending, уведомление создателю
- AC-9: Повторная заявка на то же задание → 409 «Вы уже отправили заявку на это задание»
- AC-10: Пустой эскроу → предупреждение в панели создателя
- AC-11: Лидерборд показывает топ-10 участников по сумме начисленных поинтов
- AC-13: Несколько кампаний у одного создателя с независимыми эскроу
- AC-14: Удаление задания с pending заявками → все pending автоматически rejected

Из tech-spec:
- TC-1: `pm2 status simplebounty-web` → online, порт 8097
- TC-2: `curl http://localhost:8097` → 200
- TC-3: POST `/api/bounty/campaigns` с валидной сессией → 201, `{id, ...}`
- TC-4: POST escrow deposit `{amount: -1}` → 400
- TC-5: POST escrow deposit `{amount: 100}` → balance=100 в escrow файле
- TC-6: POST submission с `proof: "<script>alert(1)</script>"` → сохраняется как `&lt;script&gt;alert(1)&lt;/script&gt;`
- TC-7: Дубликат submission (creatorId+taskId+participantId) → 409
- TC-8: Approve при balance=0 → 402
- TC-9: Approve чужой кампании → 403
- TC-10: GET my-status с `submittedAt = now - 49h`, escrow > 0 → статус `approved`
- TC-11: GET leaderboard → ответ не содержит поля `email`
- TC-12: `node tests/test_bounty_unit.js` → 0 failures
- TC-13: `node tests/test_bounty_integration.js` → 0 failures
- TC-14: `node tests/test_sdk_methods.js` → 0 failures (SimpleDashboard regression)
- TC-15: POST publish кампании → draft→published; повторный → 200 (idempotent); без заданий → 400

**Coverage-направление:** проверить, что для каждого из scope-файлов (`bounty-api.ts`, `bounty-notifications.ts`, `webchat.ts` в части bounty-mount, `index.html.template`) есть соответствующие тесты в одном из трёх файлов тест-сьюта. Файл без покрытия — severity `critical`.

**Критерии, вероятно не_верифицируемые до деплоя (not_verifiable):**
- AC-1 (webchat AI-диалог + реальный `d{userid}.wpmix.net`): требует live DNS и Google OAuth
- AC-2 (Google auth участника на `d{userid}.wpmix.net`): требует live OAuth redirect
- AC-6 (Telegram уведомление создателю): требует `SIMPLEBOUNTY_BOT_TOKEN` в prod
- Части AC-11 (UI лидерборд в браузере) — можно проверить через CDP E2E тест

- Запись в `decisions.md` — краткий итог
