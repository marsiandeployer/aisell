## Description

Задача добавляет escrow-логику в `botplatform/src/bounty-api.ts` — файл, созданный в Task 2. Эскроу — это зарезервированные поинты создателя кампании, из которых списываются награды при одобрении заявок участников.

Каждая кампания имеет свой независимый файл `escrow_{campaignId}.json` в `group_data/user_{creatorId}/data/`. Файл содержит текущий баланс и историю транзакций (`EscrowTx[]`).

Два публичных эндпоинта:
- `POST /api/bounty/campaigns/:id/escrow/deposit` — создатель пополняет эскроу (amount > 0)
- `GET /api/bounty/campaigns/:id/escrow/balance` — создатель запрашивает текущий баланс

Внутренняя функция `debit(campaignId, amount, ref, initiatedBy)` — используется из Task 4 (approve flow). Работает по схеме atomic read-modify-write с post-write rollback: читает файл, уменьшает balance, записывает, затем читает снова — если balance < 0, записывает старое значение обратно и возвращает 409 Conflict. Это обработка race condition при одновременных approve (Decision 7).

`EscrowTx.initiatedBy: 'creator' | 'auto-approve'` — поле аудит-трейла, разделяющее ручные deposit (creator) от автоматических debit при auto-approve.

Эндпоинт approve (из Task 2 или Task 4) при вызове debit обязан проверить: если balance < task.reward — вернуть 402 Payment Required до записи (pre-check). Rollback — дополнительная страховка от race condition.

GET balance возвращает `{ balance, warning: true }` если `balance === 0` — чтобы создатель видел предупреждение в панели.

## What to do

1. В `botplatform/src/bounty-api.ts` добавить TypeScript-интерфейсы `Escrow` и `EscrowTx` согласно Data Models в tech-spec.

2. Добавить вспомогательные функции для работы с escrow-файлом:
   - `getEscrowPath(creatorId, campaignId)` — путь к `escrow_{campaignId}.json`
   - `readEscrow(creatorId, campaignId)` — читает файл через `readJsonFile` с fallback `{ campaignId, balance: 0, transactions: [] }` (cold-start при отсутствующем файле — ENOENT не бросает ошибку)
   - `writeEscrow(creatorId, campaignId, escrow)` — сохраняет через `writeJsonAtomic`

3. Реализовать `POST /api/bounty/campaigns/:id/escrow/deposit`:
   - Требует webchat сессию (`requireSessionApi`) + проверка `campaign.creatorId === req.webUser.userId` (403 если не совпадает)
   - Валидация: `amount` должен быть числом > 0, иначе 400
   - Читает текущий escrow, увеличивает balance, добавляет `EscrowTx { type: 'deposit', amount, ref: 'manual', initiatedBy: 'creator', createdAt }`
   - Сохраняет через `writeJsonAtomic`
   - Возвращает `{ balance }` — 200

4. Реализовать `GET /api/bounty/campaigns/:id/escrow/balance`:
   - Требует webchat сессию + ownership check (403)
   - Читает escrow, возвращает `{ balance, warning: balance === 0 }`

5. Реализовать экспортируемую функцию `debit(creatorId, campaignId, amount, ref, initiatedBy)`:
   - Читает escrow (snapshot для rollback)
   - Pre-check: если `snapshot.balance < amount` → бросает ошибку с кодом 402 (до записи)
   - Уменьшает balance, добавляет `EscrowTx { type: 'debit', amount, ref, initiatedBy, createdAt }`
   - Записывает через `writeJsonAtomic`
   - Post-check: читает файл снова; если `balance < 0` → записывает snapshot обратно, бросает ошибку с кодом 409 (Decision 7 rollback)
   - При успехе возвращает новый balance

6. В обработчике approve submission (добавленном в Task 2 или добавляемом здесь): вызывать `debit` с `initiatedBy: 'creator'`; при ошибке 402 → вернуть 402 клиенту; при 409 → вернуть 409.

7. Написать unit-тесты в `botplatform/tests/test_bounty_unit.js` (TDD — до реализации) для escrow-логики согласно TDD Anchor ниже.

- Никаких новых npm-пакетов не требуется: `crypto` (built-in Node.js), `fs` (built-in), `path` (built-in).

**Edge cases:**

- Cold-start: `escrow_{campaignId}.json` не существует при первом обращении. `readJsonFile` с fallback `{ campaignId, balance: 0, transactions: [] }` обрабатывает это автоматически.
- Concurrent approvals (race condition): два approve одновременно при balance=50, reward=50 каждый. Post-write rollback ловит второй — один из approve вернёт 409. Это accepted limitation (Decision 7).
- Deposit при `amount = 0` — должен быть отклонён (amount должен быть строго > 0).
- Deposit с нечисловым amount (`"abc"`) → 400.
- Campaign не найдена → 404 до ownership check.
- Ownership check должен быть до любой модификации файлов.

**Implementation hints:**

- `writeJsonAtomic` пишет атомарно через tmp-файл и `fs.renameSync` — уже реализована в `webchat.ts` (line 100-105). Импортировать или скопировать в `bounty-api.ts`.
- `readJsonFile<T>(path, fallback)` возвращает fallback при ENOENT и при parse error — не бросает исключений (line 107-118 webchat.ts).
- Путь к escrow-файлу: `group_data/user_{creatorId}/data/escrow_{campaignId}.json` — следует той же структуре что и `campaigns.json`, `tasks.json`.
- `ensureDir` нужен перед `writeJsonAtomic` если директория может не существовать — уже вызывается внутри `writeJsonAtomic` (line 101: `ensureDir(path.dirname(filePath))`).
- Для rollback: сохранить snapshot ПЕРЕД записью (`const snapshot = readEscrow(...)`), после записи прочитать снова (`const check = readEscrow(...)`), если `check.balance < 0` — `writeEscrow(..., snapshot)` + throw 409.
- Pre-check перед debit (`if snapshot.balance < amount → throw 402`) не заменяет rollback — он лишь избегает заведомо бесполезной записи. Rollback нужен для гонки.
- `EscrowTx.createdAt` — ISO 8601, `new Date().toISOString()`.
- `initiatedBy: 'auto-approve'` используется в Task 4 когда auto-approve вызывает debit из GET .../submissions/my. Функция `debit` принимает `initiatedBy` как параметр — Task 4 передаст 'auto-approve'.
