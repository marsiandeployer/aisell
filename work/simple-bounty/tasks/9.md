# Task 9: Creator management panel

## Description

Добавляем creator-панель управления через новый endpoint `GET /api/bounty/panel` в `botplatform/src/webchat.ts`. Панель отображает HTML-страницу с полной информацией по кампаниям создателя: список кампаний с балансом эскроу, submissions с доказательствами и статусами, кнопки одобрения/отклонения заявок.

Endpoint реализуется именно в `webchat.ts` (НЕ в `bounty-api.ts`) — это намеренное архитектурное решение для избежания конфликтов при параллельной разработке с Task 8 в той же волне. Task 8 модифицирует `bounty-api.ts`, поэтому Task 9 вносит изменения только в `webchat.ts`.

Панель читает данные из JSON-файлов пространства создателя: `campaigns.json`, `tasks.json`, `submissions.json`, `escrow_{campaignId}.json`. Все значения proof отображаются HTML-escaped (данные уже хранятся в HTML-encoded виде согласно Task 4, поэтому render as-is безопасен). Кнопки «Одобрить» / «Отклонить» вызывают `POST /api/bounty/submissions/:id/approve|reject` через fetch с AJAX, без перезагрузки страницы. При balance=0 эскроу отображается предупреждающий badge. Счётчик pending submissions показывается на уровне кампании.

Панель является частью AC-10 (предупреждение при пустом эскроу) и пользовательского требования видеть все заявки с доказательствами в одном месте.

## What to do

1. Добавить `GET /api/bounty/panel` endpoint в `webchat.ts` с `requireSessionApi` middleware — endpoint защищён webchat-сессией создателя.

2. Endpoint читает данные из пространства текущего пользователя (`req.webUser.userId`): `campaigns.json`, `tasks.json`, `submissions.json`, и `escrow_{campaignId}.json` для каждой кампании. Путь к данным: `WORKSPACES_ROOT/user_{userId}/data/`.

3. Endpoint возвращает HTML-страницу (Content-Type: text/html) с:
   - Списком кампаний создателя (title, status, id)
   - Для каждой кампании: balance эскроу с badge-предупреждением если balance === 0, счётчик pending submissions
   - Для каждой кампании: submissions-таблица с колонками: participantName, taskTitle, proof (as-is, уже HTML-encoded), status, submittedAt, кнопки «Одобрить» / «Отклонить» для pending submissions

4. Кнопки «Одобрить» / «Отклонить» вызывают `POST /api/bounty/submissions/:id/approve` или `/reject` через JavaScript fetch. После успешного ответа — обновить строку в таблице без перезагрузки страницы (изменить status label, скрыть кнопки).

5. Написать тест `botplatform/tests/test_bounty_panel.js` (или добавить в `test_bounty_integration.js`): GET `/api/bounty/panel` без сессии → 401; с валидной сессией создателя → 200, response содержит HTML с кампаниями; проверить наличие pending submissions в HTML; проверить badge при balance=0.

6. Убедиться, что нет регрессий в существующих тестах SimpleDashboard (`test_sdk_methods.js`).

## Details

**Files:**

`botplatform/src/webchat.ts` — добавить один новый GET-route `/api/bounty/panel` внутри функции `startWebchat()`. Текущее состояние файла: 7012 строк, все bounty-эндпоинты находятся в `bounty-api.ts` и монтируются в webchat.ts. В webchat.ts уже есть паттерн HTML-рендеринга (например `/crawl-tests` endpoint на line 5183), который можно использовать как образец структуры (req → read data → res.end(html)). `requireSessionApi` (line 4799) — middleware, аттачит `req.webUser` с `userId`. `WORKSPACES_ROOT` (line 82) = `/root/aisell/botplatform/group_data`.

Путь к данным пользователя: `path.join(WORKSPACES_ROOT, 'user_' + userId, 'data', filename)`. Файлы: `campaigns.json` (массив Campaign), `tasks.json` (массив Task с campaignId), `submissions.json` (массив Submission с campaignId, taskId), `escrow_{campaignId}.json` (объект Escrow с balance).

`botplatform/tests/test_bounty_panel.js` — новый тест-файл (или секция в `test_bounty_integration.js`). Использует http-запросы к `http://localhost:8097`. Для получения сессионного cookie нужно предварительно создать тестового пользователя через существующий auth flow (см. паттерн в `test_bounty_integration.js`).

**Dependencies:**
- Task 5 (Leaderboard API) должен быть выполнен — это зависимость, после которой разблокируется Task 9. Task 8 (Telegram notifications) выполняется параллельно в той же волне 6 и не является зависимостью для Task 9.
- Данные: `campaigns.json`, `tasks.json`, `submissions.json`, `escrow_{campaignId}.json` — читаются через `readJsonFile` с пустым массивом/объектом как fallback (ENOENT safe).

**Edge cases:**
- Создатель без кампаний — показать пустой список (не падать)
- Кампания без submissions — отображать пустую таблицу
- Файл `escrow_{campaignId}.json` отсутствует — баланс = 0, показать warning badge
- submissions.json отсутствует — отображать пустую таблицу submissions
- proof в submissions.json хранится HTML-encoded (e.g. `&lt;script&gt;`) — вставлять в HTML as-is без дополнительного escape (иначе двойной encoding), браузер покажет как текст `<script>` (не исполнит)
- Создатель имеет несколько кампаний — отобразить все, каждую в отдельном разделе с собственным эскроу и submissions

**Implementation hints:**
- Endpoint регистрируется через `app.get('/api/bounty/panel', requireSessionApi, (req, res) => { ... })` — аналогично другим protected API-routes в webchat.ts
- HTML для панели строится как template string с loop по campaigns; каждая кампания — секция с submissions-таблицей; fetch-вызовы для approve/reject — встроенный JS в тот же HTML
- Для кнопок approve/reject: атрибут `data-id` на кнопке содержит submissionId; JS слушает click, делает fetch POST, при 200 обновляет DOM строки (меняет текст статуса, скрывает кнопки)
- Fetch в JS панели использует relative URL `/api/bounty/submissions/${id}/approve` — работает т.к. панель подаётся с того же origin
- Не добавлять rate-limiter на этот endpoint — создатель вызывает редко, существующих ограничений на webchat достаточно
- Типы Campaign, Task, Submission, Escrow можно импортировать из `bounty-api.ts` если они экспортированы, или объявить inline-интерфейсы в функции для простоты
