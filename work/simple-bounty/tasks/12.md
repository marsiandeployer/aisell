## Description

Финальный тестовый слой simple-bounty — E2E тесты через Chrome CDP. Задача стоит в Wave 8 и запускается после того, как все integration-тесты (Task 11) и unit-тесты (Task 10) уже зелёные.

E2E покрывает сквозные сценарии через реальный HTTP к `http://localhost:8097` и через Chrome CDP (localhost:9222) для проверки HTML-рендеринга страницы кампании `d{userId}.wpmix.net`. В отличие от integration-тестов (Task 11), E2E включает:
- Открытие страницы кампании в браузере и проверку что задания видны в реальном DOM
- Проверку безопасного рендеринга XSS-пробы (`<script>` → `&lt;script&gt;` на странице участника)

Инструментарий Chrome CDP уже запущен в памяти (`pm2 start chrome-cdp`) на порту 9222. Новая вкладка открывается через `PUT /json/new?{url}`, консоль читается через WebSocket CDP, вкладка закрывается через GET `/json/close/{id}`. Готовый helper — `/root/tools/chrome-daemon/cspy.js`.

Все HTTP вызовы к API — напрямую через `http.request` к `localhost:8097` (как в test_sdk_methods.js), с заголовком `Host: d{userId}.wpmix.net` для эндпоинтов участника. Никаких внешних npm-пакетов не нужно — только встроенный `http`, `crypto`, `fs`, `path`.

## What to do

1. Создать файл `botplatform/tests/test_bounty_e2e.js` (новый) с тест-харнессом аналогичным `test_sdk_methods.js` — цветной вывод `[PASS]`/`[FAIL]`, счётчик `passed`/`failed`, exit code 1 при любом провале.

2. Реализовать вспомогательные функции:
   - `httpRequest(url, options)` — обёртка над `http.request`, возвращает `{status, body}`. Поддерживает заголовок `Host` для участниковых эндпоинтов.
   - `cdpPut(path)` — PUT к Chrome CDP `localhost:9222` (открытие вкладки)
   - `cdpGetRaw(path)` — GET к Chrome CDP (закрытие вкладки, список)
   - `wsConnect(wsUrl)` — минимальный WebSocket клиент без зависимостей (можно переиспользовать паттерн из `cspy.js`)
   - `openTabGetHtml(url, timeoutMs)` — открыть вкладку CDP, дождаться загрузки через Runtime.evaluate `document.documentElement.outerHTML`, закрыть вкладку, вернуть HTML
   - `getBalancesJson(creatorId)` — читать `group_data/user_{creatorId}/data/balances.json` напрямую с диска через `fs.readFileSync`
   - `getSubmissionsJson(creatorId)` — читать `group_data/user_{creatorId}/data/submissions.json` напрямую с диска

3. Написать следующие сценарии (каждый независим, имеет cleanup):

   **Scenario 1: Creator creates campaign → page shows tasks in HTML**
   - POST `/api/bounty/campaigns` с webchat сессией создателя → получить campaignId
   - POST `/api/bounty/campaigns/:id/tasks` → добавить задание (reward: 50)
   - POST `/api/bounty/campaigns/:id/escrow/deposit` {amount: 100}
   - POST `/api/bounty/campaigns/:id/publish`
   - Открыть `http://localhost:8097` с заголовком `Host: d{creatorId}.wpmix.net` через Chrome CDP (`openTabGetHtml`)
   - Assert: HTML содержит title задания

   **Scenario 2: Participant submits → creator approves → balance updated**
   - Создать кампанию + задание + deposit (аналогично Scenario 1)
   - POST submission от participantId `test-participant-e2e-1`
   - POST `/api/bounty/submissions/:id/approve` от создателя
   - Прочитать `balances.json` с диска → assert: `balances["test-participant-e2e-1"] === 50`

   **Scenario 3: Auto-approve via direct file write → GET my-status → approved**
   - Создать кампанию + задание + deposit
   - POST submission → получить submissionId
   - Прочитать `submissions.json`, найти запись, установить `submittedAt = new Date(Date.now() - 49 * 3600 * 1000).toISOString()`, записать файл обратно
   - GET `/api/bounty/campaigns/:id/tasks/:taskId/submissions/my?participantId=...`
   - Assert: status === 'approved'

   **Scenario 4: Duplicate submit → 409**
   - Создать кампанию + задание
   - POST submission (первый раз) → 201
   - POST submission (тот же participantId + taskId) → assert status 409

   **Scenario 5: Leaderboard updates after approve**
   - Создать кампанию + задание + deposit
   - POST submission → approve
   - GET `/api/bounty/campaigns/:id/leaderboard`
   - Assert: список содержит запись с participantId и totalPoints === reward
   - Assert: ответ не содержит поле `email`

   **Scenario 6: XSS probe in proof → rendered safe in page HTML**
   - Создать кампанию + задание + deposit + publish
   - POST submission с `proof: "<script>alert('xss')</script>"`
   - POST approve
   - Открыть страницу `d{creatorId}.wpmix.net` через CDP
   - Assert: HTML содержит `&lt;script&gt;` и НЕ содержит сырой `<script>alert`

4. В `main()` запустить все сценарии последовательно, собрать итог, вывести summary и выйти с кодом 1 если есть провалы.

## Details

**Files:**

- `botplatform/tests/test_bounty_e2e.js` (новый файл) — все E2E тесты. Запускается из директории `/root/aisell/botplatform/` командой `node tests/test_bounty_e2e.js`.

**Dependencies:**

- Task 10 (unit tests) и Task 11 (integration tests) должны быть выполнены и зелёные — E2E строится поверх рабочего API
- `pm2 start chrome-cdp` должен быть запущен до начала тестов (Chrome CDP на :9222)
- `simplebounty-web` процесс на порту 8097
- Только встроенные Node.js модули: `http`, `fs`, `path`, `crypto`, `net` — внешних пакетов нет

**Chrome CDP — ключевые детали (из cspy.js):**

- Новая вкладка: `PUT http://localhost:9222/json/new?{encodeURIComponent(url)}` — возвращает JSON с `{id, webSocketDebuggerUrl}`
- Закрыть вкладку: `GET http://localhost:9222/json/close/{tab.id}` — не DELETE
- WebSocket handshake: upgrade через `net.connect`, ключ `Sec-WebSocket-Key`, версия 13
- Получить HTML страницы: отправить через WS `{id:1, method:'Runtime.enable'}` и `{id:2, method:'Runtime.evaluate', params:{expression:'document.documentElement.outerHTML', returnByValue:true}}`, дождаться `result.id === 2` в ответе — значение в `result.result.value`
- После получения HTML — закрыть WS (`sock.destroy()`) и закрыть вкладку

**Webchat сессия для создателя:**

- Эндпоинт `/api/auth/claim` (POST `{name, email}`) создаёт сессию и возвращает `Set-Cookie`. Использовать уникальный email для каждого прогона тестов (включить timestamp).
- Все creator API вызовы идут с `Cookie: <session>` заголовком к `http://localhost:8097`
- `req.webUser.userId` в API = числовой userId из сессии — это и есть creatorId для `d{creatorId}.wpmix.net` и пути `group_data/user_{creatorId}/`

**Participant API вызовы:**

- Участниковые эндпоинты в bounty-api.ts проверяют Google OAuth сессию. Для E2E тестов передавать `participantId` и `participantName` через специальный тестовый заголовок или напрямую в body — проверить как это реализовано в Task 4. Если API требует реального Google токена, использовать `GOOGLE_AUTH_TEST_SECRET` механизм (аналогично test_auth_api.js).
- Альтернатива: для сценариев где нужен участник, использовать прямой HTTP вызов с `X-Test-ParticipantId` заголовком если такой механизм реализован в Task 4; иначе использовать реальную webchat сессию участника через `/api/auth/claim`.

**Путь к данным на диске:**

- `group_data/user_{creatorId}/data/balances.json`
- `group_data/user_{creatorId}/data/submissions.json`
- `group_data/user_{creatorId}/data/campaigns.json`
- Base path: `/root/aisell/botplatform/group_data/`

**Cleanup:**

- После каждого сценария удалять JSON-файлы созданного creatorId: `campaigns.json`, `tasks.json`, `submissions.json`, `escrow_*.json`, `balances.json`
- Использовать уникальный email для каждого сценария — изолирует данные между запусками

**Edge cases:**

- CDP timeout: если страница медленно загружается, ждать через `Runtime.evaluate` в цикле с таймаутом 10-15 секунд
- Если Chrome-вкладка открылась но страница вернула ошибку — HTML будет содержать текст ошибки; проверять что HTML содержит `taskTitle`, а не просто что он не пустой
- Тест должен работать и при повторном запуске (идемпотентность через уникальные email + cleanup)
- Сценарий 3 (auto-approve через прямую запись файла): после записи `submittedAt` нужно подождать что сервер не кэширует submissions в памяти — если в bounty-api.ts используется `readJsonFile` (без кэша) то всё ок; если кэш есть — нужно рестартовать процесс или использовать другой подход

**Implementation hints:**

- Паттерн `httpRequest` из `test_sdk_methods.js` (строки 70-120) — хорошая база для Node.js http без лишних зависимостей
- Паттерн `wsConnect` из `cspy.js` (строки 56-134) — минимальный WebSocket клиент без npm-пакетов
- Сценарии запускать последовательно через `await scenario1()` и т.д. — не параллельно, чтобы не было race condition на файлах
- `section('Scenario N: ...')` для группировки вывода аналогично test_sdk_methods.js
