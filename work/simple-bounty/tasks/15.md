## Description

Live-верификация фичи simple-bounty на production-окружении. После деплоя (Task 14) необходимо пройти полный список шагов из user-spec "Как проверить" через curl и прямые API-вызовы к `https://simplebounty.wpmix.net` и `https://d{userid}.wpmix.net`.

Задача не пишет код — она запускает агента-верификатора, который последовательно проходит весь Agent Verification Plan (AVP) из tech-spec и все Acceptance Criteria из user-spec на живом окружении. Результат — JSON-отчёт по формату post-deploy-qa SKILL.md.

Если шаг нельзя проверить автоматически (например, Google OAuth redirect требует браузера, или Telegram уведомление требует реального бот-токена) — шаг помечается `blocked` с инструкцией для ручной проверки.

## What to do

1. Прочитать pre-deploy QA отчёт (`logs/working/qa-report.json`) — взять `deferredToPostDeploy` критерии, если они есть.
2. Выполнить все 8 шагов из таблицы user-spec "Как проверить агент" через curl к prod-хосту `https://simplebounty.wpmix.net`:
   - Создать кампанию (POST /api/bounty/campaigns) → убедиться в 200 OK и наличии campaign_id в ответе
   - Проверить доступность страницы кампании (GET d{userid}.wpmix.net) → 200 OK, задания в HTML
   - Подать заявку (POST /api/bounty/submissions) → статус "pending"
   - Повторная заявка (тот же task+user) → 409 Conflict
   - Одобрить заявку (POST /api/bounty/submissions/:id/approve) → эскроу уменьшился, поинты участника увеличились
   - Auto-approve: установить submittedAt = now−49h, GET /api/bounty/submissions/:id → статус "approved"
   - Approve при balance=0 → 402 с текстом ошибки
   - Лидерборд (GET /api/bounty/campaigns/:id/leaderboard) → список участников с поинтами
3. Проверить HTTPS: `curl -I https://simplebounty.wpmix.net` → 200, заголовок `strict-transport-security` присутствует.
4. Проверить Google OAuth redirect: убедиться что redirect_uri ведёт на `https://simplebounty.wpmix.net/api/auth/google-dashboard-callback` (не на simpledashboard.wpmix.net). Если нет браузера — пометить `blocked`, описать шаги для ручной проверки.
5. Проверить Telegram уведомление: если `SIMPLEBOUNTY_BOT_TOKEN` задан в env — выполнить шаг с новой заявкой и убедиться что бот прислал сообщение. Если токен не задан — пометить `blocked`.
6. Дополнительно: проверить что leaderboard response не содержит поля `email` (PII protection) и что XSS probe в proof (`<script>alert(1)</script>`) сохраняется как `&lt;script&gt;`.
7. Сверить все acceptance criteria из user-spec (AC-1 — AC-14) и tech-spec (TC-1 — TC-15) с live-поведением.
8. Сформировать итоговый JSON-отчёт по формату из post-deploy-qa SKILL.md: `status`, `summary`, `agentVerification`, `acceptanceCriteria`, `findings`.
9. Сохранить отчёт в `logs/working/task-15/post-deploy-qa-1.json`.

## Verification Steps

- `curl -I https://simplebounty.wpmix.net` → HTTP 200, заголовок `strict-transport-security` присутствует
- `curl -s -X POST https://simplebounty.wpmix.net/api/bounty/campaigns -H "Content-Type: application/json" -b "<session_cookie>" -d '{"title":"Verify Campaign","description":"Test"}' | jq .id` → непустой campaign_id
- `curl -s https://d{userid}.wpmix.net` → HTML-страница с заданиями
- `curl -s -X POST .../tasks/:taskId/submissions -d '{"proof":"test proof"}' -b "<participant_session>"` → `{"status":"pending"}`
- Повторный POST тех же данных → HTTP 409
- `curl -s -X POST .../submissions/:id/approve -b "<creator_session>"` → escrow balance уменьшился (проверить через GET balance)
- Установить submittedAt = now−49h в submissions.json → GET my-status → `{"status":"approved"}`
- POST approve при balance=0 → HTTP 402
- `curl -s .../leaderboard | jq 'map(keys)'` → массив keys без поля `email`
- `cat logs/working/task-15/post-deploy-qa-1.json | jq .status` → `"passed"`

## Details

**Prod-хост:** `https://simplebounty.wpmix.net` → reverse proxy 62.109.14.209 → VM104:8097. Базовый URL для всех curl-запросов.

**Аутентификация для curl:**
- Создатель: сессионная кука webchat (получить через логин в webchat или взять из браузера DevTools). Передавать как `-b "session=<value>"`.
- Participant endpoints (submissions, my-status): требуют Google OAuth участника. Если нет браузера — протестировать через Chrome CDP (`localhost:9222`) или отметить шаги `blocked` с manual plan.

**Порядок выполнения:**
1. Infrastructure check (HTTPS, 200 OK) — без auth
2. Creator flow: создать кампанию → добавить задание → пополнить эскроу → опубликовать
3. Participant flow: сабмит заявки → проверка статуса → auto-approve мок
4. Деструктивные проверки: approve при balance=0 (402), повторный сабмит (409)
5. PII check: лидерборд без `email`
6. XSS check: proof с `<script>` тегом

**Auto-approve мок (шаг 6):**
Прямая запись на сервере: в файле `group_data/user_{creatorId}/data/submissions.json` установить `submittedAt` = ISO 8601 строка (now − 49 часов). Затем GET `/api/bounty/campaigns/:id/tasks/:taskId/submissions/my` → должен вернуть `status: "approved"`. Эскроу должен уменьшиться, balances.json обновиться.

**Google OAuth проверка (может быть blocked):**
`redirect_uri` задаётся в `start-webchat-simplebounty.sh` переменной `GOOGLE_OAUTH_REDIRECT_URI`. Через curl невозможно проверить browser-based flow. Инструкция для `blocked`: открыть `https://simplebounty.wpmix.net` в браузере → Google Sign-In → в адресной строке OAuth экрана убедиться что `redirect_uri=https%3A%2F%2Fsimplebounty.wpmix.net%2Fapi%2Fauth%2Fgoogle-dashboard-callback`.

**Telegram уведомление (может быть blocked):**
Если `SIMPLEBOUNTY_BOT_TOKEN` не задан в `.env` процесса `simplebounty-web` — шаг `blocked`. Manual plan: задать токен, перезапустить `pm2 restart simplebounty-web`, создать новую заявку через API, проверить в Telegram боте входящее сообщение.

**Дополнительные edge-case проверки:**
- Approve чужой кампании → 403 (ownership check, TC-9)
- Deposit с `amount: -1` → 400 (TC-4)
- XSS probe: `proof: "<script>alert(1)</script>"` → сохраняется как `&lt;script&gt;alert(1)&lt;/script&gt;` (TC-6)

**Dependencies:**
- Task 14 (Deploy) должна быть complete: `https://simplebounty.wpmix.net` отвечает 200
- pre-deploy QA отчёт (`logs/working/qa-report.json`) — прочитать `deferredToPostDeploy` секцию перед запуском

**Отчёт:**
Сохранить в `logs/working/task-15/post-deploy-qa-1.json`. Формат точно по post-deploy-qa SKILL.md: `status`, `summary`, `agentVerification[]`, `acceptanceCriteria[]`, `findings[]`. Статус `passed` = нет critical находок.
