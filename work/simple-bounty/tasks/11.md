# Task 11: Integration tests

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Write HTTP integration tests for the running `simplebounty-web` server (port 8097). The tests make real HTTP requests to the actual server and verify the full bounty API contract end-to-end: campaign creation, task management, escrow deposit, submissions, approve/reject, auto-approve, leaderboard, ownership checks, duplicate detection, and concurrent approve race condition.

This task runs in Wave 7 alongside Task 10 (unit tests). Unlike unit tests (which mock the filesystem), integration tests rely on the actual running server and real JSON file storage under `group_data/`. The tests must set up their own test data (using a dedicated test creator userId) and clean up after themselves.

The test file follows the pattern established by `test_sdk_methods.js`: plain Node.js with no test framework, `http` module for requests, colored output, `assert()` / `section()` helpers, and `process.exit(failed > 0 ? 1 : 0)` at the end.

## What to do

1. Create `botplatform/tests/test_bounty_integration.js` as a new file using the same structure as `test_sdk_methods.js` (test harness, COLORS, assert/section helpers, httpRequest helper, main function).

2. Configure constants at the top: `BASE_URL = 'http://127.0.0.1:8097'`, a dedicated `TEST_CREATOR_ID` (a fixed fake userId such as `9000000099001`) used as webchat session identity, and a `TEST_PARTICIPANT_ID` (a fake Google sub such as `google_sub_test_001`).

3. Implement authentication helpers: the bounty API uses `requireSessionApi` (webchat session cookie). Read `botplatform/src/webchat.ts` and `botplatform/src/bounty-api.ts` (Task 2-9 output) to understand the actual auth mechanism and replicate it in tests. If there is a test-mode session header or cookie, use it. The test must not hardcode real user credentials.

4. Implement test groups (each group is a separate async function):

   **Group 1: Campaign lifecycle**
   - POST `/api/bounty/campaigns` with title+description → assert 201, response has `id`, `creatorId`, `status: 'draft'`
   - Verify campaign appears in `group_data/user_{TEST_CREATOR_ID}/data/campaigns.json` on disk
   - POST `/api/bounty/campaigns/:id/tasks` with title, description, reward=50 → 201, task has `id` and `campaignId`
   - POST `/api/bounty/campaigns/:id/publish` → 200, status becomes `published`
   - POST `/api/bounty/campaigns/:id/publish` again → 200 (idempotent)

   **Group 2: Escrow deposit**
   - POST `/api/bounty/campaigns/:id/escrow/deposit` `{amount: 100}` → 200, balance=100
   - Verify `escrow_{campaignId}.json` balance field equals 100 by reading the file directly
   - POST with `{amount: -1}` → 400

   **Group 3: Submissions — basic flow**
   - POST `…/tasks/:taskId/submissions` with participant session + proof text → 201, `{status: 'pending'}`
   - Verify entry in `submissions.json` with `status: 'pending'`
   - Duplicate POST (same participantId + taskId) → 409, response body contains "Вы уже отправили заявку"

   **Group 4: Auto-approve via direct submittedAt manipulation**
   - Create a new submission in pending state
   - Read `submissions.json` directly via `fs`, set `submittedAt = new Date(Date.now() - 49 * 60 * 60 * 1000).toISOString()` for the test submission, write back
   - Wait 50ms, then GET `…/tasks/:taskId/submissions/my` → assert `status: 'approved'`, `pointsAwarded` equals task reward
   - Verify `escrow_{campaignId}.json` balance decreased by reward
   - Verify `balances.json` entry for participant increased by reward

   **Group 5: Manual approve and reject**
   - Create a fresh campaign + task (reward=50) + deposit(50) + submission
   - POST `…/submissions/:submissionId/approve` as creator → 200, `status: 'approved'`
   - Verify escrow balance reduced to 0 by reading file
   - Verify `balances.json` entry for participant increased by 50
   - Attempt to approve a second submission against a campaign with escrow balance=0 → 402
   - Create another submission, POST `…/submissions/:submissionId/reject` → 200, `status: 'rejected'`
   - Verify escrow balance unchanged after reject by reading file

   **Group 6: Full escrow cycle to zero**
   - Create campaign, task (reward=50), deposit 50
   - Submit + approve → escrow balance = 0
   - Create second submission for a new task (reward=50)
   - POST approve for second submission → 402
   - Read `escrow_{campaignId}.json` directly and assert `balance === 0` (not negative)

   **Group 7: Ownership check (403)**
   - Create a second creator session using `TEST_CREATOR_ID_2 = '9000000099002'`
   - Create campaign + task + submission under creator 1
   - Attempt POST `…/submissions/:id/approve` as creator 2 → assert 403

   **Group 8: Leaderboard**
   - After several approvals (from groups above), GET `/api/bounty/campaigns/:id/leaderboard`
   - Assert response is an array
   - Assert each entry has `participantId`, `participantName`, `totalPoints` fields
   - Assert no entry has an `email` field
   - Assert entries are sorted descending by `totalPoints` (each element totalPoints >= next)
   - Assert array length <= 10

   **Group 9: Concurrent approve race condition**
   - Create campaign, task (reward=50), deposit 50
   - Create two submissions from two different participants
   - Send `Promise.all([approve(sub1), approve(sub2)])` simultaneously
   - Assert at least one succeeds (200) and at least one fails (402 or 409)
   - Read `escrow_{campaignId}.json` directly after both settle and assert `balance >= 0` (never negative)

5. In `main()`: verify server reachable at `http://127.0.0.1:8097`, run all groups sequentially with `delay(50)` between them, run cleanup, print summary, exit with code 0 if 0 failures or 1 if any failures.

6. Add cleanup in a `finally` block: remove `group_data/user_9000000099001/` and `group_data/user_9000000099002/` directories entirely using `fs.rmSync(path, {recursive: true, force: true})`.
