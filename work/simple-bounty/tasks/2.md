## Description

Создать новый файл `botplatform/src/bounty-api.ts` со всеми CRUD-эндпоинтами для кампаний и заданий. Этот файл станет основой bounty-продукта — все остальные tasks (Escrow, Submissions, Leaderboard) будут добавлять эндпоинты в него.

Задача закрывает первый слой бизнес-логики: создание, чтение, удаление кампаний и заданий. После этой задачи creator может создать кампанию, добавить к ней задания и опубликовать её. Участники могут читать задания публично (без auth).

Файл экспортирует Express Router, который монтируется в `webchat.ts` одной строкой. Роутер обращается к JSON-файлам в `group_data/user_{creatorId}/data/` через уже существующие в webchat.ts функции `writeJsonAtomic` и `readJsonFile` (они недоступны напрямую — нужно скопировать/переиспользовать паттерн или параметризовать через dependency injection при монтировании).

Ключевая особенность архитектуры: `bounty-api.ts` существует потому, что `webchat.ts` уже содержит 7000+ строк кода и дальнейшее расширение вызовет merge-конфликты при параллельной разработке. Вся bounty-логика должна оставаться в `bounty-api.ts`, webchat.ts только монтирует router.

## What to do

1. Создать `botplatform/src/bounty-api.ts` — новый файл с Express Router:
   - Объявить TypeScript-интерфейсы `Campaign`, `Task`, `Submission` (только те поля, что нужны для этой задачи — `Submission` нужен только для каскадного rejected при delete задания)
   - Реализовать хелперы для чтения/записи JSON (файлы кампаний и заданий в `group_data/user_{creatorId}/data/`)
   - `POST /api/bounty/campaigns` — создать кампанию (требует session auth, status='draft', генерирует uuid id)
   - `GET /api/bounty/campaigns` — список кампаний creator (требует session auth)
   - `GET /api/bounty/campaigns/:campaignId` — одна кампания (публичный)
   - `POST /api/bounty/campaigns/:campaignId/tasks` — создать задание (требует session auth + ownership, reward > 0)
   - `GET /api/bounty/campaigns/:campaignId/tasks` — список заданий (публичный, без auth)
   - `DELETE /api/bounty/campaigns/:campaignId/tasks/:taskId` — удалить задание (требует session auth + ownership; при наличии pending submissions — автоматически reject все pending заявки для этого taskId)
   - `POST /api/bounty/campaigns/:campaignId/publish` — перевести кампанию draft→published (требует session auth + ownership; требует хотя бы одно задание; idempotent: повторный вызов → 200)

2. Модифицировать `botplatform/src/webchat.ts` — смонтировать router:
   - Добавить import bounty router в начало файла
   - Добавить `app.use('/api/bounty', bountyRouter)` в блок регистрации роутов (после глобального rate limiter, до fallback-хендлера)

**Edge cases:**
- `campaignId` в params — проверять что кампания принадлежит req.webUser.userId (403 если нет)
- `reward` — проверять `typeof reward === 'number' && reward > 0` (не строка, не NaN, не Infinity)
- Каскадный reject при DELETE задания: читаем submissions.json, фильтруем по `taskId + status === 'pending'`, меняем на `rejected`, пишем обратно атомарно. Submissions.json может не существовать (ENOENT) — нормально, каскадить нечего
- Publish idempotency: если кампания уже `published` → вернуть 200 без ошибки, не менять данные
- UUID для id: использовать `crypto.randomUUID()` (Node.js 14.17+, доступен без пакетов)
- Path traversal: `campaignId` и `taskId` из params валидировать regex `/^[a-zA-Z0-9_-]{1,64}$/` перед использованием в file path

**Implementation hints:**
- `writeJsonAtomic` и `readJsonFile` нельзя импортировать из webchat.ts (не экспортируются). Либо продублировать эти ~20 строк в `bounty-api.ts`, либо вынести в `botplatform/src/utils.ts` и переиспользовать. Дублирование приемлемо для MVP — эти функции тривиальны
