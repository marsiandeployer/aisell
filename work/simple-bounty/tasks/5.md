# Task 5: Leaderboard API

## Required Skills

Before starting, load: `/skill:code-writing` ā€” [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Add a public `GET /api/bounty/campaigns/:id/leaderboard` endpoint to `botplatform/src/bounty-api.ts`.

The endpoint aggregates approved submissions for a given campaign: it sums `pointsAwarded` per `participantId`, sorts participants in descending order by total points, and returns the top 10. The response contains only `{participantId, participantName, totalPoints}` ā€” no email field is allowed under any circumstances (PII protection per tech-spec Decision 5 and AC-11 / TC-11).

This is a Wave 5 task that depends on Task 4 (Submissions API). By the time this task runs, `bounty-api.ts` already contains campaign CRUD (Task 2), escrow logic (Task 3), and submissions with approve/reject (Task 4). This task adds the final read-only aggregation endpoint to complete the public participant-facing API surface.

The leaderboard reads `submissions.json` directly on every request ā€” no caching, no intermediate store. This keeps the implementation simple and ensures the leaderboard always reflects the current state after any approval.

## What to do

1. Add a `LeaderboardEntry` TypeScript interface inside `bounty-api.ts`:
   `{ participantId: string, participantName: string, totalPoints: number }`
   This is the only shape that may appear in the leaderboard response body.

2. Add the route `GET /api/bounty/campaigns/:id/leaderboard` to the bounty Express router in `bounty-api.ts`. The route must be public (no `requireSessionApi` middleware, no auth header required).

3. Implement the aggregation logic inside the route handler:
   - Look up the campaign from `campaigns.json` using `campaignId` to get `campaign.creatorId` (needed to locate the data directory).
   - Read `submissions.json` for that creator using `readJsonFile` with a `[]` fallback (ENOENT-safe).
   - Filter to submissions where `campaignId === req.params.id` AND `status === 'approved'`.
   - Accumulate `pointsAwarded` per `participantId` into a Map. Track `participantName` as the last seen value for that participant.
   - Convert Map to array, sort by `totalPoints` descending, slice to top 10.
   - Respond with `200` and `{ leaderboard: LeaderboardEntry[] }`.

4. Handle the case where no approved submissions exist for the campaign: return `200` with `{ leaderboard: [] }` ā€” not a 404, since the endpoint is public and a campaign may simply have no approvals yet.

5. Apply rate limiting on the endpoint using the existing `enforceRateLimit` / `SlidingWindowRateLimiter` pattern already imported in `bounty-api.ts` from earlier tasks. Reuse one of the existing global rate limiters consistent with other public endpoints in the router.

6. Extract the aggregation into a named pure function `aggregateLeaderboard(submissions: Submission[], campaignId: string): LeaderboardEntry[]` so it can be unit-tested in isolation.

**Data model (from tech-spec):**

```
// submissions.json entry fields relevant to leaderboard:
// participantId: string  ā€” Google sub, never an email address
// participantName: string ā€” Google display name
// pointsAwarded?: number ā€” set on approve; undefined if not yet approved
// status: 'pending' | 'approved' | 'rejected'
// campaignId: string

// Response shape ā€” only these three fields, constructed explicitly:
// { participantId: string, participantName: string, totalPoints: number }
```

**Edge cases:**
- `pointsAwarded` may be `undefined` on some approved submissions if set before the field was introduced ā€” treat `undefined` as `0` when summing
- A participant may have multiple approved submissions across different tasks in the same campaign ā€” all must be summed
- Two participants with equal `totalPoints` ā€” stable sort order is acceptable, no tiebreaker required
- Campaign does not exist in `campaigns.json` ā€” cannot determine `creatorId`, return `200` with empty leaderboard (do not 404, endpoint is public)
- `submissions.json` does not exist yet ā€” `readJsonFile(..., [])` fallback handles ENOENT transparently

**Security note:**
- Never spread a `Submission` object into the response body ā€” always construct `LeaderboardEntry` explicitly with only three fields. A future addition of an `email` or `proof` field to `Submission` must not accidentally leak through
- Apply IP-based rate limiting to prevent scraping, consistent with other public bounty endpoints
