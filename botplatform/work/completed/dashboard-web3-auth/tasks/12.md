## Description

Live environment verification of the dashboard-web3-auth feature after successful deploy (task 11).

This task validates the entire feature end-to-end on the production infrastructure: Auth API health via nginx proxy, register/login flows with real signature verification, PG data persistence, key backup sync, and dashboard browser flows (blur without extension, full auth with extension, service-unavailable state). All checks run against live infrastructure — `d*.wpmix.net` dashboards, Auth API on port 8095 proxied via nginx, PostgreSQL on LXC 102 (10.10.10.2).

Verification is the review — no separate reviewers. The agent produces a JSON report according to the post-deploy-qa skill output format. Clean result (zero criticals, zero failures) means the feature is production-ready.

## What to do

1. Read pre-deploy QA report at `logs/working/qa-report.json` — collect all `deferredToPostDeploy` criteria.

2. Verify PG is accessible and schema is applied: connect to pg-db (10.10.10.2) and confirm tables `users`, `dashboards`, `dashboard_access` exist.

3. Verify Auth API health through nginx proxy: `GET https://d9000000000001.wpmix.net/api/auth/health` → 200 with `{status: "ok", pg: "connected"}`.

4. Register a test user: `POST /api/auth/register` with a fresh Ethereum keypair, test email, and a test dashboardId → expect 201.

5. Login with valid signature: generate a valid challenge (JSON with dashboardId, timestamp, nonce), sign it with the test keypair via ethers.js, `POST /api/auth/login` → expect 200 + JWT.

6. Login with invalid signature: send a login request signed by a different (random) keypair → expect 401.

7. Duplicate email: register again with the same email → expect 409 with "Напишите в support@onout.org" message.

8. Verify key backup: query `SELECT address, private_key FROM users WHERE email = '<test_email>'` → `private_key` field is not empty.

9. Sharing flow: register a second test user, call `POST /api/auth/share` to add that user's email to the test dashboard → login from the second user's address → expect 200 + JWT.

10. Nginx proxy restriction: confirm that `POST /api/auth/register` and `POST /api/auth/share` return 403 or are unreachable from outside localhost (test via reverse proxy — direct request from external should be blocked).

11. Dashboard blur without extension: use Puppeteer (headless, no extension) to open a protected test dashboard → verify title element is visible, data container has blur CSS applied, install-extension overlay is present in DOM.

12. Dashboard auth flow with extension: use Puppeteer with extension loaded, pre-seeded keypair matching the test dashboard's owner address → navigate to dashboard → verify blur is removed, data is visible.

13. Auth API down scenario: stop `dashboard-auth-api` PM2 process, open dashboard in Puppeteer → verify "service unavailable" message is shown → restart PM2 process.

14. Backup sync: verify backup LXC contains a recent dump of the `dashboard_auth` database (file exists and mtime is within expected window).

15. Cleanup: remove test data from PG — delete test users and test dashboard records created during verification. Restart `dashboard-auth-api` if it was stopped.

16. Write verification report to `logs/working/task-12/post-deploy-qa-report.json` in the format defined by the post-deploy-qa skill (AVP steps + acceptance criteria + findings).

## Verification Steps

All verification targets the live environment. Each step below corresponds to a "What to do" step above.

1. `psql -h 10.10.10.2 -U dashboard_auth -d dashboard_auth -c '\dt'` — output lists `users`, `dashboards`, `dashboard_access`
2. `curl https://d9000000000001.wpmix.net/api/auth/health` — HTTP 200, body `{"status":"ok","pg":"connected"}`
3. `curl -X POST https://d9000000000001.wpmix.net/api/auth/register -H "Authorization: Bearer <INTERNAL_API_KEY>" -H "Content-Type: application/json" -d '{...}'` — HTTP 201
4. `curl -X POST https://d9000000000001.wpmix.net/api/auth/login -H "Content-Type: application/json" -d '{signature, challenge, dashboardId}'` — HTTP 200, body contains `token`
5. Same login with signature from a different keypair — HTTP 401
6. Same register with same email — HTTP 409, body contains "support@onout.org"
7. `psql -h 10.10.10.2 -U dashboard_auth -d dashboard_auth -c "SELECT private_key FROM users WHERE email='<test_email>'"` — non-empty value
8. Share + second login sequence — all return expected HTTP codes (200)
9. External POST to `/api/auth/register` via `https://d9000000000001.wpmix.net` returns 403 (nginx deny)
10. Puppeteer: `page.evaluate(() => document.querySelector('.dashboard-data').style.filter)` → contains `blur`
11. Puppeteer with extension: blur removed, data visible after extension auto-signs challenge
12. Puppeteer with Auth API stopped: overlay shows service-unavailable text
13. `ssh root@<backup-lxc-ip> "ls -lt /backup/dashboard_auth/ | head -3"` — recent dump file (within last 24h)
14. `psql -h 10.10.10.2 -U dashboard_auth -d dashboard_auth -c "SELECT count(*) FROM users WHERE email LIKE 'test-%'"` → 0 (cleanup complete)
15. Report file exists: `cat logs/working/task-12/post-deploy-qa-report.json` — valid JSON with `status` field

## Details

**Dependencies:**
- Task 11 (deploy) must be fully complete: PM2 process `dashboard-auth-api` running, nginx reloaded, extension published
- PG on LXC 102 must be accessible from VM104 (`pg_hba.conf` configured in task 1)
- `INTERNAL_API_KEY` env var must be set on `dashboard-auth-api` PM2 process (configured in task 11)
- `JWT_SECRET` env var must be set (configured in task 11)

**Infrastructure topology:**
- VM104 (95.217.227.164) — application server, runs Auth API on port 8095
- Reverse proxy (62.109.14.209) — SSL termination, forwards `d*.wpmix.net` to VM104:80, which then nginx-proxies `/api/auth/*` to 127.0.0.1:8095
- LXC 102 (10.10.10.2) — pg-db, runs PostgreSQL with `dashboard_auth` database
- Backup LXC — separate container for pg_dump (find actual IP from task 9 output or decisions.md)

**Auth API endpoints (live base URL: `https://d9000000000001.wpmix.net`):**
- `GET /api/auth/health` — public
- `POST /api/auth/register` — requires `Authorization: Bearer <INTERNAL_API_KEY>`, nginx restricts to localhost
- `POST /api/auth/login` — public (browser-callable)
- `POST /api/auth/share` — requires `Authorization: Bearer <INTERNAL_API_KEY>`, nginx restricts to localhost

**Generating a valid signature for login test:**
Use `node -e` with ethers.js v6 to generate a test keypair and sign a challenge. Challenge format: `JSON.stringify({dashboardId, timestamp: Date.now(), nonce: crypto.randomUUID()})`. Sign with `wallet.signMessage(challenge)` (EIP-191 personal_sign). Auth API uses `ethers.verifyMessage(challenge, signature)` for ecrecover.

**Puppeteer for browser flows:**
- Headless Chrome with `--headless=new` flag supports extension loading
- Load extension via `--load-extension=/path/to/built/extension` and `--disable-extensions-except=...`
- Pre-seed keypair via `chrome.scripting.executeScript` before navigation to dashboard
- Use `waitForFunction(() => window.ethereum !== undefined)` after navigation to wait for content script injection

**Nginx external restriction test:**
The reverse proxy at 62.109.14.209 forwards all traffic to VM104. The VM104 nginx config restricts `/api/auth/register` and `/api/auth/share` with `allow 127.0.0.1; deny all`. External requests for these endpoints should receive 403. Test by sending POST to `https://d9000000000001.wpmix.net/api/auth/register` without using localhost — expect 403.

**Test user naming:**
Use a consistent prefix like `test-postdeploy-` for test email addresses (e.g., `test-postdeploy-1@example.com`) to make cleanup reliable and identifiable.

**Challenge replay protection (5-minute window):**
When testing login, ensure challenge timestamp is recent (Date.now()). Optionally test the rejection case: send a challenge with `timestamp` set to `Date.now() - 400000` (400 seconds ago, over 5 min) → should return 401.

**Edge cases to check:**
- Auth API under load (not required for this task — covered in unit tests)
- CORS headers: if making requests from browser context in Puppeteer, verify `Access-Control-Allow-Origin` header matches dashboard origin
- PM2 restart after stopping: `pm2 start dashboard-auth-api` must restore service; verify by re-running health check
