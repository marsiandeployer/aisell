# Task 1: Setup PostgreSQL on LXC 102

## Required Skills

Before starting, load: `/skill:infrastructure-setup` — [SKILL.md](/root/.claude/skills/infrastructure-setup/SKILL.md)

## Description

PostgreSQL is not running on pg-db (LXC 102, IP 10.10.10.2) — the port 5432 connection is currently refused, though the container is pingable. This task installs and configures PostgreSQL on that container, creates the `dashboard_auth` database, applies the schema (tables: `users`, `dashboards`, `dashboard_access`), creates a dedicated DB user, and configures `pg_hba.conf` to accept connections from VM104 (the current application server).

This is a prerequisite for Task 2 (Auth API service), which connects to PostgreSQL at `10.10.10.2:5432` using the `dashboard_auth` user and database.

Access path: you must SSH into the Proxmox host (`ssh -i ~/.ssh/github_deploy_key root@95.217.227.164 -p 22`), then enter the LXC container with `pct exec 102 -- bash` (or `pct enter 102`).

## What to do

1. SSH into Proxmox host and enter LXC 102. Verify the container is reachable and check PostgreSQL installation status.

2. Install PostgreSQL if it is not installed. LXC 102 runs Debian — use `apt install postgresql` to install. Start and enable the service (`systemctl start postgresql && systemctl enable postgresql`).

3. Create the `dashboard_auth` database and a dedicated PostgreSQL user `dashboard_auth` with a strong password. The password must be saved to an env var reference (document the variable name as `PG_PASSWORD`).

4. Apply the schema — create the three tables `users`, `dashboards`, and `dashboard_access` with the exact DDL from the tech-spec. Tables must be created in the `dashboard_auth` database owned by the `dashboard_auth` role.

5. Configure `pg_hba.conf` to allow the `dashboard_auth` user to connect to the `dashboard_auth` database from VM104's IP address using password (md5 or scram-sha-256) authentication. VM104's IP inside the Proxmox bridge network is `10.10.10.104` (or the actual IP — confirm via `ip a` on VM104 before editing `pg_hba.conf`).

6. Configure `postgresql.conf` — set `listen_addresses` to include `10.10.10.2` (or `'*'`) so the server accepts remote connections from the bridge network.

7. Reload PostgreSQL config (`pg_ctlcluster reload` or `systemctl reload postgresql`).

8. Verify connectivity from VM104 using the `psql` client (v14.20 is already installed on VM104): `psql -h 10.10.10.2 -U dashboard_auth -d dashboard_auth -c '\dt'` — must list the three tables.

## Details

**SSH access to LXC 102:**

```bash
# Step 1: SSH to Proxmox host
ssh -i ~/.ssh/github_deploy_key root@95.217.227.164 -p 22

# Step 2: Enter container
pct exec 102 -- bash
# OR:
pct enter 102
```

**VM104 bridge IP to use in pg_hba.conf:**

Before editing `pg_hba.conf`, confirm VM104's IP on the bridge network:

```bash
# Run on VM104 (or via proxmox exec on CT 104):
ip a | grep "10.10.10"
```

Expected to be `10.10.10.104` based on the pattern (101 = CT101, 102 = pg-db). Confirm before hardcoding.

**Schema DDL (exact, from tech-spec):**

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  address VARCHAR(42) NOT NULL UNIQUE,
  email VARCHAR(255) NOT NULL UNIQUE,
  private_key TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE dashboards (
  id SERIAL PRIMARY KEY,
  dashboard_id VARCHAR(50) NOT NULL UNIQUE,
  owner_address VARCHAR(42) NOT NULL REFERENCES users(address),
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE dashboard_access (
  id SERIAL PRIMARY KEY,
  dashboard_id VARCHAR(50) NOT NULL REFERENCES dashboards(dashboard_id),
  address VARCHAR(42) NOT NULL REFERENCES users(address),
  granted_by VARCHAR(42) NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(dashboard_id, address)
);
```

Run as the `dashboard_auth` user (or as `postgres` superuser against the `dashboard_auth` database):

```bash
psql -U postgres -d dashboard_auth -f /tmp/schema.sql
```

**pg_hba.conf entry to add:**

```
# dashboard_auth remote access from VM104
host    dashboard_auth    dashboard_auth    10.10.10.104/32    scram-sha-256
```

If VM104 IP is not `10.10.10.104`, replace accordingly. Add this line BEFORE the default `host all all 127.0.0.1/32` or similar entries so it is evaluated first.

**postgresql.conf change:**

Find and change (or add):

```
listen_addresses = 'localhost,10.10.10.2'
```

Or more permissively for the bridge network only:

```
listen_addresses = '*'
```

Note: `listen_addresses = '*'` is safe here because `pg_hba.conf` still controls who can authenticate.

**Password management:**

The `dashboard_auth` role password must be recorded for use by the Auth API service (Task 2). Document it as the environment variable `PG_PASSWORD` that will be set in the PM2 process env. Do NOT commit the password in any file. It will be added to the `.env` file on VM104 (same pattern as other secrets in `/root/aisell/noxonbot/.env`).

**LXC 102 OS:**

pg-db runs Debian. PostgreSQL version available via `apt` is typically 13 or 15 depending on Debian release. If the installed version differs from the psql client on VM104 (v14.20), ensure protocol compatibility — PostgreSQL clients are generally backward compatible across minor versions; libpq protocol is stable.

**Edge cases:**

- If PostgreSQL is already partially installed but not running, check `systemctl status postgresql` and `journalctl -xe` for errors before reinstalling.
- If `pg_hba.conf` already has `host all all 0.0.0.0/0 md5` (open to all), remove or restrict it — the security auditor will flag this.
- If `pct exec 102` is not available (Proxmox version differences), try `pct enter 102` interactively or use `ssh root@10.10.10.2` if the container has SSH configured.
- The `dashboard_auth` user should NOT be a superuser. Grant only: CONNECT on database, CREATE on schema public (or specific table-level grants).
