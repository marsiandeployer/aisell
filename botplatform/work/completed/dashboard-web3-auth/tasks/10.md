# Task 10: Pre-deploy QA

## Required Skills

Before starting, load: `/skill:pre-deploy-qa` — [SKILL.md](/root/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Acceptance testing for the dashboard-web3-auth feature before deploy. This task runs the full test suite (unit, integration, E2E), verifies all acceptance criteria from user-spec (AC1–AC25) and tech-spec (16 criteria), and confirms no regressions in pre-commit tests.

This task does not implement code. It exercises everything built in tasks 1–9: PostgreSQL schema (task 1), Auth API service (task 2), extension wallet and content scripts (task 3), nginx proxy (task 4), CLAUDE.md.template auth section (task 5), webchat keypair integration (task 6), unit/integration tests (task 7), E2E tests (task 8), and key backup (task 9).

QA is its own verification — no separate reviewer. Output is a JSON report at `logs/working/qa-report.json` and a brief entry in `decisions.md`.

## What to do

1. Run unit and integration test suite: `node tests/test_auth_api.js` — record pass/fail counts.

2. Run E2E test suite: `node tests/test_dashboard_auth_e2e.js` — record pass/fail counts.

3. Run pre-commit regression tests one by one and confirm none fail:
   - `node tests/test_folder_structure.js`
   - `node tests/test_claude_md_templates.js`
   - `node tests/test_security_precommit.js`
   - `node tests/test_ts_syntax.js`

4. Verify all user-spec ACs (AC1–AC25) using the verification table in Verification Steps — for each criterion record status (passed / failed / not_verifiable) and evidence.

5. Verify all tech-spec ACs (16 criteria from Acceptance Criteria section below) — same approach.

6. Check PM2 process stability: `pm2 status dashboard-auth-api` — process should be online with no restart loops.

7. Save full report to `logs/working/qa-report.json` following the pre-deploy-qa output format (see SKILL.md).

8. Write concise entry to `decisions.md` with summary and link to report.

## Acceptance Criteria

### User-spec ACs (AC1–AC25)

**Auth API:**
- [ ] AC1: `GET /api/auth/health` returns 200 with `{status: "ok", pg: "connected"}`
- [ ] AC2: `POST /api/auth/register` with `{address, email, privateKey, dashboardId}` (server-to-server) returns 201 and creates owner record
- [ ] AC3: `POST /api/auth/login` with valid signature (ecrecover matches owner address) returns 200 + JWT; invalid signature returns 401
- [ ] AC4: Unparseable signature in login returns 401 Unauthorized
- [ ] AC5: Duplicate email in register returns 409 with message pointing to support@onout.org
- [ ] AC6: After register, `private_key` field is non-empty in DB (verify via psql query to users table)
- [ ] AC7: Sharing — adding email to dashboard's allowed list grants that user's address access (login returns 200)

**Extension:**
- [ ] AC8: Extension injects `window.ethereum` provider on `d*.wpmix.net` pages
- [ ] AC9: Extension generates keypair on webchat request and stores in `chrome.storage`
- [ ] AC10: Extension signs challenge on dashboard request via `window.ethereum`
- [ ] AC11: Extension supports keypair import (for access recovery)

**Dashboard:**
- [ ] AC12: Protected dashboard without auth — title visible, data blurred
- [ ] AC13: Protected dashboard with valid JWT — all data visible
- [ ] AC14: No extension installed — overlay with Chrome Web Store install link shown
- [ ] AC15: Auth API unavailable — message "Сервис авторизации временно недоступен" shown
- [ ] AC16: Dashboard without auth — works as before, fully public
- [ ] AC17: Expired JWT — extension auto-re-signs, new JWT obtained, data stays visible
- [ ] AC18: Dashboard regeneration — owner address and auth code preserved
- [ ] AC25: Challenge contains timestamp; Auth API rejects challenges older than 5 minutes (replay protection)

**Infrastructure:**
- [ ] AC19: PostgreSQL running on pg-db (LXC 102, 10.10.10.2); tables `users`, `dashboards`, `dashboard_access` exist
- [ ] AC20: Nginx proxies `/api/auth/*` to Auth API service
- [ ] AC21: Auth API responds with CORS headers for `d*.wpmix.net` origin
- [ ] AC22: Key backup synced to isolated LXC

**Integration:**
- [ ] AC23: `CLAUDE.md.template` contains "Auth" section with OWNER_ADDRESS placeholder and auth overlay patterns
- [ ] AC24: Keypair generated during dashboard creation in webchat (extension → webchat → Auth API flow works)

### Tech-spec ACs (16 criteria)

- [ ] Auth API responds to all endpoints with correct HTTP status codes (200, 201, 401, 404, 409)
- [ ] PG schema applied: tables `users`, `dashboards`, `dashboard_access` exist with correct columns

- [ ] All unit tests pass
- [ ] All integration tests pass
- [ ] No regressions in existing pre-commit tests (test_folder_structure, test_claude_md_templates, test_security_precommit, test_ts_syntax)
- [ ] PM2 process `dashboard-auth-api` runs stable (no restart loops)
- [ ] JWT_SECRET configured via environment variable (not hardcoded)
- [ ] INTERNAL_API_KEY configured via environment variable for register/share endpoints
- [ ] Nginx restricts /api/auth/register and /api/auth/share to localhost
- [ ] All PG queries use parameterized statements (no string interpolation)
- [ ] Auth API CORS restricted to `d{digits}.wpmix.net` origin pattern (not arbitrary `*.wpmix.net`)

## Details

**Test files:**

- `botplatform/tests/test_auth_api.js` — unit + integration tests for Auth API (written in task 7). Covers: ecrecover verification, challenge validation (valid/expired/malformed), JWT creation and verification, rate limiter behavior, register/login/share/health endpoints, replay protection, CORS headers, duplicate email → 409, non-owner share attempt → 403, PG error → 503.

- `botplatform/tests/test_dashboard_auth_e2e.js` — Puppeteer E2E tests (written in task 8). Covers: dashboard without extension shows blur + install overlay, dashboard with extension performs full auth flow and shows data, wrong keypair → data stays blurred + "no access" overlay, Auth API stopped → "service unavailable" message, content script isolation (window.ethereum undefined on non-matching URL), JWT expiry + reactive re-auth.

- Pre-commit tests (existing, must not regress): `tests/test_folder_structure.js`, `tests/test_claude_md_templates.js`, `tests/test_security_precommit.js`, `tests/test_ts_syntax.js`.

**Dependencies:** All tasks 1–9 must be complete before running QA.

**Criteria likely not_verifiable pre-deploy:**

- AC22 (backup sync to isolated LXC) — requires SSH access to backup LXC; may not be reachable in QA environment. Document concrete post-deploy steps.
- AC8, AC10 (window.ethereum injection, extension signs challenge) — E2E tests cover this with Puppeteer + extension loading; mark as not_verifiable only if Puppeteer test environment cannot load extensions.
- AC19 (PostgreSQL on pg-db) — requires network access to 10.10.10.2; if unavailable in test environment, mark as not_verifiable and defer to post-deploy.
- AC24 (full webchat keypair generation flow) — requires running webchat + extension interaction; partially covered by unit/integration tests.

**Edge cases to verify explicitly:**

- Challenge timestamp exactly at 5-minute boundary — boundary test for replay protection.
- CORS: origin `https://d123.wpmix.net` (valid) vs `https://dabc.wpmix.net` (non-digits, must be blocked).
- PM2 restart count > 3 is a blocker — investigate logs before declaring stable.
- JWT claims: decode the JWT returned by login and verify address, dashboardId, iat, exp fields are all present and exp = iat + 3600.
- Nginx `deny all` on register/share: simulate external request through nginx (not direct 8095) — must return 403.

**Environment variables to verify are set (not hardcoded):**

```bash

# Expected: both present and non-empty
```
