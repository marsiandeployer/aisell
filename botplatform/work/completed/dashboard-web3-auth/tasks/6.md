# Task 6: Webchat keypair generation integration

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

This task integrates keypair generation into the webchat flow for SimpleDashboard. When Claude generates a dashboard with auth, the webchat must obtain an Ethereum address to embed as `OWNER_ADDRESS` in the generated `index.html`.

The flow lives on the client side of webchat (the JavaScript embedded in the HTML served by `webchat.ts`). The webchat iframe, running inside the extension panel, sends a `generate_keypair` postMessage to the extension. The extension (Task 3) responds with `{address, privateKey}`. The webchat frontend then POSTs that data to a new server-side endpoint `/api/auth/register-owner`, which calls the Auth API server-to-server, and then passes `OWNER_ADDRESS` back to Claude as a system-level instruction.

On timeout (10 seconds) or error, the flow degrades gracefully: the user sees a notification, and the dashboard is created without auth. This ensures existing dashboard creation never breaks.

This task modifies only `botplatform/src/webchat.ts` — both the client-side JavaScript (embedded HTML template with the postMessage logic) and the server-side Express routes (the `/api/auth/register-owner` endpoint that calls Auth API).

The feature covers AC24 from user-spec: keypair generation during webchat dashboard creation.

## What to do

1. Add a client-side `requestKeypairFromExtension(requestId)` function in the embedded JavaScript of `webchat.ts`. The function sends `{ type: 'generate_keypair', requestId }` to `window.parent` and registers a one-time `message` event listener scoped to that `requestId`. It resolves with `{ address, privateKey }` from the extension response, or rejects after 10 seconds.

2. Add a pending-requests registry on the client side: a `Map<string, { resolve, reject, timer }>` object keyed by `requestId`. The existing `window.addEventListener('message', ...)` handler (line 3119 in webchat.ts) must be extended to handle `{ type: 'response', requestId, data }` messages from the extension by looking up the pending map and settling the corresponding promise.

3. Add a `triggerKeypairFlow(email, dashboardId)` async function on the client side that calls `requestKeypairFromExtension`, then POSTs `{ address, privateKey, email, dashboardId }` to a new server-side endpoint `/api/auth/register-owner`. On success, it returns the `address`. On timeout or fetch error, it sends a user-visible warning message ("Extension не отвечает, проверьте что SimpleDashboard установлен") and returns `null`.

4. Integrate `triggerKeypairFlow` into the message send flow: when the bot response contains a signal that auth is requested (e.g., a special marker or when the product is SimpleDashboard and auth is indicated in the CLAUDE.md.template context), call `triggerKeypairFlow` before submitting the message to Claude, and inject `OWNER_ADDRESS: <address>` into the system context passed with the message.

5. Add the server-side Express route `POST /api/auth/register-owner` in `webchat.ts`. The route: requires an authenticated session, reads `{ address, privateKey, email, dashboardId }` from the body, validates them (address 0x format, email format, non-empty strings), then POSTs to `http://127.0.0.1:8095/api/auth/register` with `Authorization: Bearer <INTERNAL_API_KEY>` header. Returns `{ address }` on 201 from Auth API, or proxies the error status.

6. Add the `INTERNAL_API_KEY` environment variable read in `webchat.ts` (alongside the Auth API URL, defaulting to `http://127.0.0.1:8095`). Both must be read from env, not hardcoded.

## Details

**File: `botplatform/src/webchat.ts`**

This is a 4600+ line file. Changes are in two places:

**Client-side JS (embedded HTML template, around line 2500-3180):**

The embedded HTML string returned by `buildWebchatHtml()` (or equivalent function) contains the client-side JavaScript. The existing `window.addEventListener('message', function(ev) { ... })` at line 3119 handles `developer_mode_changed` responses from the extension. This listener must be extended (or a separate listener added) to handle the `{ type: 'response', requestId, ... }` pattern.

Add a module-level `pendingExtensionRequests` map:
```
const pendingExtensionRequests = new Map();
```

In the message listener, add a case: if `ev.data.type === 'response'` and `ev.data.requestId` is in the map, resolve or reject the pending promise and clean up the timer.

The `requestKeypairFromExtension(requestId)` function should use `window.parent.postMessage` with explicit `targetOrigin` (the webchat URL origin, not wildcard `'*'`). Note: existing code at line 2512 uses `'*'` but tech-spec Decision requires explicit targetOrigin for security — update both existing and new postMessage calls. The 10s timeout uses `setTimeout` to reject and delete from the map.

**Server-side route (in the `main()` function around line 4748 where other `app.post` routes are):**

Add `POST /api/auth/register-owner` after the existing auth routes. It should call `requireSessionApi` middleware. Read `INTERNAL_API_KEY` from `process.env.INTERNAL_API_KEY` at startup (not inside the handler). Read `AUTH_API_URL` from `process.env.AUTH_API_URL` with default `'http://127.0.0.1:8095'`. Use Node's built-in `fetch` (available since Node 18, confirmed Node v22.21.1 on this server).

**Dependencies:**
- Task 2 (Auth API) must be running at port 8095 — this task calls it server-to-server.
- Task 3 (Extension wallet) must implement the `generate_keypair` postMessage handler in panel.js — this task sends to it.
- No new npm packages needed: `fetch` is built-in in Node 22, address format validated with a regex.

**Edge cases:**
- Extension not installed (not in iframe context): `window.parent === window` means postMessage is a no-op. The listener will never receive a response and the 10s timeout fires. Handle via the timeout path.
- Extension installed but `generate_keypair` not yet implemented (Task 3 incomplete): same as above — timeout fires.
- Duplicate registration (user already has address in Auth API): Auth API returns 409. The server-side route should proxy the 409 with `{ error: "Email already registered" }`. The client should treat this as success if an address is already known (or surface the error for the user to handle).
- `dashboardId` must be the actual webchat userId string (e.g., `'9000000000126'`). It comes from `window.__WEBCHAT_USER_ID__` on the client side.
- The `requestId` for the postMessage should be a UUID or random hex to prevent collisions if multiple requests are in flight.

**Implementation hints:**
- The `notifyExtensionOnFileCreated` function at line 2505 is a good reference for how the client-side code posts to the parent (extension panel). Note: existing code uses `'*'` as targetOrigin — for the keypair flow, use explicit origin per tech-spec security requirement.
- The extension responds with `{ type: 'response', requestId, data: { address, privateKey } }` — this is the same pattern used by `sendResponse(requestId, data)` in panel.js (line 217).
- For the server-side `fetch` call to Auth API, handle network errors (ECONNREFUSED) — the Auth API may not be running yet. Wrap in try/catch and return 503.
- The `isSimpleDashboardProduct()` check at line 371 can gate the feature — keypair generation only makes sense for SimpleDashboard, not for other webchat products.
- For the CHANGE/WHY/REF comment pattern used throughout webchat.ts, use:
  ```
  // CHANGE: Add keypair generation flow for dashboard auth
  // WHY: Webchat needs OWNER_ADDRESS to embed in generated index.html
  // REF: tech-spec Task 6, user-spec AC24
  ```
