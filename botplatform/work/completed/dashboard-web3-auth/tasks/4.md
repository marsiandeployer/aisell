# Task 4: Nginx proxy for Auth API

## Required Skills

Before starting, load:
- `/skill:infrastructure-setup` — [SKILL.md](/root/.claude/skills/infrastructure-setup/SKILL.md)

## Description

Add an `location /api/auth/` proxy block to the nginx wildcard vhost `/etc/nginx/sites-enabled/d-wildcard.wpmix.net` on VM104. This block proxies all `/api/auth/*` requests to the Auth API service running on `127.0.0.1:8095`.

Two specific sub-locations restrict access to server-to-server endpoints: `/api/auth/register` and `/api/auth/share` are locked to localhost only (`allow 127.0.0.1; deny all`), blocking browser calls from the internet. `/api/auth/login` and `/api/auth/health` remain publicly accessible.

This makes dashboard auth calls same-origin: dashboard pages on `d{uid}.wpmix.net` call `/api/auth/login` without any CORS complexity (same-origin via nginx proxy). The Auth API still sets CORS headers defensively, but they are not required for the browser auth flow.

Without this task the Auth API on port 8095 is only reachable directly on the host, not via the public domain. This task enables the full browser auth flow and enforces the nginx-side restriction from Decision 10 and Decision 11.

## What to do

1. Open `/etc/nginx/sites-enabled/d-wildcard.wpmix.net` and add the auth proxy block. It must appear before the catch-all `location /` block. Insert three location blocks: the restricted `/api/auth/register`, the restricted `/api/auth/share`, and the general `/api/auth/` that proxies everything else.

2. Test the nginx config with `nginx -t`. Fix any syntax errors before reloading.

3. Reload nginx with `nginx -s reload` (no downtime).

4. Verify the health endpoint is reachable through nginx using the Host header approach (see Verification Steps).

5. Verify that `/api/auth/register` and `/api/auth/share` return 403 Forbidden when called from outside localhost.

## Details

### File: `/etc/nginx/sites-enabled/d-wildcard.wpmix.net`

**Current state** (complete file as read):
```nginx
server {
    listen 80;
    server_name ~^d(?<userid>\d+)\.wpmix\.net$;

    root /root/aisell/botplatform/group_data/user_$userid;
    index index.html;

    # Main page
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Static assets caching
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|webp|json)$ {
        expires 7d;
        add_header Cache-Control "public";
        try_files $uri =404;
    }

    # Security: deny access to dotfiles (CLAUDE.md is ok, but .history.json etc)
    location ~ /\. {
        deny all;
    }

    # Deny CLAUDE.md access from web
    location = /CLAUDE.md {
        deny all;
    }
}
```

**What to add** — insert the three auth location blocks inside the `server {}` block, before `location /`:

```nginx
    # Auth API: register (server-to-server only, localhost restricted)
    location = /api/auth/register {
        allow 127.0.0.1;
        deny all;
        proxy_pass http://127.0.0.1:8095;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $http_x_forwarded_proto;
    }

    # Auth API: share (server-to-server only, localhost restricted)
    location = /api/auth/share {
        allow 127.0.0.1;
        deny all;
        proxy_pass http://127.0.0.1:8095;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $http_x_forwarded_proto;
    }

    # Auth API: all other endpoints (login, health) - public
    location /api/auth/ {
        proxy_pass http://127.0.0.1:8095;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $http_x_forwarded_proto;
    }
```

**Placement is critical:** these three blocks must appear before `location /` in the server block. Nginx uses the most specific matching location — `location = /api/auth/register` (exact match) takes priority over `location /api/auth/` (prefix match), which in turn takes priority over `location /` (catch-all).

### Dependencies

- Task 1 (PostgreSQL setup) must be done: Auth API needs PG running on 10.10.10.2.
- Task 2 (Auth API service) must be done: Auth API must be running on 127.0.0.1:8095 for proxy_pass to succeed. If the Auth API is not yet running, `nginx -t` will still pass (nginx does not validate upstream availability at config test time), but the proxy will return 502.

### Edge cases

- `proxy_set_header X-Forwarded-Proto $http_x_forwarded_proto` — the reverse proxy (62.109.14.209) sends the original proto as `X-Forwarded-Proto`. VM104 should pass it through using `$http_x_forwarded_proto`, not `$scheme` (which would always be `http` on VM104 since SSL is terminated at the reverse proxy).
- `allow 127.0.0.1; deny all` in the restricted locations: when a request comes from the internet through the reverse proxy, nginx sees the source IP as the reverse proxy's internal IP (or `$remote_addr` from the forwarded connection). Verify that external requests are actually blocked — the reverse proxy `proxy_pass http://95.217.227.164` sets `proxy_set_header Host $host` but the TCP source is the reverse proxy IP, not the original client IP. This means `allow 127.0.0.1; deny all` correctly blocks reverse proxy traffic (the reverse proxy IP is not 127.0.0.1).
- Webchat backend on port 8094 calls Auth API for register and share server-to-server. These calls go to `http://127.0.0.1:8095` directly (not through nginx). However, if webchat ever uses the domain URL internally, the nginx localhost restriction would allow it since webchat runs on the same host. Prefer direct port access from webchat to Auth API to avoid the nginx routing overhead.
- Do not add `proxy_read_timeout` or `proxy_connect_timeout` in this config — the defaults (60s) are fine for auth operations. Auth API responses should be fast (<1s).
- Do not add CORS headers in nginx. Auth API Express handles CORS with the `d{digits}.wpmix.net` origin restriction (Decision 6). Adding CORS headers in both nginx and Express would cause duplicate header errors.

### Infrastructure context

- VM104 is the application server (95.217.227.164). All nginx vhosts are on this server.
- The reverse proxy at 62.109.14.209 terminates SSL and proxies everything to 95.217.227.164:80, passing `Host: d{userid}.wpmix.net` through. No changes needed on the reverse proxy for this task.
- After `nginx -s reload`, zero downtime — existing connections are not dropped.
