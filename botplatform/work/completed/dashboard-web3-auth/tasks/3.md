## Description

This task adds web3 wallet capability to the existing Chrome extension (webchat-sidebar). The extension currently has no content scripts and no crypto functionality ‚Äî it only manages a side panel with a webchat iframe.

The goal is to make the extension act as an Ethereum wallet for `d*.wpmix.net` dashboard pages. This is required for the Web3 auth flow: dashboard pages need `window.ethereum` to request signatures, and the extension needs to respond to keypair management requests from the webchat iframe.

**What is being built:**

1. **`ethereum-provider.js`** (new file, MAIN world) ‚Äî sets `window.ethereum = SimpleDashboardProvider` on `d*.wpmix.net` pages. The provider handles `eth_requestAccounts` (returns stored address) and `personal_sign` (returns signature). Communicates with isolated world via `CustomEvent('sd-eth-request')` / `CustomEvent('sd-eth-response')`.

2. **`content-script-ethereum.js`** (new file, ISOLATED world) ‚Äî relay script automatically injected by Chrome into all `d*.wpmix.net` pages. Listens for `CustomEvent('sd-eth-request')` from page world, forwards to background.js via `chrome.runtime.sendMessage({type: 'eth_request', ...})`, receives response from background, dispatches `CustomEvent('sd-eth-response')` back to page world.

3. **`background.js`** (modify) ‚Äî add `chrome.runtime.onMessage` handler for `type: 'eth_request'` messages from content scripts. Accesses `chrome.storage.local` to read stored keypair, uses bundled ethers.js to sign challenges. Returns `{result}` on success, `{error: {code: 'NO_KEYPAIR'}}` when no keypair exists.

4. **`panel.js`** (modify) ‚Äî add cases in the `handleMessage` switch for four new postMessage types from the webchat iframe: `generate_keypair` (generate or return existing keypair), `get_address` (return stored address only), `sign_challenge` (sign a string with stored key), `import_keypair` (store an externally-provided keypair for recovery). All use `chrome.storage.local` for persistence.

5. **`build.js`** (modify) ‚Äî add `'storage'` to `permissions` array, add `content_scripts` manifest entries for both worlds, copy the two new source files to the build output.

6. **`ethers.min.js`** ‚Äî bundled in the extension for use in background.js (CDN cannot be used in service workers per MV3 CSP). Must be downloaded and placed in `src/`.

This task covers ACs 8, 9, 10, 11 from user-spec (extension injection, keypair generation, signing, import).

## Details

**Files and current state:**

- `extensions/webchat-sidebar/src/background.js` (75 lines) ‚Äî MV3 service worker. Currently has one `chrome.runtime.onMessage` listener that handles `open_preview`. The listener does NOT return `true`, meaning it's synchronous. The new `eth_request` branch MUST return `true` to enable async `sendResponse` (required when accessing `chrome.storage.local`). Add the new branch inside the existing listener ‚Äî check `message.type === 'eth_request'` and return `true` in that branch only.

- `extensions/webchat-sidebar/src/panel.js` (519 lines) ‚Äî `handleMessage(event)` at line 48; switch statement at line 55 with cases for `get_tab_info`, `read_page_content`, `capture_screenshot`, `set_developer_mode`, `get_developer_mode_state`, `clear_selected_element`, `file_created`, `open_showcases`, `open_webchat_prompt`. New keypair cases go inside this switch. The `sendResponse(requestId, data)` helper at line 217 calls `sendToWebchat({type: 'response', requestId, data})`. Use this same helper for new cases. The `iframeOrigin` variable (line 25) is the expected origin ‚Äî `sendToWebchat` already enforces it.

- `extensions/webchat-sidebar/build.js` (273 lines) ‚Äî `manifest` object starts at line 146. `permissions` array is at line 152. After adding `'storage'`, add `content_scripts` property to the manifest object. Then after line 211 (where `onboarding-screenshots` are copied), add `copyFile` calls for the three new files. Do NOT use `copyDir` for individual files ‚Äî use `fs.copyFileSync`.

- `extensions/webchat-sidebar/src/ethereum-provider.js` (new) ‚Äî runs in MAIN world, has NO access to `chrome.*` APIs. Uses only `window.dispatchEvent` and `window.addEventListener`. Must use a Promise-based queue: each pending request is stored by a unique request ID; when `sd-eth-response` arrives, the matching promise is resolved/rejected. Use `crypto.randomUUID()` for request IDs. The `sd-eth-request` event carries `{requestId, method, params}` in `detail`.

- `extensions/webchat-sidebar/src/content-script-ethereum.js` (new) ‚Äî runs in ISOLATED world, has access to `chrome.runtime`. Listens for `sd-eth-request` events on `window`. When received, calls `chrome.runtime.sendMessage({type: 'eth_request', method, params, requestId})`. In the callback, dispatches `new CustomEvent('sd-eth-response', {detail: {requestId, result, error}})` on `window`. Guard against `chrome.runtime.lastError` to avoid unchecked error warnings.

**Two-world relay pattern (Decision 4 from tech-spec):**
MAIN world can set `window.ethereum` but cannot call `chrome.runtime.sendMessage`. ISOLATED world can call Chrome APIs but cannot touch page globals. The bridge between them is CustomEvent dispatched on `window` ‚Äî both worlds share the same `window` object in MV3.

**ethers.min.js bundling:**
- Download: `curl -L https://cdn.jsdelivr.net/npm/ethers@6/dist/ethers.min.js -o extensions/webchat-sidebar/src/ethers.min.js`
- In background.js: `importScripts('ethers.min.js')` ‚Äî MV3 service workers support `importScripts`. This makes `ethers` available as a global.
- In panel.js: ethers.js is NOT available as a global. Panel runs in the side panel page (HTML context). Load it via a `<script>` tag in `panel.html`, or use dynamic import. Check panel.html to see what scripts are already loaded.

**`chrome.storage.local` keypair schema:**
```
chrome.storage.local.get('sd_keypair') ‚Üí { sd_keypair: { address: '0x...', privateKey: '0x...' } | undefined }
chrome.storage.local.set({ sd_keypair: { address, privateKey } })
```
Use the key `'sd_keypair'` consistently across background.js and panel.js.

**`import_keypair` validation:**
Before storing, validate the private key is a valid 32-byte hex string (with or without `0x` prefix). Use `new ethers.Wallet(privateKey)` ‚Äî if it throws, return error response. Do not store invalid keys.

**`sign_challenge` in panel.js vs `personal_sign` in background.js:**
These are two separate signing paths. `sign_challenge` is called by webchat via postMessage (panel.js). `personal_sign` is called by the dashboard page via `window.ethereum` (background.js via content script relay). Both should produce the same EIP-191 signature for the same message+key combination, verifiable with `ethers.verifyMessage`.

**PostMessage security:**
All postMessage calls from panel.js to the webchat iframe MUST use `iframeOrigin` as `targetOrigin` (already enforced by `sendToWebchat`). The `handleMessage` guard at line 49 (`if (event.origin !== iframeOrigin) return;`) prevents spoofing ‚Äî do not bypass this for new handlers.

**Dependencies:**
- `ethers` (v6) ‚Äî download ethers.min.js for background.js, use `new ethers.Wallet(privateKey)` and `wallet.signMessage(message)`, `ethers.Wallet.createRandom()`
- No new npm packages needed for the extension itself (ethers.min.js is bundled as a static file)
- Task 3 is independent (wave 1) ‚Äî no dependency on Task 1 (PostgreSQL) or Task 2 (Auth API)

**Edge cases:**
- `generate_keypair` called multiple times: must return the same keypair (read from storage first, generate only if absent)
- `personal_sign` called when no keypair: return `{error: {code: 'NO_KEYPAIR'}}` ‚Äî dashboard will show "contact support" overlay (this is the `window.ethereum.request()` rejection path)
- Content script injected, but extension context invalidated (e.g., extension updated mid-session): `chrome.runtime.sendMessage` will throw; catch and dispatch error response
- `import_keypair` with a key that does NOT correspond to the registered address: this is allowed at the extension level ‚Äî the Auth API will reject login if the recovered address doesn't match
- Multiple dashboard tabs open simultaneously: each tab has its own content script instance, but all share the same background.js service worker and `chrome.storage.local`

**Build.js `content_scripts` manifest entry structure:**
```json
"content_scripts": [
  {
    "matches": ["https://d*.wpmix.net/*"],
    "js": ["ethereum-provider.js"],
    "world": "MAIN",
    "run_at": "document_start"
  },
  {
    "matches": ["https://d*.wpmix.net/*"],
    "js": ["content-script-ethereum.js"],
    "run_at": "document_start"
  }
]
```
Note: `world: "MAIN"` requires Chrome 111+. No `world` field for ISOLATED world (it is the default).
