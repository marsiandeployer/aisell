# Task 8: E2E tests with Puppeteer

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Write E2E tests for the dashboard Web3 auth flow using Puppeteer with the SimpleDashboard Chrome extension loaded into the browser. The tests validate the complete browser flow: content script injection, `window.ethereum` availability, signature-based auth via the Auth API, and correct overlay rendering across all states (no extension, wrong keypair, API down, JWT expiry).

This is the final validation gate for the auth feature before pre-deploy QA (task 10). It depends on the Auth API (task 2), extension wallet (task 3), nginx proxy (task 4), and CLAUDE.md template (task 5) all being in place, because E2E tests exercise the full stack without mocking.

The test file is `botplatform/tests/test_dashboard_auth_e2e.js` — a standalone Node.js script (no test framework), consistent with the existing test pattern in the project. Puppeteer 24.32.1 is already installed as a devDependency.

**Key challenge:** Loading a Chrome extension in headless mode and pre-seeding keypair state in `chrome.storage.local` before the dashboard page navigates. Each test scenario requires a freshly launched browser with the extension loaded, then programmatic setup of the storage state before navigating to the test URL.

## What to do

1. Create `botplatform/tests/test_dashboard_auth_e2e.js` as a standalone Node.js script that imports puppeteer and runs all E2E scenarios sequentially.

2. Implement the test runner infrastructure:
   - `assert(condition, message)` helper function following the project pattern from `test_claude_md_templates.js`
   - `launchBrowserWithExtension(extensionPath)` — launches Chrome with extension loaded via `--load-extension` and `--disable-extensions-except` flags, using `headless: 'new'`
   - `seedKeypairToStorage(browser, extensionId, keypair)` — pre-seeds a keypair (`{address, privateKey}`) into `chrome.storage.local` before navigating to dashboard
   - `getExtensionId(browser)` — retrieves the loaded extension's ID from `chrome://extensions` or via background page
   - Per-test browser launch and cleanup to keep tests isolated

3. Implement test fixture setup:
   - Use a real registered test keypair (owner address registered in Auth API for a test dashboard ID)
   - Or generate a fixture dashboard HTML that embeds a known `OWNER_ADDRESS` — create `botplatform/tests/fixtures/test_dashboard.html` with the auth overlay code pattern from tech-spec and a known owner address
   - Serve the fixture via a lightweight local HTTP server on a free port, or use the actual `d{id}.wpmix.net` test dashboard created during integration setup

4. Implement all 6 E2E test scenarios (see TDD Anchor section for details).

5. Add test result summary at the end: passed count, failed count, per-test status lines.

6. Verify the script exits with code 0 on all pass, code 1 on any failure.

## TDD Anchor

All tests are written first, then run to confirm they fail (since the feature isn't live yet), then the prior tasks are expected to make them pass. Each test scenario is a named async function.

- `tests/test_dashboard_auth_e2e.js::test_no_extension_shows_blur_overlay` — Open dashboard URL in a browser WITHOUT the extension loaded → verify page title is visible, data sections are blurred (CSS filter or class `blurred` present), install overlay element is in DOM with link to Chrome Web Store. No `window.ethereum` should exist (`page.evaluate(() => typeof window.ethereum) === 'undefined'`).

- `tests/test_dashboard_auth_e2e.js::test_with_extension_auth_succeeds_and_unblurs` — Launch browser WITH extension, seed owner keypair into `chrome.storage.local`, navigate to dashboard → `waitForFunction(() => window.ethereum !== undefined)`, wait for auth flow to complete → assert blur CSS is removed, data sections are visible, JWT is stored in `sessionStorage` (`page.evaluate(() => !!sessionStorage.getItem('dashboard_jwt'))`).

- `tests/test_dashboard_auth_e2e.js::test_wrong_keypair_shows_no_access_overlay` — Launch browser WITH extension, seed a DIFFERENT keypair (not the owner) into storage, navigate to dashboard → wait for auth attempt to complete → assert data stays blurred, "no access" overlay is present in DOM (check for text "нет доступа" or element with `data-overlay="no-access"`).

- `tests/test_dashboard_auth_e2e.js::test_auth_api_down_shows_service_unavailable` — Stop the Auth API PM2 process before the test (`execSync('pm2 stop dashboard-auth-api')`), launch browser WITH extension, seed owner keypair, navigate to dashboard → wait for fetch to fail → assert "service unavailable" message appears in DOM. Restart Auth API after test in `finally` block.

- `tests/test_dashboard_auth_e2e.js::test_content_script_not_injected_on_non_matching_url` — Launch browser WITH extension, navigate to `https://simpledashboard.wpmix.net` (non-matching URL pattern) → assert `window.ethereum` is `undefined` on that page (content script should not inject outside `d*.wpmix.net`).

- `tests/test_dashboard_auth_e2e.js::test_jwt_expiry_triggers_reauth` — Launch browser WITH extension, seed owner keypair, configure a short JWT TTL (modify Auth API env or use a separate test endpoint), navigate to dashboard → wait for initial auth → wait past TTL → trigger a page action that causes a re-fetch → assert extension auto-re-signs (check for second JWT in sessionStorage with different `exp` claim) and data remains visible. Skip this test with a warning message if short TTL configuration is not available in the environment.

**Extension path:**
The extension must be built before tests run. Default path: `/root/aisell/extensions/webchat-sidebar/out/webchat-sidebar/`. The test script should check that this path exists and abort with a clear error if not (`if (!fs.existsSync(extensionPath)) throw new Error('Build extension first: node build.js ...')`).

**Getting the extension ID:**
After launching with `--load-extension`, the extension ID is not known in advance. Retrieve it by navigating to `chrome://extensions/` and extracting it from the page, or by targeting the background service worker. Pattern:
```
const targets = await browser.targets();
const extensionTarget = targets.find(t => t.type() === 'service_worker' && t.url().startsWith('chrome-extension://'));
const extensionId = extensionTarget.url().split('/')[2];
```

**Seeding chrome.storage.local:**
The extension uses MV3 with a service worker (not MV2 background page — there is no `_generated_background_page.html`). Use the DevTools protocol to execute storage operations in the service worker context:
```javascript
// MV3 service worker approach (the only correct one for this extension):
const swTarget = targets.find(t => t.type() === 'service_worker' && t.url().includes(extensionId));
const client = await swTarget.createCDPSession();
await client.send('Runtime.evaluate', {
  expression: `chrome.storage.local.set({ keypair: { address: '${address}', privateKey: '${privateKey}' } })`
});
```
**Note:** Do NOT use `_generated_background_page.html` — that's MV2 only. This extension is MV3 with `background.js` as service worker.

**Test dashboard fixture:**
Use the test user registered in Auth API during integration setup (task 7). The test should have a config object at the top of the file:
```javascript
const TEST_CONFIG = {
  dashboardUrl: process.env.TEST_DASHBOARD_URL || 'https://d9000000000126.wpmix.net',
  ownerAddress: process.env.TEST_OWNER_ADDRESS || '0x...',  // from task 2 integration test setup
  ownerPrivateKey: process.env.TEST_OWNER_PRIVATE_KEY || '0x...',
  wrongPrivateKey: process.env.TEST_WRONG_PRIVATE_KEY || '0x...',  // any other key
};
```

**Puppeteer launch args for extension loading:**
```javascript
const browser = await puppeteer.launch({
  headless: 'new',
  args: [
    `--disable-extensions-except=${extensionPath}`,
    `--load-extension=${extensionPath}`,
    '--no-sandbox',
    '--disable-setuid-sandbox',
  ],
  enableExtensions: true,  // REQUIRED: without this, Puppeteer adds --disable-extensions flag and extension won't load
  executablePath: puppeteer.executablePath(),
});
```
**Critical:** Without `enableExtensions: true`, Puppeteer automatically adds `--disable-extensions` to Chrome args, which overrides `--load-extension`.

**waitForFunction for content script:**
After navigation, content scripts are injected asynchronously. Use:
```javascript
await page.waitForFunction(() => window.ethereum !== undefined, { timeout: 5000 });
```

**DOM selectors for overlays:**
The dashboard HTML (generated by Claude per the CLAUDE.md.template instructions) should use specific class names or data attributes for overlay elements. Agree on these in task 5:
- Install overlay: `[data-overlay="install-extension"]` or `.overlay-install`
- No access overlay: `[data-overlay="no-access"]` or `.overlay-no-access`
- Service unavailable: `[data-overlay="service-unavailable"]` or `.overlay-service-unavailable`
- Blur container: `.data-protected` or `.blurred`

If the actual generated HTML uses different selectors, the test must adapt. Read the generated dashboard HTML before writing final selectors.

**Auth API down test:**
Requires `child_process.execSync`. Must wrap in try/finally to restart Auth API even if assertions fail:
```javascript
try {
  execSync('pm2 stop dashboard-auth-api', { stdio: 'inherit' });
  // ... test assertions ...
} finally {
  execSync('pm2 start dashboard-auth-api', { stdio: 'inherit' });
  // wait for API to come back
  await new Promise(r => setTimeout(r, 2000));
}
```

**JWT expiry test:**
This test is environment-sensitive. If Auth API doesn't expose a way to set short TTL (e.g., `JWT_TTL_SECONDS` env var), skip it gracefully: `console.warn('SKIP jwt_expiry_test: JWT_TTL_SECONDS env var not available'); return 'skipped';`

**Dependencies:**
- Task 2 (Auth API) must be running on port 8095
- Task 3 (Extension) must be built at `extensions/webchat-sidebar/out/webchat-sidebar/`
- Task 4 (Nginx proxy) must proxy `/api/auth/*` from `d*.wpmix.net` to 8095
- Task 5 (CLAUDE.md template) must have auth overlay DOM selectors documented
- A test dashboard registered in Auth API (can be set up in the test script's `beforeAll` equivalent, or pre-registered via task 7 integration tests)

**No Fallback Policy applies:**
- Do not fall back to a fake DOM structure if real dashboard URL fails — raise a clear error
- Do not silently pass tests if the extension fails to load — abort with explicit message
- Exception: JWT expiry test can be skipped if env var not available (this is a known environment constraint, not a fallback for logic)

**Edge cases:**
- Extension takes time to inject content script after page load — always use `waitForFunction(() => window.ethereum)` before asserting auth flow
- `chrome.storage.local` seed must happen BEFORE the dashboard page navigates, not after
- Browser cleanup (`browser.close()`) must run in `finally` to avoid zombie Chrome processes
- If `pm2 stop` fails (process already stopped), the catch in finally must still attempt `pm2 start`
- The wrong-keypair test must use a valid Ethereum private key (just not the owner's) — use `ethers.Wallet.createRandom().privateKey` inline or hardcode a known-bad key
