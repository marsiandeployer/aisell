# Task 7: Auth API unit and integration tests

## Required Skills

Before starting, load: `/skill:code-writing` — [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Write a standalone Node.js test script (`botplatform/tests/test_auth_api.js`) that covers both unit and integration scenarios for the Auth API service built in Task 2.

The test script runs as `node tests/test_auth_api.js` and exits with code 0 on full pass, non-zero on any failure. No test framework (Jest, Mocha) — use the same pattern as the existing `test_claude_md_templates.js` and `test_folder_structure.js`: plain `assert()` function, colored console output, summary at the end.

This task is the primary guard for auth security correctness. It verifies that ecrecover, challenge validation, JWT issuance, rate limiting, and all HTTP endpoints behave exactly as the security model requires. Integration tests run against a live Auth API on port 8095 — the service must be running before tests execute.

Unit tests extract and test pure logic functions directly (by requiring `auth-api.ts` compiled output or wrapping relevant logic in testable form). Integration tests use the built-in `http` module (or `node-fetch` if already present in package.json) to send real HTTP requests to `http://localhost:8095`.

## What to do

1. Create `botplatform/tests/test_auth_api.js` as a single standalone script.

2. Implement a test runner that: defines `assert(condition, description)`, tracks `passed`/`failed` counts, prints colored results per test, prints summary, exits with `process.exit(failed === 0 ? 0 : 1)`.

3. Implement unit test section — import or inline the relevant pure functions from auth-api (ecrecover wrapper via ethers, challenge validator, JWT create/verify, rate limiter logic). Test each function in isolation with no HTTP calls.

4. Implement integration test section — send HTTP requests to `http://localhost:8095`. Each test registers unique users (generate fresh Ethereum keypairs via ethers.Wallet.createRandom() per test). Clean up PG test data in a teardown block after all tests.

5. All test scenarios listed in the TDD Anchor section must be covered. Each scenario is a separate named test.

6. Add a startup check: if Auth API is not reachable on port 8095, print an error and exit with code 1 immediately (do not fail silently with connection errors mid-test).

7. Add a teardown block that deletes test-created rows from PG using the `pg` client (connect to 10.10.10.2, `dashboard_auth` DB, delete WHERE email LIKE 'test-%@test.test').

## Details

**Files:**
- `botplatform/tests/test_auth_api.js` (new) — the entire test script. Single file, no imports outside of Node.js built-ins, `ethers`, `pg`, and `jsonwebtoken` (all already in package.json from Task 2).

**Setup required before integration tests run:**
- Auth API must be running: `pm2 status dashboard-auth-api` → online, or `curl http://localhost:8095/api/auth/health` → 200.
- Task 4 (nginx proxy) must be done for the nginx proxy test to pass. If nginx test fails, it should fail with a clear message, not crash the whole suite.
- PG must be accessible from VM104 on 10.10.10.2:5432 (Task 1 done).

**Unit test implementation approach:**
- Auth API (`src/auth-api.ts`) is compiled TypeScript. The built JS is in `dist/`. For unit tests, either: (a) require the compiled `dist/auth-api.js` and export test-friendly pure functions, or (b) inline the pure logic in the test file itself using `ethers` and `jsonwebtoken` directly. Option (b) is safer since it doesn't depend on the auth-api export surface. Check `src/auth-api.ts` to decide which approach is cleaner.
- For ecrecover unit test: use `ethers.Wallet.createRandom()`, sign a message, then use `ethers.verifyMessage()` to recover address. Compare with wallet.address. For invalid sig test: mutate the signature bytes and verify recovery returns a different address (not "throws" — ethers.verifyMessage recovers a wrong address, it doesn't throw for malformed-but-parseable sigs).
- For challenge validation unit test: construct a mock challenge object `{dashboardId, timestamp, nonce}`, serialize to JSON, test the validation function with various timestamps.
- For rate limiter unit test: construct a rate limiter instance directly (if exported), call it N times with the same key, verify the Nth call is rejected, then advance the clock (mock Date.now or wait for window reset if window is short enough).

**Integration test implementation approach:**
- Use Node.js built-in `https` / `http` module or `node-fetch`. Prefer built-in `http` to avoid dependency issues.
- Each integration test that needs a registered user: generate a fresh keypair with `ethers.Wallet.createRandom()`, use `wallet.address` and a unique email like `test-${Date.now()}-${Math.random().toString(36).slice(2)}@test.test`.
- Register endpoint requires `Authorization: Bearer <INTERNAL_API_KEY>` header — read the value from env var `INTERNAL_API_KEY` or the test must set it. Document in the test that this env var must be set.
- For login: generate a challenge JSON string `{dashboardId, timestamp: Date.now(), nonce: crypto.randomUUID()}`, sign with `wallet.signMessage(challengeStr)`, POST to `/api/auth/login`.
- For replay test: use a challenge with `timestamp` set to `Date.now() - 6 * 60 * 1000` (6 minutes ago). Should return 401.
- For rate limit test (429): send 11 POST /api/auth/register requests in a loop. This will create up to 10 real users — make sure teardown cleans them all.
- For CORS test: add `Origin: https://d12345.wpmix.net` header to the request and check the `Access-Control-Allow-Origin` response header.
- For nginx proxy test: send request to `http://localhost:80/api/auth/health` with `Host: d9000000000000.wpmix.net`. This requires nginx to be running and configured (Task 4). If localhost:80 returns connection refused, skip test with a warning.

**Teardown:**
- At end of all tests, connect to PG directly: `const { Pool } = require('pg')`, pool to `10.10.10.2:5432`, database `dashboard_auth`, user `dashboard_auth`. Delete from `dashboard_access`, `dashboards`, `users` where email matches test pattern. Wrap in try/catch — teardown failure should warn but not change exit code.

**Edge cases:**
- Rate limiter test: if previous test runs already exhausted the window, the 429 test may fail. Use a unique IP-like key if rate limiter is keyed by something controllable, or document that this test may need a fresh rate limiter window (wait or restart Auth API).
- Share authorization (403): send share request with `ownerAddress` set to a different address than the actual owner. Auth API must check that `ownerAddress` matches the stored owner in PG.
- JWT tampered test: decode the JWT, modify the payload, re-encode with base64 (without re-signing). `jsonwebtoken.verify()` should throw.

**Dependencies:**
- Task 2 (Auth API service) must be complete and running.
- Task 4 (nginx proxy) must be complete for the nginx proxy test.
- Packages `ethers`, `pg`, `jsonwebtoken` available in `botplatform/package.json` (added by Task 2).
