# Task 9: Key backup to isolated LXC

## Required Skills

Before starting, load: `/skill:infrastructure-setup` — [SKILL.md](/root/.claude/skills/infrastructure-setup/SKILL.md)

## Description

Task 1 sets up PostgreSQL on LXC 102 (pg-db, 10.10.10.2) and stores all auth data there — including plaintext private keys (conscious MVP tradeoff per Decision 8). If LXC 102 is compromised, all user private keys are exposed. This task establishes a periodic backup to an isolated LXC container so that recovery remains possible even if the primary PG is lost, and provides a secondary copy for forensic use if the primary is compromised.

The backup mechanism: a cron job on LXC 102 (or VM104) runs `pg_dump` on the `dashboard_auth` database and transfers the dump to an isolated LXC via `scp` or `rsync`. LXC 101 (Alpine, 10.10.10.101) is the designated backup target — it is reachable from LXC 102 over the internal bridge network (10.10.10.0/24) and is already accessible via `ssh root@95.217.227.164 -p 22101`.

The backup destination should have minimal access: no outbound internet, no other services, SSH key-only access. The dump file should be timestamped and the most recent dump must be verifiable as less than 24 hours old.

This task covers: SSH key setup between LXC 102 and LXC 101, the pg_dump script, the cron schedule, and verification that the dump contains expected tables.

## What to do

1. Confirm LXC 101 is reachable from LXC 102 over the internal bridge (ping 10.10.10.101 from inside LXC 102). If not reachable, establish internal routing.

2. Generate an SSH keypair on LXC 102 dedicated to backup (`ssh-keygen -t ed25519 -f /root/.ssh/backup_to_lxc101 -N ""`). Add the public key to LXC 101's `/root/.ssh/authorized_keys` — restrict to the backup user only (no shell access preferred; if Alpine supports `command=` restriction in authorized_keys, apply it).

3. Create the backup script `/root/backup_dashboard_auth.sh` on LXC 102:
   - Run `pg_dump -U dashboard_auth -d dashboard_auth -F c -f /tmp/dashboard_auth_$(date +%Y%m%d_%H%M%S).dump`
   - Transfer dump to LXC 101: `scp -i /root/.ssh/backup_to_lxc101 /tmp/dashboard_auth_*.dump root@10.10.10.101:/backups/dashboard_auth/`
   - Delete local `/tmp/` dump after successful transfer (check scp exit code)
   - Log result (timestamp + success/failure) to `/var/log/dashboard_auth_backup.log`
   - Make the script executable (`chmod +x`)

4. Create backup directory on LXC 101: `mkdir -p /backups/dashboard_auth` with restricted permissions (`chmod 700`).

5. Add cron job on LXC 102 to run the script daily at 02:00 UTC:
   ```
   0 2 * * * /root/backup_dashboard_auth.sh >> /var/log/dashboard_auth_backup.log 2>&1
   ```
   On Alpine/Debian LXC, use `crontab -e` or write to `/etc/cron.d/dashboard-auth-backup`.

6. Run the script manually once to verify it produces a dump and transfers it successfully to LXC 101.

7. Verify the dump file on LXC 101: check it exists, is timestamped within the last 24 hours. To verify the dump contains the expected tables, run `pg_restore --list` on LXC 102 (Debian, where `pg_restore` is installed) — not on LXC 101 (Alpine, which likely does not have `pg_restore`). Transfer the dump back temporarily or verify on LXC 102 before the scp step.

1. SSH to Proxmox host: `ssh -i ~/.ssh/github_deploy_key root@95.217.227.164 -p 22`

2. Enter LXC 102 (pg-db): `pct enter 102`

3. Run backup script manually:
   ```bash
   /root/backup_dashboard_auth.sh
   ```
   Expected: exits 0, prints success to stdout.

4. Check backup log:
   ```bash
   cat /var/log/dashboard_auth_backup.log
   ```
   Expected: timestamp + "backup successful" or equivalent.

5. Verify dump on LXC 101 — SSH to LXC 101 from Proxmox host:
   ```bash
   ssh root@95.217.227.164 -p 22101
   find /backups/dashboard_auth -name "*.dump" -mtime -1
   ```
   Expected: at least one dump file listed.

6. Check dump contents (run on LXC 102 where pg_restore is available, not on LXC 101 Alpine):
   ```bash
   # On LXC 102:
   scp -i /root/.ssh/backup_to_lxc101 root@10.10.10.101:/backups/dashboard_auth/dashboard_auth_*.dump /tmp/verify_dump.dump
   pg_restore --list /tmp/verify_dump.dump | grep -E "TABLE DATA|TABLE"
   rm /tmp/verify_dump.dump
   ```
   Expected: entries for `users`, `dashboards`, `dashboard_access`.
   Alternative: install `postgresql-client` on LXC 101 (`apk add postgresql-client`) if ongoing verification is needed on the backup host.

7. Verify cron is installed on LXC 102:
   ```bash
   crontab -l
   ```
   Expected: line with `/root/backup_dashboard_auth.sh` scheduled at `0 2 * * *`.

8. Verify backup directory permissions on LXC 101:
   ```bash
   ls -ld /backups/dashboard_auth
   ```
   Expected: `drwx------` (700).

## Details

**Infrastructure topology:**
- Proxmox host: `95.217.227.164` (SSH port 22, key `~/.ssh/github_deploy_key`)
- LXC 102 (pg-db): internal IP `10.10.10.2`, Debian, PostgreSQL source
- LXC 101 (Alpine): internal IP `10.10.10.101`, SSH via `ssh root@95.217.227.164 -p 22101`, backup target
- Internal bridge: `10.10.10.0/24` — LXC 102 and LXC 101 can reach each other directly

**Entering LXC containers from Proxmox host:**
- LXC 102: `pct enter 102` (no SSH port, enter via Proxmox)
- LXC 101: `ssh root@95.217.227.164 -p 22101` or `pct enter 101`

**pg_dump command:**
```bash
PGPASSWORD="$PG_PASSWORD" pg_dump \
  -h localhost \
  -U dashboard_auth \
  -d dashboard_auth \
  -F c \
  -f /tmp/dashboard_auth_$(date +%Y%m%d_%H%M%S).dump
```
Custom format (`-F c`) is binary, compressed, and restorable via `pg_restore`. Plain SQL (`-F p`) is human-readable but larger — either is acceptable; custom format is recommended.

**Cron alternatives on Alpine (LXC 101 is Alpine, LXC 102 is Debian):**
- Debian LXC 102: standard `cron` daemon. `crontab -e` or write to `/etc/cron.d/`.
- If `cron` is not installed on LXC 102: `apt-get install -y cron && systemctl enable cron && systemctl start cron`.

**Retention policy (optional but recommended):**
Keep only the last 7 dumps on LXC 101 to avoid disk fill:
```bash
ls -t /backups/dashboard_auth/*.dump | tail -n +8 | xargs rm -f
```
Add this to the backup script after the scp transfer.

**Security — backup destination isolation:**
- LXC 101 (Alpine) should have no outbound internet routes if possible.
- The SSH authorized_keys entry for the backup key on LXC 101 should use `command=` restriction if Alpine's OpenSSH supports it. This limits what the key can do even if stolen.
- Example restricted authorized_keys entry:
  ```
  command="scp -t /backups/dashboard_auth/",no-pty,no-agent-forwarding,no-port-forwarding ssh-ed25519 AAAA...
  ```
  Note: `scp -t` is the receiving end of scp; this restricts the key to only accepting incoming file copies.

**Decision 8 context:**
Storing private keys in backup is a conscious tradeoff (tech-spec Decision 8). The backup provides admin recovery capability. The tradeoff is documented — do not change the decision here, but ensure the backup destination is as isolated as possible to contain the blast radius.

**Dependencies:**
- Task 1 must be complete (PostgreSQL must be running on LXC 102 with `dashboard_auth` database and schema applied) before this task can run the first backup.

**Edge cases:**
- LXC 101 unreachable: the backup script should exit non-zero and log the failure; do not silently drop failed backups.
- PG not running: `pg_dump` will fail; log and exit non-zero.
- Disk full on LXC 101: `scp` will fail; retention cleanup (keep last 7) mitigates this.
- First run: no prior dumps exist — that is expected; script should succeed on first run.

**No PM2 for this task:** The cron job runs inside LXC 102, not on VM104. PM2 is not available inside LXC containers (it lives on VM104). Use standard Unix cron.
