# Task 2: Auth API service

## Required Skills

Before starting, load: `/skill:code-writing` ŌĆö [SKILL.md](/root/.claude/skills/code-writing/SKILL.md)

## Description

Create a standalone Express service (`botplatform/src/auth-api.ts`) running on port 8095 that provides Ethereum-based authentication for SimpleDashboard pages.

The service handles four endpoints: `POST /api/auth/register` (server-to-server from webchat ŌĆö stores a new owner keypair and dashboard mapping in PostgreSQL), `POST /api/auth/login` (from dashboard browser ŌĆö verifies an Ethereum signature via ecrecover and returns a JWT), `POST /api/auth/share` (server-to-server ŌĆö grants a second user's address access to a dashboard by email lookup), and `GET /api/auth/health`.

Authentication flow: the dashboard page signs a JSON challenge `{dashboardId, timestamp, nonce}` using the extension's wallet. The Auth API receives `{signature, challenge, dashboardId}`, runs `ethers.verifyMessage(challenge, signature)` to recover the signer's address, checks that address is in the `dashboard_access` list for the given `dashboardId`, and issues a 1-hour JWT if authorized.

This is Wave 1 ŌĆö it runs independently from PG setup (Task 1), but full integration testing requires Task 1's PostgreSQL schema to be in place.

## What to do

1. Install new packages in `botplatform/package.json`: `ethers` (v6), `pg`, `jsonwebtoken`, `@types/pg`, `@types/jsonwebtoken`.

2. Create `botplatform/src/auth-api.ts` as a self-contained Express application with:
   - CORS middleware restricted to origins matching `^https://d\d+\.wpmix\.net$` (Decision 6)
   - `SlidingWindowRateLimiter` copied from `webchat.ts` (~56 lines, the class is self-contained)
   - Rate limiters: register 10/hour per IP, login 30/hour per IP
   - PostgreSQL connection pool to host `10.10.10.2`, database `dashboard_auth`
   - JWT signing with `JWT_SECRET` from env, 1-hour TTL (`{address, dashboardId, iat, exp}`)
   - `INTERNAL_API_KEY` from env, checked on register and share endpoints via `Authorization: Bearer` header

3. Implement input validation as the first layer on each endpoint ŌĆö return 400 before touching business logic if:
   - address does not match `/^0x[0-9a-fA-F]{40}$/`
   - email does not contain `@`
   - challenge is not valid JSON with `dashboardId`, `timestamp`, `nonce` fields
   - challenge `timestamp` is older than 5 minutes (`Date.now() - challenge.timestamp > 5 * 60 * 1000`)

4. Implement all four endpoints with correct HTTP status codes:
   - `POST /api/auth/register` ŌåÆ 201 on success, 409 on duplicate email, 503 on PG error
   - `POST /api/auth/login` ŌåÆ 200 + `{token}` on success, 401 on bad/unrecognized signature, 503 on PG error

   - `GET /api/auth/health` ŌåÆ 200 + `{status: "ok", pg: "connected"}` (or `"error"` with 503)

5. Add the `dashboard-auth-api` process entry to `botplatform/ecosystem.config.js` so PM2 can start it with `tsx src/auth-api.ts`.

6. Write tests in `botplatform/tests/test_auth_api.ts` (unit + integration) ŌĆö see TDD Anchor.

### PostgreSQL queries (all parameterized)

Register flow ŌĆö three inserts in sequence:
```sql
INSERT INTO users (address, email, private_key) VALUES ($1, $2, $3)
  ON CONFLICT (email) DO NOTHING RETURNING id
INSERT INTO dashboards (dashboard_id, owner_address) VALUES ($1, $2)
INSERT INTO dashboard_access (dashboard_id, address, granted_by) VALUES ($1, $2, $3)
```

Login ŌĆö single access check:
```sql
SELECT 1 FROM dashboard_access WHERE dashboard_id = $1 AND address = $2
```

Share ŌĆö three steps:
```sql
SELECT address FROM users WHERE email = $1
SELECT owner_address FROM dashboards WHERE dashboard_id = $1
INSERT INTO dashboard_access (dashboard_id, address, granted_by) VALUES ($1, $2, $3)
  ON CONFLICT DO NOTHING
```

### CORS implementation

CORS is restricted to origins matching `^https://d\d+\.wpmix\.net$`. Either use the `cors` npm package (add it to dependencies) with an `origin` function, or implement CORS headers manually ŌĆö both are acceptable. Manual approach avoids an extra dependency:

```typescript
const DASHBOARD_ORIGIN_RE = /^https:\/\/d\d+\.wpmix\.net$/;
// On each response: if origin matches RE, set Access-Control-Allow-Origin to that origin
// Handle OPTIONS preflight with 204 and required headers
```

### Rate limiting ŌĆö 429 response shape

Match the webchat.ts pattern so clients can parse `retryAfterSeconds`:
```json
{"error": "Too many requests", "retryAfterSeconds": 3512}
```
`retryAfterSeconds = Math.ceil(retryAfterMs / 1000)` from the limiter's `check()` result.

### ecrecover

Use `ethers.verifyMessage(challenge, signature)` ŌĆö returns the recovered address as a checksummed string. Normalize both sides with `ethers.getAddress()` before comparing to stored address. If `verifyMessage` throws (malformed signature), catch the error and return 401. Do not propagate the raw error.

### Challenge validation order

The `challenge` field in `LoginRequest` is a JSON string (not an object). Validate in this order:
1. Parse with `JSON.parse` in try/catch ŌĆö if throws: 400
2. Check `dashboardId`, `timestamp`, `nonce` fields present ŌĆö if missing: 400
3. Check `timestamp` is a number ŌĆö if not: 400
4. Check `Date.now() - timestamp <= 5 * 60 * 1000` ŌĆö if expired: 401 (not 400)

Step 4 is 401 not 400 because the request was structurally valid ŌĆö just timed out.

### Edge cases

- Two simultaneous register calls for the same email: `ON CONFLICT (email) DO NOTHING RETURNING id` returns empty rows for the second ŌĆö detect empty `rows.length === 0` and return 409.
- `ethers.verifyMessage` on syntactically valid but wrong signature: returns a valid-looking address that simply won't match any DB row ŌĆö the access check returns false ŌåÆ 401. Correct behavior.
- PG pool errors: wrap all `pool.query` calls in try/catch, return 503 on any pool error. Log the error server-side but do not expose pg error messages to clients.
- Missing `INTERNAL_API_KEY` env var: fail fast at startup with a thrown Error ŌĆö do not start the service without it.

### No Fallback Policy

Per project CLAUDE.md rules ŌĆö do NOT silently fall back to defaults for required config:
```typescript
// WRONG
const secret = process.env.JWT_SECRET || 'default-secret';

// CORRECT
const secret = process.env.JWT_SECRET;
if (!secret) throw new Error('JWT_SECRET env var is required');
```
